## 10.2 概念铺垫：Agent 的遗忘问题

大多数 AI 都有「金鱼记忆」——**对话一结束，什么都忘了**。

```
第一天：「我叫小张，我是做前端开发的。」
Agent：好的小张，记住了！
第二天：「帮我写个组件。」
Agent：好的，请问你用什么框架？（← 忘了你是前端的了！）
```

这种「每次都从零开始」的体验很差。理想状态是：

```
第二天：「帮我写个组件。」
Agent：好的小张！根据你的习惯，我用 React + TypeScript 帮你写，
      你通常偏好函数式风格对吧？（← 记住了！）
```

这就是**长期记忆（Long-Term Memory）**的价值——让 Agent 跨对话记住：
- 你是谁（身份、职业）
- 你喜欢什么（偏好、风格）
- 你们聊过什么（历史要点）
- 关于你的具体事实（「养了一只猫」「用 Mac」「讨厌 PHP」）

> 💡 **类比**：没有记忆的 Agent 像「每天换一个新客服」，你每次都要重新自我介绍。有记忆的 Agent 像「你的专属管家」，越来越懂你。

## 10.3 记忆的数据结构

deer-flow 的记忆存在一个 JSON 文件里（默认 `backend/.deer-flow/users/{userId}/memory.json`），分三大块：

```json
{
  "user_context": {
    "work_context": "小张是前端开发者，主要用 React/TypeScript",
    "personal_context": "养了一只橘猫，喜欢摄影",
    "top_of_mind": "最近在做一个可视化 dashboard 项目"
  },
  "history": {
    "recent_months": "本月聊过 React 性能优化、CSS Grid 布局",
    "earlier_context": "",
    "long_term_background": "半年前在学 Three.js"
  },
  "facts": [
    {
      "id": "fact_001",
      "content": "用户偏好使用函数式组件而非类组件",
      "category": "preference",           // 类别
      "confidence": 0.95,                 // 置信度 0-1
      "created_at": "2026-07-01T...",
      "source": "conversation"            // 来源
    },
    {
      "id": "fact_002",
      "content": "用户的代码风格偏好：使用双引号、4 空格缩进",
      "category": "preference",
      "confidence": 0.9,
      ...
    }
  ]
}
```

### 三大块解读

| 块 | 内容 | 作用 |
|----|------|------|
| **user_context** | 3 句话总结（工作/个人/当前关注） | 让 Agent 快速了解「你是谁」 |
| **history** | 时间维度的历史摘要 | 让 Agent 知道「我们聊过什么」 |
| **facts** | 离散的事实列表 | 细颗粒度的具体事实，带置信度 |

### 事实的类别（category）

每条事实归类到以下之一：

| 类别 | 含义 | 例子 |
|------|------|------|
| `preference` | 偏好 | 「喜欢深色主题」 |
| `knowledge` | 知识 | 「熟悉 React 内部原理」 |
| `context` | 上下文 | 「在 A 公司工作」 |
| `behavior` | 行为 | 「总是先要测试用例」 |
| `goal` | 目标 | 「想学 Rust」 |

### 置信度（confidence）

每条事实有 0-1 的置信度。低于阈值（默认 0.7）的事实会被丢弃——避免把「可能只是随口一说」的话当成确定事实记住。

> 💡 这种「**结构化 + 带置信度**」的记忆设计，比单纯「把所有对话历史存下来」聪明得多——它提炼了「有价值的信息」，控制了存储量，还能按置信度过滤噪音。

## 10.4 ★核心★记忆抽取的完整流程

记忆不是凭空产生的，它要经历一套完整的流水线。这是本章重点。

```
对话发生
   │
   ▼
① MemoryMiddleware.after_agent（对话结束后触发）
   │  过滤消息：只留「用户输入」和「AI 最终回复」
   │  丢弃工具调用细节（太杂，没记忆价值）
   ▼
② 入队：get_memory_queue().enqueue(...)
   │  注意：不立即处理，只是塞进队列
   ▼
③ 去抖（Debounce）：等待 30 秒（可配置）
   │  如果这 30 秒内又来新对话 → 合并，重新计时
   │  目的：积攒一批再处理，避免频繁调 LLM
   ▼
④ 批量处理：后台线程拿出一批对话
   ▼
⑤ LLM 抽取：调用 LLM 分析这批对话
   │  提示词大意：「阅读这些对话，提取关于用户的事实和上下文更新」
   │  LLM 返回结构化结果（新事实、上下文变化）
   ▼
⑥ 原子写入：更新 memory.json
   │  先写临时文件 → 重命名（原子操作，防写坏）
   │  去重：跳过已存在的相同事实
   ▼
⑦ 失效缓存：下次读取会拿到最新记忆
```

我们逐环节精读。

### 环节 ①：MemoryMiddleware 过滤 + 入队

第 06 章我们看过这个中间件的骨架，现在看细节：

```python
# 文件：agents/memory/middleware.py（简化）

class MemoryMiddleware(AgentMiddleware):
    def after_agent(self, state, runtime):
        """Agent 执行完后触发。"""
        config = get_memory_config()
        if not config.enabled:
            return None

        messages = state.get("messages", [])

        # ★过滤：只留有价值的信息★
        filtered = filter_messages_for_memory(messages)
        # filter_messages_for_memory 的逻辑：
        #   - 保留 HumanMessage（用户说的）
        #   - 保留最终的 AIMessage（Agent 回的）
        #   - 丢弃中间的 ToolMessage（工具调用细节）
        #   - 丢弃思考过程
        # 因为工具调用的细节没有记忆价值，只记「聊了什么」

        if not filtered:
            return None

        # ★入队（不立即处理）★
        queue = get_memory_queue()
        queue.enqueue(
            thread_id=thread_id,
            messages=filtered,
            user_id=get_effective_user_id(),   # ★按用户隔离
        )
        return None
```

**关键设计**：这里**只入队，不处理**。因为 LLM 抽取很慢（要调模型），如果在 `after_agent` 同步做，用户要等很久才能看到 Agent 的回复（回复和记忆抽取是绑在一起的）。入队后立即返回，用户无感。

### 环节 ②③：去抖队列（Debounce Queue）

这是记忆系统的性能关键。`MemoryUpdateQueue` 实现了「去抖」：

```python
# 文件：agents/memory/queue.py（简化）

class MemoryUpdateQueue:
    def __init__(self):
        self._queue = []
        self._lock = threading.Lock()
        self._timer = threading.Timer | None = None   # 去抖定时器

    def enqueue(self, thread_id, messages, user_id=None, ...):
        """加入队列，启动/重置去抖定时器。"""
        with self._lock:
            # 去重：同一个对话的多次更新合并
            key = (thread_id, user_id, agent_name)
            self._remove_existing(key)          # 移除旧的
            self._queue.append(ConversationContext(...))

            # ★去抖：每次有新的，就重置 30 秒计时器★
            if self._timer:
                self._timer.cancel()             # 取消旧计时
            self._timer = threading.Timer(
                self._debounce_seconds,          # 默认 30 秒
                self._process_queue,
            )
            self._timer.start()
```

### 什么是去抖（Debounce）

去抖是前端常见的概念，用生活例子理解：

> 你在等电梯，电梯门开着。每进来一个人，门就**重新等待 5 秒**（计时重置）。只要一直有人来，门就一直不关。等没人来了，等满 5 秒才关门。

记忆队列的去抖一样：
```
10:00:00  对话A结束 → 入队，启动 30 秒计时
10:00:10  对话B结束 → 入队，★重置★ 30 秒计时
10:00:20  对话C结束 → 入队，★重置★ 30 秒计时
10:00:50  （最后一个入队后 30 秒没人来了）
          → 批量处理 A、B、C 三段对话（一次 LLM 调用搞定）
```

**好处**：如果用户连续聊了好几轮，不会每轮都调一次 LLM 抽记忆，而是**攒一批一次处理**，省 token、省时间。

### 环节 ⑤：LLM 抽取

到时间后，后台线程批量处理队列。核心是调 LLM 分析对话：

```python
# 简化示意：updater.py

class MemoryUpdater:
    def update(self, contexts, existing_memory):
        """用 LLM 从对话中抽取记忆。"""
        # 构造提示词
        prompt = f"""
        你是一个记忆管理助手。阅读以下对话，提取关于用户的信息。

        现有记忆：
        {existing_memory.to_prompt()}

        最近对话：
        {format_conversations(contexts)}

        请提取：
        1. 新的事实（带置信度 0-1）
        2. 用户上下文的变化
        3. 历史要点更新

        以 JSON 格式返回。
        """

        # 调 LLM
        response = self._model.invoke(prompt)
        extracted = parse_json(response)

        # 合并到现有记忆（去重、按置信度过滤）
        return self._merge_memory(existing_memory, extracted)
```

注意：抽取用的模型可以单独配（`memory.model_name`），不一定要用主对话模型——可以用个便宜的小模型来干这活，省钱。

### 环节 ⑥：原子写入

抽取完要写入 `memory.json`。这里有个重要技巧——**原子写入（Atomic Write）**：

```python
# 简化示意：storage.py
import os

def save_memory_atomic(memory, path):
    """原子写入：先写临时文件，再重命名。"""
    tmp_path = path + ".tmp"
    with open(tmp_path, "w") as f:
        json.dump(memory.to_dict(), f)
        f.flush()
        os.fsync(f.fileno())       # 确保写到磁盘

    # ★rename 是原子操作★
    # 要么成功（整个文件替换），要么失败（原文件不动）
    # 不会出现「写一半崩了，文件损坏」的情况
    os.replace(tmp_path, path)
```

**为什么要原子写入**？因为记忆文件很重要，如果在写入过程中崩溃（断电、异常），普通写入会留下**损坏的半截文件**，导致所有记忆丢失。原子写入（写临时文件 + 重命名）保证：要么完整写入，要么完全不变，永远不会损坏。

> 💡 这是生产级系统的基本素养——**重要数据的写入必须是原子的**。

## 10.5 按用户隔离的存储

记忆是**按用户隔离**的。不同用户的记忆互不可见：

```
backend/.deer-flow/users/
├── default/                    ← 未登录用户（默认）
│   └── memory.json
├── user_alice/                 ← Alice 的记忆
│   └── memory.json
└── user_bob/                   ← Bob 的记忆
    └── memory.json
```

### user_id 怎么来

```python
from deerflow.runtime.user_context import get_effective_user_id

user_id = get_effective_user_id()
# 开了认证 → 返回真实用户 ID
# 没开认证（本地开发）→ 返回 "default"
```

### 为什么按用户隔离

- **隐私**：Alice 的记忆不能被 Bob 看到
- **准确**：不同人的偏好不同，不能混
- **多租户**：一个部署服务多个用户时必需

如果配置了 `memory.storage_path` 为绝对路径，则**退出按用户隔离**（所有用户共享一个 memory.json）——这只在特殊场景用。

## 10.6 记忆注入：影响下次对话

记忆抽取出来存在文件里，怎么影响 Agent 行为呢？答案是**注入到 system prompt**。

### 注入时机

在第 06 章的 `DynamicContextMiddleware`（或 prompt 组装阶段），会把记忆注入到系统提示词：

```python
# 简化示意
def build_system_prompt_with_memory(memory):
    facts_text = format_top_facts(memory.facts, limit=15)
    context_text = format_context(memory.user_context)

    return f"""
    ...（其他提示词）...

    <memory>
    ## 关于用户
    {context_text}

    ## 用户的关键事实
    {facts_text}
    </memory>

    ...（其他提示词）...
    """
```

### 注入策略

不是所有记忆都注入（太多了会撑爆上下文），而是有策略：

| 注入项 | 策略 | 配置 |
|--------|------|------|
| 事实（facts） | 只注入 Top 15（按置信度/时间排序） | `max_facts` |
| 上下文 | 注入 3 句话总结 | 始终注入 |
| Token 预算 | 总量不超过 2000 token | `max_injection_tokens` |

```python
# 伪代码
def format_top_facts(facts, limit=15, max_tokens=2000):
    """选 Top N 条事实，控制在 token 预算内。"""
    sorted_facts = sorted(facts, key=lambda f: f.confidence, reverse=True)
    selected = []
    token_count = 0
    for fact in sorted_facts[:limit]:
        fact_tokens = count_tokens(fact.content)
        if token_count + fact_tokens > max_tokens:
            break                       # 超预算，停
        selected.append(fact)
        token_count += fact_tokens
    return format_facts(selected)
```

### 注入的效果

注入后，Agent 的 system prompt 里就有了「用户画像」。下次对话：

```
用户：「写个按钮组件」
Agent 内心（看到记忆）：小张是前端的，喜欢 React + TS，函数式风格，双引号...
Agent 回答：「好的，我用 React + TypeScript 函数式组件帮你写，
           按你的习惯用双引号和 4 空格缩进...」
```

记忆就这样**默默影响**了 Agent 的行为——不是显式地「我记住了」，而是体现在每次回答的细节里。

## 10.7 记忆系统的工程细节

deer-flow 在记忆系统上做了很多细致的工程优化，挑几个值得学习的：

### 细节 1：token 计数的降级策略

注入记忆要算 token（控制预算）。deer-flow 支持两种计数方式：

| 方式 | 准确度 | 问题 |
|------|--------|------|
| `tiktoken`（默认） | 准确 | 首次用会下载 BPE 数据（联网，可能卡住） |
| `char` | 估算（CJK 感知） | 不联网，永远可用 |

deer-flow 做了**优雅降级**：tiktoken 加载失败时，冷却 10 分钟，期间用 char 估算；10 分钟后自动重试。这样网络问题不会永久影响记忆注入。

### 细节 2：事实去重

```python
# 更新记忆时，跳过已存在的相同事实
def _merge_facts(existing, new_facts):
    existing_contents = {normalize(f.content) for f in existing}
    for f in new_facts:
        if normalize(f.content) not in existing_contents:
            existing.append(f)
```

`normalize` 会做空白字符归一化——「喜欢猫」和「  喜欢 猫  」算同一条，避免重复累积。

### 细节 3：缓存失效

记忆读取会缓存（避免每次对话都读文件），但保存后要失效：

```python
# 简化
_memory_cache = {}

def get_memory(user_id):
    if user_id in _memory_cache:
        return _memory_cache[user_id]
    memory = load_from_file(user_id)
    _memory_cache[user_id] = memory
    return memory

def invalidate_cache(user_id):
    _memory_cache.pop(user_id, None)   # 保存后调用
```

### 细节 4：user_id 跨线程传递

记忆队列用 `threading.Timer` 后台处理，而 Python 的 `ContextVar`（存 user_id 的地方）**不会自动跨线程传递**。deer-flow 的解法是**在入队时就捕获 user_id**，存在 `ConversationContext` 里：

```python
# queue.py 的注释明确说了这点
@dataclass
class ConversationContext:
    user_id: str | None    # ★入队时捕获，不依赖 ContextVar
    # 因为 ContextVar 在 threading.Timer 线程里拿不到
```

这是异步/多线程编程的经典坑，deer-flow 处理得很细致。

## 10.8 动手实验：观察记忆的生成

### 步骤 1：开启记忆

确认 `config.yaml` 里：
```yaml
memory:
  enabled: true
  injection_enabled: true
```

### 步骤 2：告诉 Agent 一些偏好

开一个新对话，告诉它：
> 顺便记一下，我是做 Python 后端的，特别喜欢用 FastAPI，代码风格偏好类型注解齐全。

### 步骤 3：等待抽取

Agent 回复后，记忆不会立刻生成（要去抖 30 秒 + LLM 抽取）。等约 1 分钟。

### 步骤 4：查看记忆文件

打开 `backend/.deer-flow/users/default/memory.json`（未登录情况）。你应该能看到类似：

```json
{
  "facts": [
    {
      "content": "用户是 Python 后端开发者",
      "category": "context",
      "confidence": 0.95
    },
    {
      "content": "用户偏好使用 FastAPI 框架",
      "category": "preference",
      "confidence": 0.9
    }
  ]
}
```

### 步骤 5：验证记忆生效

开一个**新对话**，问：
> 帮我写一个 API 接口

如果记忆生效，Agent 应该**主动用 FastAPI**（而不是问你用什么框架），并且类型注解齐全。这就证明记忆被注入并影响了行为。

## 10.9 本章小结

✅ **长期记忆**让 Agent 跨对话记住用户，解决「金鱼记忆」问题
✅ 记忆数据分三块：**user_context（3 句总结）+ history（历史摘要）+ facts（带置信度的事实列表）**
✅ 记忆抽取是条流水线：**MemoryMiddleware 过滤入队 → 去抖 30 秒 → LLM 批量抽取 → 原子写入**
✅ **去抖（Debounce）**是性能关键：攒一批对话一次处理，省 token 省时间
✅ **原子写入**（临时文件 + rename）保证记忆文件永不损坏
✅ 记忆**按用户隔离**存储（`users/{userId}/memory.json`），保护隐私
✅ 下次对话时，**Top 15 事实 + 上下文**注入 system prompt，默默影响 Agent 行为
✅ 工程细节：token 计数降级、事实去重、缓存失效、user_id 跨线程捕获
