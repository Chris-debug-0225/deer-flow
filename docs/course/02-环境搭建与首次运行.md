# 第 02 章 环境搭建与首次运行

> 🎯 本章目标：把 deer-flow 在你自己的电脑上跑起来，和它完成第一次对话。这是后面所有章节的前置条件——跑都跑不起来，后面学架构就是空中楼阁。

## 2.2 环境要求（先对清单）

在动手之前，先把「装备」备齐。deer-flow 是个全栈项目（Python 后端 + Node.js 前端），所以依赖稍微多一点。

### 必装清单

| 依赖 | 版本要求 | 用途 | 怎么验证 |
|------|---------|------|---------|
| **Python** | 3.12+ | 后端语言 | `python --version` |
| **uv** | 任意 | Python 包管理器（替代 pip） | `uv --version` |
| **Node.js** | 22+ | 前端运行 | `node --version` |
| **pnpm** | 任意 | 前端包管理器 | `pnpm --version` |
| **Git** | 任意 | 拉代码 | `git --version` |
| **nginx** | 任意 | 反向代理（统一入口） | `nginx -v` |
| **Make** | 任意 | 执行 Makefile 命令 | `make --version` |

> 💡 **什么是 uv？** 它是一个用 Rust 写的、超快的 Python 包管理器（`pip` 的替代品）。deer-flow 后端用它来管理依赖。安装：macOS/Linux 执行 `curl -LsSf https://astral.sh/uv/install.sh | sh`，Windows 执行 `powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"`。

### 操作系统说明

| 系统 | 支持度 | 说明 |
|------|-------|------|
| **Linux + Docker** | ⭐⭐⭐⭐⭐ | 生产部署推荐 |
| **macOS** | ⭐⭐⭐⭐ | 开发体验好 |
| **Windows** | ⭐⭐⭐ | 必须**在 Git Bash 里运行**，cmd/PowerShell 不支持 |

> ⚠️ **Windows 用户特别注意**：deer-flow 的启动脚本是基于 bash 的，**原生 cmd.exe 和 PowerShell 不支持**。你需要安装 Git for Windows，然后在 **Git Bash** 里执行所有命令。WSL 不保证可用（部分脚本依赖 Git for Windows 的 `cygpath` 工具）。

### 硬件建议

| 部署场景 | 最低配置 | 推荐配置 |
|---------|---------|---------|
| 本地开发体验 | 4 核 8G | 8 核 16G |
| Docker 开发 | 4 核 8G + 25G 硬盘 | 8 核 16G |
| 长期服务 | 8 核 16G + 40G 硬盘 | 16 核 32G |

> 如果你用的是 OpenAI/DeepSeek 这种云端 API（不是本地跑模型），那 4 核 8G 完全够用。本地跑大模型的话，另算显存。

## 2.3 三步走：从零到跑起来

deer-flow 把复杂的配置过程封装成了一个**交互式向导**，核心就三步：

```
第 1 步：git clone      ← 下载代码
第 2 步：make setup     ← 配置向导（选模型、填 Key、生成 config.yaml）
第 3 步：make dev       ← 启动服务
```

我们一步步来。

---

### 第 1 步：克隆代码

```bash
git clone https://github.com/bytedance/deer-flow.git
cd deer-flow
```

> 如果你已经在这个项目的目录里（比如你就是在 deer-flow 仓库里看这份课件），跳过这步。

---

### 第 2 步：运行配置向导 `make setup`

```bash
make setup
```

这个命令会启动一个**交互式向导**（大约 2 分钟），它会一步步问你：

1. **选 LLM 提供方**：OpenAI / DeepSeek / 豆包 / Anthropic / Ollama（本地）…
2. **填 API Key**：你选的那个提供方的密钥
3. **选搜索工具**（可选）：Tavily / Jina / DuckDuckGo / 跳过
4. **选执行与安全偏好**：沙箱模式、是否允许 bash、是否允许写文件

向导结束后，它会自动帮你生成两个文件：

- **`config.yaml`** —— 主配置（模型、工具、沙箱、记忆等所有设置）
- **`.env`** —— 环境变量（你的 API Key 存在这里，**不要提交到 git**）

> 💡 **不想用向导？** 也可以手动配置：执行 `make config` 会从模板复制一份完整的 `config.yaml`，然后你自己改。但对小白来说，强烈推荐用 `make setup` 向导。

#### 几种典型的 LLM 配置选择

**选择 A：用 OpenAI（最省心，要花钱）**

在 `.env` 里：
```bash
OPENAI_API_KEY=sk-你的key
```

向导会在 `config.yaml` 生成：
```yaml
models:
  - name: gpt-4o
    display_name: GPT-4o
    use: langchain_openai:ChatOpenAI   # ← 用哪个 LangChain 适配器
    model: gpt-4o                       # ← 模型名
    api_key: $OPENAI_API_KEY            # ← 从环境变量读 key
    supports_vision: true               # ← 支持看图
```

**选择 B：用 DeepSeek（便宜，国内友好）**

`.env`：
```bash
DEEPSEEK_API_KEY=你的key
```

`config.yaml`：
```yaml
models:
  - name: deepseek-chat
    display_name: DeepSeek Chat
    use: langchain_openai:ChatOpenAI     # DeepSeek 兼容 OpenAI 接口
    model: deepseek-chat
    api_key: $DEEPSEEK_API_KEY
    base_url: https://api.deepseek.com/v1   # ← 关键：指向 DeepSeek 的地址
```

**选择 C：用本地 Ollama（完全免费，不吃 API 额度）**

先安装 [Ollama](https://ollama.com)，然后拉一个模型：
```bash
ollama pull qwen3:32b
```

`config.yaml`：
```yaml
models:
  - name: qwen3-local
    display_name: Qwen3 32B (Ollama)
    use: langchain_ollama:ChatOllama      # ← 注意：Ollama 要用专用适配器
    model: qwen3:32b
    base_url: http://localhost:11434      # ← 不要加 /v1 后缀
```

> 🔑 **关键概念**：`use` 字段告诉 deer-flow「用哪个 LangChain 适配器去连这个模型」。OpenAI 兼容的（DeepSeek、豆包）都用 `langchain_openai:ChatOpenAI`；Ollama 用 `langchain_ollama:ChatOllama`。这个机制我们在第 05 章「模型工厂」会深入讲。

---

### 第 3 步：启动服务 `make dev`

配置完成后，第一次启动前，建议先装依赖：

```bash
make install    # 安装后端 + 前端依赖 + git pre-commit 钩子
```

然后启动：

```bash
make dev
```

这个命令会同时拉起三个服务：

| 服务 | 端口 | 作用 |
|------|------|------|
| **Nginx** | 2026 | 统一入口（你访问这个） |
| **Gateway API** | 8001 | 后端 API + Agent 运行时 |
| **Frontend** | 3000 | Next.js 前端界面 |

启动成功后，你会看到类似这样的输出：

```
[nginx]  开始监听 :2026
[gateway] Uvicorn running on http://0.0.0.0:8001
[web]     ▲ Next.js  ready in 1200ms
✅ DeerFlow 已启动：http://localhost:2026
```

打开浏览器访问 **http://localhost:2026**，你应该能看到 deer-flow 的聊天界面。🎉

> 💡 **关于端口**：**始终访问 2026**（Nginx），不要直接访问 3000 或 8001。因为 Nginx 做了路由分发和跨域处理，直接访问子端口会有问题。

## 2.4 第一次对话：观察 Agent 在做什么

界面打开后，我们在输入框里输入一个简单的问题，比如：

> 帮我在工作目录创建一个 hello.txt 文件，内容是「Hello from deer-flow」

发送后，你会看到 Agent 的「思考过程」和「工具调用」实时展示出来。大致是这样：

```
🤔 思考：用户想创建一个文件，我需要用 write_file 工具。
🔧 调用工具：write_file(path="/mnt/user-data/workspace/hello.txt",
                        content="Hello from deer-flow")
✅ 工具返回：文件已写入
🗣️ 回答：好的，我已经在工作目录创建了 hello.txt，内容是「Hello from deer-flow」。
```

注意路径里的 `/mnt/user-data/workspace/` —— 这就是 deer-flow 的**虚拟路径**。Agent 看到的是这个统一的虚拟路径，但实际文件写到了你硬盘上的某个具体目录（比如 `backend/.deer-flow/users/default/threads/xxx/workspace/`）。

这个「虚拟路径 → 物理路径」的映射，是沙箱系统的核心，我们在第 08 章会深入讲。现在你只要知道：**Agent 创建的文件确实存在你电脑上**就行。

### 去哪里找 Agent 生成的文件？

默认情况下，文件在：
```
backend/.deer-flow/users/{用户id}/threads/{对话id}/user-data/workspace/
```

如果你没开认证（本地开发默认），用户 id 是 `default`，对话 id 是一串 UUID。你可以在 `.deer-flow/users/default/threads/` 下找到刚才的 `hello.txt`。

## 2.5 用 `make doctor` 做健康检查

如果跑起来之后感觉哪里不对（比如 Agent 不回话、工具用不了），用一个命令自检：

```bash
make doctor
```

它会检查：
- ✅ 配置文件是否存在、格式是否正确
- ✅ API Key 是否填了（不会泄露具体值，只检查是否存在）
- ✅ 模型是否能连通
- ✅ 必要的目录是否有写权限
- ✅ 端口是否被占用

并给出**可执行的修复建议**。这是排查问题的第一利器。

## 2.6 常见报错排查

下面列几个新手最容易踩的坑。

### ❌ 报错 1：`command not found: make`

**原因**：系统没有 `make` 命令。

**解法**：
- Windows：安装 [Git for Windows](https://git-scm.com/download/win) 后，在 **Git Bash** 里就有 make；或者装 [MSYS2](https://www.msys2.org/)。
- macOS：`xcode-select --install`
- Linux：`sudo apt install make`（Ubuntu/Debian）

### ❌ 报错 2：`ModuleNotFoundError: No module named 'xxx'`

**原因**：后端依赖没装全。

**解法**：
```bash
cd backend
uv sync          # 重新同步所有依赖
```

### ❌ 报错 3：`Port 2026/3000/8001 already in use`

**原因**：端口被别的程序占了。

**解法**：
```bash
# 先停掉之前可能没关干净的服务
make stop

# 看看是谁在占端口（以 8001 为例）
# macOS/Linux:
lsof -i :8001
# Windows:
netstat -ano | findstr :8001
```

找到占用进程的 PID 后，关掉它，再重新 `make dev`。

### ❌ 报错 4：启动了但 Agent 不回话 / 报 API 错误

**原因**：多半是 API Key 没配对，或者 `config.yaml` 里模型配置有问题。

**解法**：
1. 先 `make doctor` 看诊断
2. 检查 `.env` 里的 key 是否正确
3. 检查 `config.yaml` 里模型的 `base_url` 是否正确（OpenAI 兼容服务必须填对地址）
4. 看后端日志（Gateway 的终端输出），通常会有明确的错误信息

### ❌ 报错 5：Windows 下 `make dev` 各种报错

**原因**：在 cmd/PowerShell 里跑了。

**解法**：**一定、一定、一定要在 Git Bash 里运行**。打开 Git Bash，`cd` 到项目目录，再 `make dev`。

### ❌ 报错 6：前端能打开，但发消息转圈不响应

**原因**：前端连不上后端（Gateway 没起来，或 nginx 没起来）。

**解法**：
1. 看 `make dev` 的终端输出，确认三个服务（nginx/gateway/web）都打印了启动成功
2. 单独访问 `http://localhost:8001/health`，应该返回 `{"status":"ok"}` 之类
3. 如果 Gateway 没起来，看它的报错日志，多半是配置或依赖问题

## 2.7 启动模式速查表

deer-flow 有多种启动方式，新手用 `make dev` 就行，但了解一下全貌有助于后面调试：

| | 本地前台 | 本地后台 | Docker 开发 | Docker 生产 |
|---|---------|---------|------------|------------|
| **开发模式** | `make dev` | `make dev-daemon` | `make docker-start` | — |
| **生产模式** | `make start` | `make start-daemon` | — | `make up` |
| **停止** | `make stop` | `make stop` | `make docker-stop` | `make down` |

**新手建议**：开发阶段一直用 `make dev`（前台，能看到日志，方便调试）。等熟悉了再试 Docker。

## 2.8 项目目录结构初探

跑起来之后，我们看看项目根目录长什么样，建立一点感性认识（第 03 章会详细讲）：

```
deer-flow/
├── Makefile                  ← 你刚才敲的 make 命令都在这里定义
├── config.yaml               ← make setup 生成的主配置（★核心★）
├── config.example.yaml       ← 配置模板（参考用）
├── extensions_config.json    ← MCP 服务器和技能的开关
├── .env                      ← 你的 API Key（★别提交到 git★）
├── backend/                  ← 后端（Python，Agent 运行时在这里）
│   ├── packages/harness/deerflow/   ← Agent 框架核心代码（第 3-11 章讲这里）
│   ├── app/gateway/                 ← Gateway API（第 12 章讲）
│   └── docs/                        ← 后端文档
├── frontend/                 ← 前端（Next.js 网页）
├── skills/                   ← 技能目录（第 11 章讲）
│   ├── public/               ← 自带的技能
│   └── custom/               ← 你自己加的技能
└── .deer-flow/               ← 运行时数据（对话历史、生成的文件、记忆）
```

> 💡 现在只要记住两个最重要的文件：**`config.yaml`**（控制行为）和 **`.env`**（存密钥）。其他的我们后续逐个深入。

## 2.9 动手实验：换一个模型试试

完成上面的步骤后，做个小实验巩固一下：

1. 打开 `config.yaml`，在 `models:` 下再**加一个模型配置**（比如你本来配了 OpenAI，再加一个 DeepSeek）
2. 保存文件（**不需要重启服务**，deer-flow 会自动热加载配置）
3. 在浏览器里刷新页面，点界面上方的模型选择器，你应该能看到新加的模型
4. 切换到新模型，再问一次刚才的问题

如果你成功了，说明你已经掌握了「**改配置 → 生效**」这个最基本的操作循环。这也是后面所有定制化操作的基础。

> 🔑 **热加载**：deer-flow 的很多配置（模型、工具、技能）改了 `config.yaml` 后不需要重启，下次请求自动生效。但有些基础设施配置（数据库、沙箱类型）需要重启，第 04 章会详细说哪些能热加载、哪些不能。

## 2.10 本章小结

- ✅ deer-flow 需要 **Python 3.12+、uv、Node.js 22+、pnpm、nginx、Make**；Windows 必须**用 Git Bash**
- ✅ 三步跑起来：**`git clone` → `make setup` → `make dev`**
- ✅ 访问 **http://localhost:2026**（始终走 Nginx，别直接访问子端口）
- ✅ 出问题先用 **`make doctor`** 自检
- ✅ Agent 生成的文件在 **`.deer-flow/users/.../workspace/`** 下
- ✅ 改 `config.yaml` 加模型可以**热加载**，不用重启
