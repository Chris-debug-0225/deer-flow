## 8.2 概念铺垫：为什么 Agent 需要沙箱

回忆第 01 章：普通聊天机器人只能「说话」，Agent 能「动手」。但「动手」这件事有个大问题——

> ⚠️ **如果 Agent 执行了 `rm -rf /`（删光硬盘）怎么办？如果它读了你的密码文件怎么办？**

Agent 能跑代码、能写文件、能执行命令，这些能力如果不加约束，就是一个**安全炸弹**。沙箱（Sandbox）就是解决这个问题的。

### 沙箱是什么

沙箱是一个**隔离的执行环境**，给 Agent 一个「看起来完整、但实际受限」的电脑：

| 特性 | 沙箱提供了什么 |
|------|--------------|
| **独立文件系统** | Agent 有自己的工作目录，写文件不影响你的真实系统 |
| **受限执行** | 命令在隔离环境里跑，`rm -rf /` 也只删沙箱里的东西 |
| **按对话隔离** | 每个对话有自己的目录，A 对话的文件 B 对话看不到 |
| **统一接口** | 不管底层是本地还是 Docker，Agent 看到的接口都一样 |

> 💡 **类比**：沙箱就像给 Agent 一个「沙盘」——它可以在沙盘里随便堆城堡、挖沟渠，但弄不脏你家的地板。而且每个对话一个新沙盘，互不干扰。

### 没有/有沙箱的对比

**没有沙箱**（危险）：
```
Agent 决定写文件 → write_file("/etc/passwd", "hacked") → 改了你的系统文件！
```

**有沙箱**（安全）：
```
Agent 决定写文件 → write_file("/mnt/user-data/workspace/passwd", ...)
  → 虚拟路径被翻译成 → .deer-flow/users/.../workspace/passwd
  → 只写到了隔离的工作目录，系统文件毫发无损
```

## 8.3 Sandbox 抽象接口

deer-flow 把沙箱抽象成了一个接口（`sandbox/sandbox.py`），定义了所有沙箱都必须实现的 4 个核心方法（实际还有 glob/grep 等，核心是这几个）：

```python
# 文件：packages/harness/deerflow/sandbox/sandbox.py（简化）

from abc import ABC, abstractmethod

class Sandbox(ABC):
    """所有沙箱环境的抽象基类。"""

    def __init__(self, id: str):
        self._id = id

    @property
    def id(self) -> str:
        return self._id

    @abstractmethod
    def execute_command(self, command: str) -> str:
        """执行 bash 命令，返回输出。"""
        pass

    @abstractmethod
    def read_file(self, path: str) -> str:
        """读文件内容。"""
        pass

    @abstractmethod
    def write_file(self, path: str, content: str, append: bool = False) -> None:
        """写文件（覆盖或追加）。"""
        pass

    @abstractmethod
    def list_dir(self, path: str, max_depth=2) -> list[str]:
        """列目录内容。"""
        pass
```

### 为什么用抽象基类？

因为 deer-flow 支持多种沙箱实现（本地、Docker）。用抽象基类保证：**不管底层是哪种实现，Agent 调用的都是同样的 `read_file` / `write_file` 接口**。

```
Agent 调用 sandbox.read_file("/mnt/user-data/workspace/foo.txt")
                    │
                    ▼
        ┌───────────────────────┐
        │  这是哪种沙箱？        │
        └───────────────────────┘
           │              │
           ▼              ▼
   LocalSandbox      AioSandbox
   （读本地文件）    （读 Docker 容器里的文件）
           │              │
           ▼              ▼
   都返回文件内容    都返回文件内容
```

Agent 完全不用关心底层是本地还是 Docker——它只管调 `read_file`，沙箱实现负责翻译。这就是**多态**的威力，让代码和具体实现解耦。

## 8.4 ★核心★虚拟路径系统

这是沙箱系统最精妙的设计。理解了它，你就理解了 deer-flow 怎么做到「Agent 看到的路径」和「真实硬盘路径」分离。

### 问题：路径不一致

假设 Agent 要写一个文件 `result.txt`。
- 如果是 **Docker 沙箱**：文件在容器内的 `/mnt/user-data/workspace/result.txt`
- 如果是 **本地沙箱**：文件在你硬盘的 `backend/.deer-flow/users/default/threads/xxx/workspace/result.txt`

这俩路径完全不一样！如果 Agent 的代码（工具）直接写死路径，换一种沙箱就崩了。

### 解法：虚拟路径

deer-flow 约定：**Agent 永远只看到统一的虚拟路径 `/mnt/user-data/...`**，不管底层是什么沙箱。沙箱实现负责把虚拟路径翻译成真实路径。

```
Agent 眼里（统一的虚拟路径）：
/mnt/user-data/
├── workspace/    ← 工作目录（Agent 在这干活）
├── uploads/      ← 用户上传的文件
└── outputs/      ← 最终交付物
/mnt/skills/      ← 技能目录

            │
            │ 沙箱实现负责翻译
            ▼

本地沙箱的真实路径：
backend/.deer-flow/users/{userId}/threads/{threadId}/user-data/
├── workspace/
├── uploads/
└── outputs/
deer-flow/skills/        ← 技能目录真实位置

Docker 沙箱的真实路径：
（容器内挂载点，和虚拟路径一致，因为 Docker 直接 volume mount）
```

### 翻译机制：PathMapping

deer-flow 用 `PathMapping`（路径映射）描述「虚拟路径 → 真实路径」的对应关系：

```python
# 简化示意
PathMapping(
    container_path="/mnt/user-data/workspace",   # Agent 看到的虚拟路径
    local_path="backend/.deer-flow/.../workspace",  # 真实硬盘路径
    read_only=False,                              # 是否只读
)
```

当 Agent 调用 `write_file("/mnt/user-data/workspace/result.txt")` 时，沙箱会：

```
1. 收到虚拟路径：/mnt/user-data/workspace/result.txt
2. 查 PathMapping，发现 /mnt/user-data/workspace → 真实路径
3. 翻译成：backend/.deer-flow/.../workspace/result.txt
4. 在真实路径写文件
```

> 💡 **Docker 沙箱的例外**：Docker 沙箱里，虚拟路径和容器内真实路径是**一样的**（因为 Docker 用 volume mount 直接把宿主目录挂载到容器的 `/mnt/user-data`）。所以 AioSandbox 不需要翻译，但接受同样的虚拟路径——这保证了 Agent 代码的一致性。

### 为什么这样设计

这个设计的核心价值是**实现无关性**：

- 写 Agent 代码的人（工具实现），只用管虚拟路径 `/mnt/user-data/...`
- 切换沙箱类型（本地 ↔ Docker），Agent 代码**一行都不用改**
- 每个对话的文件天然隔离（threadId 不同 → 真实路径不同）

这也是第 03 章 `ThreadState` 里 `thread_data`（workspace_path / uploads_path / outputs_path）字段的用途——存的就是每个对话的真实路径。

## 8.5 两种沙箱实现对比

deer-flow 内置两种沙箱 Provider（沙箱的「供应商」）：

### LocalSandboxProvider（本地沙箱）

```
配置：sandbox.use: deerflow.sandbox.local:LocalSandboxProvider
```

- **工作方式**：直接在你的机器上读写文件、跑命令
- **隔离方式**：靠「不同对话用不同目录」实现软隔离
- **bash**：★默认禁用★（`allow_host_bash: false`）
- **适合**：本地开发、完全可信的环境
- **优点**：简单、快、不需要 Docker
- **缺点**：隔离弱（Agent 理论上能访问你整块硬盘）

### AioSandboxProvider（Docker 沙箱）

```
配置：sandbox.use: deerflow.community.aio_sandbox:AioSandboxProvider
```

- **工作方式**：每个任务在独立的 Docker 容器里执行
- **隔离方式**：容器级硬隔离（文件系统、进程、网络都隔离）
- **bash**：可用（在容器里跑，安全）
- **适合**：生产环境、不可信任务、需要跑任意代码
- **优点**：强隔离、安全
- **缺点**：需要 Docker、启动容器有开销

### 对比表

| 维度 | LocalSandboxProvider | AioSandboxProvider |
|------|---------------------|-------------------|
| 隔离强度 | 弱（目录级） | 强（容器级） |
| 需要 Docker | 否 | 是 |
| bash 可用 | 默认否 | 是 |
| 启动速度 | 快 | 较慢（建容器） |
| 适合场景 | 本地开发 | 生产/不可信 |

### 为什么本地模式默认禁用 bash

这是 deer-flow 的重要安全决策。本地模式下，如果允许 bash：
```
Agent 调用 bash("rm -rf ~")  → 直接删你用户目录！
Agent 调用 bash("cat ~/.ssh/id_rsa")  → 偷你私钥！
```

因为本地沙箱没有真正的隔离，bash 命令直接在你的机器上跑，威力等同于你自己敲。所以默认禁用，只有你**显式**设 `allow_host_bash: true` 才打开，并要求环境完全可信。

> 💡 **如果你想用 bash 但要安全**：切到 AioSandboxProvider（Docker 沙箱），bash 在容器里跑，怎么折腾都不影响宿主机。这是推荐做法。

## 8.6 沙箱工具的实现

第 07 章我们说工具是「函数」，沙箱工具也不例外。我们看 `write_file` 工具怎么实现（简化版）：

```python
# 文件：packages/harness/deerflow/sandbox/tools.py（简化）

from langchain.tools import tool
from deerflow.sandbox.sandbox_provider import get_sandbox_provider

@tool
def write_file_tool(
    path: str,
    content: str,
    append: bool = False,
) -> str:
    """写入文件。如果文件已存在，默认覆盖；append=True 时追加到末尾。

    Args:
        path: 文件路径（使用 /mnt/user-data/... 虚拟路径）
        content: 要写入的内容
        append: 是否追加（默认 False，覆盖）
    """
    # 第 1 步：拿到当前对话的沙箱实例
    sandbox = get_sandbox_provider().get(sandbox_id)

    # 第 2 步：调用沙箱的 write_file（沙箱内部会做路径翻译）
    sandbox.write_file(path, content, append=append)

    return f"成功写入 {len(content)} 个字符到 {path}"
```

注意：**工具函数本身不做路径翻译**，它把 `path` 原样传给 `sandbox.write_file`，由沙箱实现负责翻译。这就是「工具 → 沙箱接口 → 沙箱实现」的三层分离。

### 路径翻译发生在哪

在 LocalSandbox 的 `write_file` 里（简化）：

```python
class LocalSandbox(Sandbox):
    def write_file(self, path, content, append=False):
        # ★把虚拟路径翻译成真实路径★
        real_path = self._resolve_path(path)
        # /mnt/user-data/workspace/foo.txt
        #   → backend/.deer-flow/.../workspace/foo.txt

        # 确保目录存在
        Path(real_path).parent.mkdir(parents=True, exist_ok=True)

        # 写文件（覆盖或追加）
        mode = "a" if append else "w"
        with open(real_path, mode, encoding="utf-8") as f:
            f.write(content)
```

### 其他沙箱工具一览

| 工具名 | 接口方法 | 作用 |
|--------|---------|------|
| `bash` | `sandbox.execute_command(cmd)` | 执行命令 |
| `ls` | `sandbox.list_dir(path)` | 列目录（树形，最多 2 层） |
| `read_file` | `sandbox.read_file(path)` | 读文件（支持指定行范围） |
| `write_file` | `sandbox.write_file(path, content)` | 写/追加文件 |
| `str_replace` | 读 → 替换字符串 → 写 | 字符串替换（单次或全部） |
| `glob` | `sandbox.glob(path, pattern)` | 通配符查找文件 |
| `grep` | `sandbox.grep(path, pattern)` | 内容搜索 |

所有工具都遵循同样的模式：**接收虚拟路径 → 调沙箱接口 → 沙箱翻译并执行**。

## 8.7 沙箱的生命周期管理

沙箱不是凭空出现的，它有完整的生命周期，由 `SandboxMiddleware`（第 06 章中间件链第 3 个）管理：

```
对话开始
   │
   ▼
SandboxMiddleware.before_agent()
   → 调 provider.acquire(thread_id)
   → 获取（或创建）这个对话的沙箱实例
   → 把 sandbox_id 存到 ThreadState
   │
   ▼
Agent 执行（中间所有工具调用都用这个沙箱）
   │
   ▼
对话结束 / 沙箱释放
   → provider.release(sandbox_id)
   → Docker 沙箱会销毁容器；本地沙箱只是丢弃对象（文件保留）
```

### Provider 的四个生命周期方法

```python
class SandboxProvider(ABC):
    def acquire(self, thread_id) -> Sandbox:
        """获取沙箱（没有就创建）。"""

    def acquire_async(self, thread_id) -> Sandbox:
        """异步版（Docker 沙箱用，避免阻塞事件循环）。"""

    def get(self, sandbox_id) -> Sandbox:
        """按 id 拿已有的沙箱（工具调用时用，纯内存查找，快）。"""

    def release(self, sandbox_id):
        """释放沙箱。"""
```

注意 `acquire` 和 `get` 的区别：
- `acquire`：可能在创建沙箱（Docker 建容器），慢，只在对话开始调一次
- `get`：纯内存查找，快，每次工具调用都调

这种「**重获取（acquire）一次 + 轻量查询（get）多次**」的设计，平衡了初始化开销和调用效率。

### 本地沙箱的缓存

LocalSandboxProvider 用 **LRU 缓存**（最近最少使用）管理沙箱实例：

```python
class LocalSandboxProvider(SandboxProvider):
    def __init__(self, max_cached_threads=256):
        self._thread_sandboxes = OrderedDict()  # LRU 缓存
        self._max_cached_threads = max_cached_threads

    def acquire(self, thread_id):
        with self._lock:
            if thread_id in self._thread_sandboxes:
                # 缓存命中，移到最新（LRU）
                self._thread_sandboxes.move_to_end(thread_id)
                return self._thread_sandboxes[thread_id]
            # 缓存未命中，创建新的
            sandbox = self._create_thread_sandbox(thread_id)
            self._thread_sandboxes[thread_id] = sandbox
            # 超过上限，淘汰最久没用的
            if len(self._thread_sandboxes) > self._max_cached_threads:
                self._thread_sandboxes.popitem(last=False)
            return sandbox
```

为什么需要缓存？因为每次 `acquire` 都重建沙箱（重新算路径映射）有开销。缓存后，同一个对话的多次工具调用复用同一个沙箱实例。256 的上限防止长期运行时内存爆炸。

## 8.8 沙箱的安全防线

deer-flow 在沙箱安全上做了多层防护（defense in depth，纵深防御）：

### 防线 1：本地 bash 默认禁用
```
allow_host_bash: false（默认）→ bash 工具根本不加载
```
第 07 章 `get_available_tools()` 里有：
```python
if not is_host_bash_allowed(config):
    tool_configs = [t for t in tool_configs if not _is_host_bash_tool(t)]
```

### 防线 2：路径校验（防目录穿越）
Agent 传 `/mnt/user-data/workspace/../../etc/passwd` 这种路径想逃逸？沙箱会检测并拒绝。`tools.py` 里有专门的路径模式检测（`_DOTDOT_PATH_SEGMENT_PATTERN` 等），防止 `..` 穿越。

### 防线 3：SandboxAuditMiddleware（审计）
第 06 章中间件链里的 `SandboxAuditMiddleware`，会在执行 bash/文件操作前**记录审计日志**，所有敏感操作可追溯。

### 防线 4：工具错误处理
`ToolErrorHandlingMiddleware` 保证工具出错时不崩溃，转成错误消息让 Agent 继续处理（而不是整个 Agent 挂掉）。

### 防线 5：Docker 隔离（最强）
切到 AioSandboxProvider，所有操作在容器里，就算 Agent 想搞破坏也只能搞坏容器（随时可销毁重建）。

> 💡 **安全启示**：Agent 系统的安全不能靠一层防御，要**多层叠加**。即使一层失守，还有下一层兜底。这是生产级 Agent 和玩具 demo 的本质区别。

## 8.9 动手实验：观察虚拟路径映射

来做个实验，直观感受虚拟路径。确保你的 deer-flow 跑着，本地沙箱模式。

### 步骤 1：让 Agent 创建文件

在界面发消息：
> 在工作目录创建一个 test.txt，内容写「路径测试」

### 步骤 2：Agent 会调用 write_file

观察界面，Agent 会调用类似：
```
write_file(path="/mnt/user-data/workspace/test.txt", content="路径测试")
```
注意这个 `/mnt/user-data/workspace/` 就是**虚拟路径**。

### 步骤 3：去硬盘找真实文件

打开你的文件管理器，导航到：
```
backend/.deer-flow/users/default/threads/{一串UUID}/user-data/workspace/
```

你会看到真实的 `test.txt`！这就是「虚拟路径 → 真实路径」的映射结果。

### 步骤 4：让 Agent 读回来

再发消息：
> 读一下工作目录里的 test.txt

Agent 会调用 `read_file("/mnt/user-data/workspace/test.txt")`，沙箱翻译路径后读到真实文件，返回「路径测试」。

通过这个实验，你应该能直观看到：**Agent 操作的是虚拟路径，但文件确实存在于你硬盘的隔离目录里**。这就是沙箱的魔力。

## 8.10 本章小结

- ✅ **沙箱**是 Agent 的「电脑」，提供隔离的执行环境，防止 Agent 破坏宿主系统
- ✅ `Sandbox` 抽象接口定义了 **4 个核心方法**：execute_command / read_file / write_file / list_dir
- ✅ ★**虚拟路径系统**★：Agent 永远看 `/mnt/user-data/...`，沙箱负责翻译成真实路径，实现「接口统一 + 实现无关」
- ✅ 两种 Provider：**LocalSandboxProvider**（本地，弱隔离，bash 默认禁用）/ **AioSandboxProvider**（Docker，强隔离）
- ✅ 沙箱工具有统一模式：**接收虚拟路径 → 调沙箱接口 → 沙箱翻译并执行**
- ✅ 生命周期：`SandboxMiddleware` 在对话开始 `acquire`，工具调用时 `get`，结束后 `release`
- ✅ 安全是**多层纵深防御**：bash 默认禁 + 路径校验 + 审计日志 + 错误处理 + Docker 隔离
