## 9.2 概念铺垫：为什么一个 Agent 不够

到目前为止，我们讨论的都是「一个 Agent 干所有事」。但这有几个问题：

### 🤕 痛点 1：上下文污染

假设用户问：「**调研 A、B、C 三个竞品，对比它们的定价。**」

如果用一个 Agent 顺序做：
- 调研 A → A 的资料塞进上下文（占 3000 token）
- 调研 B → B 的资料塞进上下文（又 3000 token）
- 调研 C → C 的资料塞进上下文（又 3000 token）

到对比时，上下文里塞了三个竞品的所有细节，**又乱又长**，Agent 容易搞混，而且可能撑爆上下文窗口。

### 🤕 痛点 2：串行太慢

三个调研如果串行做，每个 2 分钟，总共 6 分钟。但如果能**并行**，理论上 2 分钟搞定。

### 解法：子代理

deer-flow 的解法是**子代理（Subagent）**：

```
主 Agent（Lead Agent）收到任务：「调研 A、B、C 三个竞品」
   │
   │ 拆解成 3 个子任务
   ├─ task("调研竞品 A")  → 子代理1（独立上下文）→ 返回 A 的摘要
   ├─ task("调研竞品 B")  → 子代理2（独立上下文）→ 返回 B 的摘要  （并行！）
   └─ task("调研竞品 C")  → 子代理3（独立上下文）→ 返回 C 的摘要
   │
   │ 主 Agent 只收到 3 份精简摘要
   ▼
   综合对比，输出最终报告
```

### 子代理的两大价值

| 价值 | 说明 |
|------|------|
| **上下文隔离** | 每个子代理有自己的独立上下文，调研过程的细节不污染主 Agent，只返回精简结果 |
| **并行执行** | 多个子代理同时跑，大幅缩短复杂任务的总耗时 |

> 💡 **类比**：主 Agent 是「项目经理」，子代理是「组员」。经理不自己干所有活，而是把任务拆给几个组员并行做，每个人只汇报结果（不汇报过程细节），经理最后汇总。这就是 deer-flow 处理复杂任务的秘诀。

## 9.3 子代理的注册：内置 + 自定义

deer-flow 的子代理分两类，都通过「注册表（Registry）」管理。

### 内置子代理

deer-flow 自带两个子代理（`subagents/builtins/`）：

| 子代理名 | 用途 | 工具 |
|---------|------|------|
| `general-purpose` | 通用任务（研究、分析、文件操作…） | 几乎所有工具（除了 task 自己） |
| `bash` | 命令执行专家（git、构建、测试、部署） | 专注命令行操作 |

```python
# 文件：subagents/builtins/__init__.py（简化）

BUILTIN_SUBAGENTS = {
    "general-purpose": GENERAL_PURPOSE_CONFIG,   # 通用子代理
    "bash": BASH_AGENT_CONFIG,                    # 命令专家
}
```

注意：`bash` 子代理只在**允许 bash 时**才暴露。第 08 章说过本地模式默认禁用 bash，所以本地模式默认看不到 `bash` 子代理。

### 自定义子代理

你可以在 `config.yaml` 的 `custom_agents` 段落定义自己的子代理：

```yaml
# config.yaml（示意）
custom_agents:
  frontend-expert:
    description: "前端开发专家，擅长 React/CSS"
    system_prompt: "你是一个资深前端工程师..."
    tools: ["read_file", "write_file", "web_search"]   # 限定工具
    model: gpt-4o                                        # 可用不同模型
    max_turns: 20
    timeout_seconds: 300
```

这样主 Agent 就多了一个 `frontend-expert` 类型的子代理可调。

### 注册表的解析逻辑

`registry.py` 的 `get_subagent_config(name)` 负责按优先级解析子代理配置：

```
解析顺序：
1. 先查内置子代理（general-purpose / bash）
2. 查不到 → 查 config.yaml 的 custom_agents
3. 应用 config.yaml 的 per-agent 覆盖（timeout、model 等）
```

```python
# 文件：subagents/registry.py（简化）

def get_subagent_config(name, *, app_config=None):
    """按优先级解析子代理配置。"""
    # Step 1: 内置
    config = BUILTIN_SUBAGENTS.get(name)
    if config is None:
        # Step 2: 自定义
        config = _build_custom_subagent_config(name, app_config=app_config)
    if config is None:
        return None   # 都没有，这个子代理不存在

    # Step 3: 应用 per-agent 覆盖
    # 比如全局设了 timeout=600，但 agents.general-purpose.timeout=300
    # 则 general-purpose 用 300
    subagents_config = _resolve_subagents_app_config(app_config)
    agent_override = subagents_config.agents.get(name)
    if agent_override:
        config = apply_overrides(config, agent_override)

    return config
```

这种「**内置默认 → 自定义 → 逐项覆盖**」的三层配置层叠，让你既能开箱即用，又能精细定制。

## 9.4 ★核心★SubagentExecutor 执行引擎

这是子代理系统的心脏。它负责「**接收 task 工具调用 → 在后台运行子代理 → 轮询状态 → 返回结果**」。

### 执行模型：线程池 + 后台任务

子代理执行的关键挑战是：**主 Agent 在等子代理结果时不能干别的，但子代理又可能跑很久**。

deer-flow 的解法是**后台线程池 + 轮询**：

```python
# 文件：subagents/executor.py（核心结构简化）

# 两个线程池
_scheduler_pool = ThreadPoolExecutor(max_workers=3, thread_name_prefix="subagent-scheduler-")

# 全局存储：所有正在跑/跑完的子代理结果
_background_tasks: dict[str, SubagentResult] = {}
_background_tasks_lock = threading.Lock()
```

### 为什么要两个池

- **调度池（scheduler_pool）**：负责「编排」——启动子代理、管理生命周期
- **执行池**：实际跑子代理的 Agent 循环

分开是为了避免「编排」和「执行」互相阻塞。

### SubagentResult：结果容器

每个子代理任务有一个 `SubagentResult` 对象，记录它的状态和结果：

```python
@dataclass
class SubagentResult:
    """子代理执行结果。"""
    task_id: str                    # 任务唯一 ID
    trace_id: str                   # 追踪 ID（关联父子日志）
    status: SubagentStatus          # 状态：PENDING/RUNNING/COMPLETED/FAILED/...
    result: str | None              # 最终结果文本
    error: str | None               # 错误信息
    started_at: datetime            # 开始时间
    completed_at: datetime          # 完成时间
    ai_messages: list[dict]         # 子代理产生的 AI 消息
    token_usage_records: list       # token 消耗记录
    cancel_event: threading.Event   # 取消信号

class SubagentStatus(Enum):
    PENDING = "pending"         # 排队中
    RUNNING = "running"         # 执行中
    COMPLETED = "completed"     # 成功完成
    FAILED = "failed"           # 失败
    CANCELLED = "cancelled"     # 被取消
    TIMED_OUT = "timed_out"     # 超时
```

### 状态转换

```
PENDING ──启动──▶ RUNNING ──┬──成功──▶ COMPLETED
                             ├──失败──▶ FAILED
                             ├──取消──▶ CANCELLED
                             └──超时──▶ TIMED_OUT
```

`try_set_terminal` 方法保证**终态只设置一次**——即使「超时检测」和「执行线程」同时想设置状态，只有第一个生效：

```python
def try_set_terminal(self, status, *, result=None, error=None, ...):
    """设置终态，只生效一次（防竞态）。"""
    if not status.is_terminal:
        raise ValueError(f"{status} 不是终态")
    with self._state_lock:          # 加锁
        if self.status.is_terminal:  # 已经是终态了
            return False             # 不再修改
        # 第一次设终态
        self.status = status
        self.result = result
        ...
        return True
```

这种**「first-write-wins」的终态保护**是并发编程的常见模式，防止竞态条件导致状态混乱。

## 9.5 完整调用链：task 工具 → 执行 → 返回

我们把整个流程串起来。主 Agent 决定调用子代理时，会调用 `task` 工具（第 07 章提过）。

```
① 主 Agent 的 LLM 决定：调用 task 工具
   task(description="调研竞品A", prompt="...", subagent_type="general-purpose")
       │
       ▼
② task_tool 函数执行
   → 创建 SubagentResult（status=PENDING）
   → 提交到调度线程池（后台执行）
   → 返回 task_id 给主 Agent
       │
       ▼
③ 后台：调度池启动子代理
   → 用 subagent_type 找到子代理配置（SubagentConfig）
   → 构造一个独立的 Agent（create_agent）
   → 在执行池里跑这个子 Agent
   → 状态变为 RUNNING
   → 通过 SSE 推送 task_running 事件给前端
       │
       ▼
④ 主 Agent：轮询等待结果
   → 每 5 秒查一次 SubagentResult.status
   → 如果还在 RUNNING，继续等
   → 如果变成终态（COMPLETED/FAILED/...），拿到结果
       │
       ▼
⑤ 后台：子代理执行完毕
   → 状态变为 COMPLETED
   → result 字段填入子代理的最终回答
   → 推送 task_completed 事件
       │
       ▼
⑥ 主 Agent 收到结果
   → task 工具返回 result 文本
   → 作为 ToolMessage 加入主 Agent 的上下文
   → 主 Agent 继续下一步（汇总/派新任务/回答用户）
```

### 关键设计：为什么用轮询而不是回调

你可能会问：为什么不直接 `await` 子代理完成，而要轮询？

因为主 Agent 本身在一个 **同步的 ReAct 循环**里（LLM 调用 → 工具执行 → LLM 调用），它不能阻塞等待。子代理在后台异步跑，主 Agent 通过**轮询 + 超时**机制定期检查。这样：
- 主 Agent 不会被一个慢子代理卡死
- 可以实现超时取消
- 可以通过 SSE 实时推送进度给前端

## 9.6 并发控制与限制

子代理虽好，但不能无限开——资源有限，太多并发会拖垮系统。deer-flow 有多层控制。

### 限制 1：每轮最多 3 个并发

```python
MAX_CONCURRENT_SUBAGENTS = 3
```

主 Agent 在**一次回复**里，最多只能发起 3 个 task 工具调用。这由 `SubagentLimitMiddleware`（第 06 章中间件链）强制执行：

```python
# 简化示意：SubagentLimitMiddleware
class SubagentLimitMiddleware(AgentMiddleware):
    def __init__(self, max_concurrent=3):
        self.max_concurrent = max_concurrent

    def after_model(self, state, runtime):
        """LLM 返回后，检查 task 工具调用数量。"""
        last_msg = state["messages"][-1]
        task_calls = [c for c in last_msg.tool_calls if c["name"] == "task"]

        if len(task_calls) > self.max_concurrent:
            # ★超过限制，截断多余的★
            # 比如 LLM 发了 5 个 task，只保留前 3 个
            last_msg.tool_calls = task_calls[:self.max_concurrent]
            logger.warning("截断了 %d 个超限的 task 调用", len(task_calls) - self.max_concurrent)
```

### 多批执行策略

如果任务需要超过 3 个子代理怎么办？系统提示词会引导 LLM **分批执行**：

```
（第 06 章的子代理提示词片段）
假设 MAX_CONCURRENT=3，LLM 被引导：
- 识别到 6 个子任务
- 因为限制是 3/轮
- 第 1 轮：发起子任务 1-3（并行）→ 等结果
- 第 2 轮：发起子任务 4-6（并行）→ 等结果
- 第 3 轮：综合所有结果，输出最终答案
```

这是一种**「分批并行 + 滚动汇总」**的策略，兼顾了并行效率和资源限制。

### 限制 2：超时保护

每个子代理有**超时**（默认 15 分钟）：

```python
# 子代理执行时启动一个定时器
# 超时后强制设置 status=TIMED_OUT
# 子代理的 cancel_event 被设置，执行循环检测到后退出
```

超时保护防止「一个卡死的子代理永远占着资源」。

### 限制 3：上下文隔离

每个子代理有**全新的 ThreadState**——它看不到主 Agent 的完整历史，只看到：
- 主 Agent 给它的 `prompt`（任务描述）
- 它自己的系统提示词
- 它自己的工具调用历史

这保证了：
- 子代理上下文干净、聚焦（不被主 Agent 的历史干扰）
- 主 Agent 的上下文也不会被子代理的细节撑爆（只收精简结果）

> 💡 第 06 章提到「子代理的 checkpointer=False」——因为子代理是一次性的，不需要持久化状态，跑完就结束，所以不继承父 Agent 的检查点。

## 9.7 SSE 事件：实时进度推送

子代理执行过程中，会通过 SSE 向前端推送实时事件，让用户看到进度：

| 事件 | 时机 | 含义 |
|------|------|------|
| `task_started` | 子代理启动 | 「我开始干活了」 |
| `task_running` | 执行中（定期） | 「还在干，进度 X%」 |
| `task_completed` | 成功完成 | 「干完了，结果是…」 |
| `task_failed` | 失败 | 「出错了：…」 |
| `task_timed_out` | 超时 | 「超时了」 |

前端收到这些事件后，会渲染成「子代理卡片」，显示每个子代理的状态和结果。这就是为什么你在界面里能看到多个子代理并行工作的实时进度。

## 9.9 动手实验：体验子代理

（本实验需要 `subagent_enabled: true`，在运行时配置里开启，通常通过界面的「Pro/Ultra 模式」切换）

### 步骤 1：开启子代理模式

在 deer-flow 界面，把执行模式切到支持子代理的档位（如「Ultra」）。这会设置 `subagent_enabled=True`。

### 步骤 2：给一个可拆分的复杂任务

输入：
> 请分别搜索 Python、JavaScript、Rust 三种语言的最新版本特性，然后对比它们。

### 步骤 3：观察并行执行

如果子代理生效，你会看到：
- 主 Agent 拆解出 3 个子任务
- 3 个子代理卡片**同时**出现在界面上
- 每个卡片显示 `task_running` → `task_completed`
- 主 Agent 最后给出对比结果

### 步骤 4：对比不开子代理

切回普通模式，问同样的问题。你会看到 Agent **串行**地一个个搜索，速度明显慢，而且中间过程更杂乱。

> 💡 通过对比，你能直观感受到「**上下文隔离 + 并行执行**」的价值——这正是子代理系统设计的核心动机。

## 9.10 本章小结

✅ **子代理**解决两大痛点：**上下文隔离**（不污染主 Agent）+ **并行执行**（加速复杂任务）
✅ 内置两个子代理：**`general-purpose`（通用）/ `bash`（命令专家）**，可在 config.yaml 加自定义
✅ 注册表按优先级解析：**内置 → 自定义 → per-agent 覆盖**
✅ `SubagentExecutor` 用**线程池 + 后台任务 + 轮询**实现并发：task 工具 → 后台执行 → 轮询状态 → 返回结果
✅ 并发控制：**每轮最多 3 个**（SubagentLimitMiddleware 截断）、**15 分钟超时**、**全新独立上下文**
✅ 超过 3 个用**分批并行**策略，SSE 实时推送 `task_started/running/completed` 事件
✅ `SubagentResult.try_set_terminal` 用「**first-write-wins**」防止并发竞态
