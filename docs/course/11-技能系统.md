## 11.2 概念铺垫：技能 vs 工具

第 07 章我们学了「工具」——Agent 的「手」。那「技能」又是什么？它和工具有什么区别？

### 工具：一个动作

工具是**一个可调用的函数**，解决「**做什么动作**」：

```
工具 web_search：执行一次搜索
工具 write_file：写一个文件
工具 bash：跑一条命令
```

工具是「原子动作」，粒度小。

### 技能：一套方法论

技能是**一套完整的方法论/工作流**，解决「**怎么完成一类任务**」。比如「深度研究」这个技能，它不是一个动作，而是一套流程：

```
技能 deep-research：
  第 1 阶段：广度探索（搜 3-5 个角度）
  第 2 阶段：深度挖掘（针对每个角度深挖）
  第 3 阶段：交叉验证（多源验证）
  第 4 阶段：综合成报告
```

### 类比

| | 工具 | 技能 |
|--|------|------|
| 是什么 | 一个动作（锤子、锯子） | 一套方法论（「怎么做一把椅子」的完整流程） |
| 粒度 | 小，原子 | 大，复合 |
| 形式 | Python 函数 | **Markdown 文档** |
| 例子 | `write_file`、`web_search` | 「深度研究」「PPT 制作」「数据分析」 |

> 💡 **关键区别**：工具是「代码」，技能是「**知识/方法论**」。技能本质上是给 Agent 看的一份「操作手册」，告诉它「遇到这类任务，按这个流程做」。而工具是它执行流程时要调用的具体动作。

### 为什么用 Markdown 而不是代码

这是 deer-flow 最巧妙的设计之一。技能用 **Markdown 文档**定义，而不是 Python 代码：

| 方式 | 优点 | 缺点 |
|------|------|------|
| 用代码定义技能 | 执行力强 | 写技能要会编程；难修改；难分享 |
| **用 Markdown 定义技能**（deer-flow 的做法） | ✅ 不会编程也能写；✅ 改起来像编辑文档；✅ 一个文件就能分享 | 依赖 LLM 理解能力 |

因为现代 LLM 能很好地理解自然语言指令，所以「**用 Markdown 写方法论，让 LLM 读完后照着做**」是完全可行的。这大大降低了「给 Agent 加能力」的门槛——你只要会写文档，就能给 Agent 加技能。

## 11.3 SKILL.md 的格式

每个技能是一个目录，里面有一个 `SKILL.md` 文件。格式如下：

```
skills/public/deep-research/
└── SKILL.md
```

`SKILL.md` 的内容分两部分：**YAML 元数据** + **Markdown 正文**。

### 完整示例（以 deep-research 为例）

```markdown
---
name: deep-research
description: Use this skill for ANY question requiring web research. 
  Trigger on queries like "what is X", "compare X and Y", "research X".
  Provides systematic multi-angle research methodology.
---

# Deep Research Skill

## Overview
This skill provides a systematic methodology for conducting thorough web research.

## When to Use This Skill
- User asks "what is X", "explain X", "research X"
- User wants in-depth understanding
- Creating content that needs real-world data

## Research Methodology

### Phase 1: Broad Exploration
Start with broad searches to understand the landscape:
1. Search for the main topic
2. Identify key subtopics
3. Map different perspectives

### Phase 2: Deep Dive
For each subtopic identified:
- Search 2-3 specific queries
- Read top results thoroughly
- Note key facts and data points

### Phase 3: Synthesis
Combine findings into a coherent answer...
```

### YAML 元数据字段

只有 `name` 和 `description` 是必须的：

| 字段 | 作用 | 重要性 |
|------|------|--------|
| `name` | 技能名（唯一标识，斜杠激活用） | ★必须 |
| `description` | 技能说明（**LLM 靠它判断何时用**） | ★必须 |
| `license` | 许可证 | 可选 |
| `allowed-tools` | 这个技能允许用哪些工具 | 可选 |

> ⚠️ **description 极其重要**！它和工具的 description 一样，是 LLM 决定「该不该用这个技能」的依据。写得好，LLM 会在合适时机主动用；写得差，LLM 永远想不起用它。

### 正文：方法论

正文是给 LLM 看的「操作手册」，可以包含：
- **使用时机**（When to use）
- **方法论/流程**（分阶段的步骤）
- **最佳实践**（Best practices）
- **示例**（Examples）
- **注意事项**（Caveats）

写正文的原则：**写给一个聪明但不了解这个领域的新员工看**。详细、具体、可执行。

## 11.4 ★核心★渐进式加载

这是技能系统省 token 的关键设计。

### 问题：全加载太贵

deer-flow 自带 20+ 个技能（deep-research、ppt-generation、data-analysis…），每个 SKILL.md 可能几千字。如果把所有技能的**完整正文**都塞进每次对话的 system prompt，会浪费大量 token（几万 token 光是技能文本）。

### 解法：两阶段加载

deer-flow 用「**元数据先行，正文按需**」的策略：

```
┌─────────────────────────────────────────────────────┐
│ 阶段 1：系统提示词只注入「技能清单」（省 token）    │
│                                                      │
│ system prompt 里有：                                 │
│   可用技能：                                          │
│   - deep-research: 深度研究方法 (/mnt/skills/...)    │  ← 只有名字 + description
│   - ppt-generation: PPT 制作 (/mnt/skills/...)       │  ← 没有完整正文！
│   - data-analysis: 数据分析 (/mnt/skills/...)        │
│                                                      │
│ Agent 知道「有这些技能」，但不知道具体怎么做         │
└─────────────────────────────────────────────────────┘
                          │
                          │ 当 Agent 判断「这个任务需要用 data-analysis」时
                          ▼
┌─────────────────────────────────────────────────────┐
│ 阶段 2：按需加载完整 SKILL.md                        │
│                                                      │
│ Agent 用 read_file("/mnt/skills/.../data-analysis/   │
│   SKILL.md") 读取完整正文                            │
│ → 现在 Agent 知道数据分析的完整流程                  │
│ → 照着流程执行                                       │
└─────────────────────────────────────────────────────┘
```

**效果**：平时只占几百 token（清单），真正要用某个技能时才加载它的几千字正文。这就是「**渐进式加载**」——不用的不加载，省下大量 token。

> 💡 这就像人查参考书——你不需要把整本《烹饪大全》背下来（全加载），只要知道「有这本书，在哪个书架」（元数据），真要做菜时再去翻（按需加载）。

## 11.5 斜杠激活：`/skill-name`

除了「Agent 自己判断要不要用技能」（自动激活），用户也可以**手动强制激活**某个技能——用斜杠命令。

### 用法

在消息开头写 `/技能名 任务`：

```
/data-analysis 分析 uploads/sales.csv 里的销售趋势
/ppt-generation 做一份关于 AI 趋势的 10 页 PPT
/deep-research 调研一下 LangGraph 这个框架
```

### 工作原理

这是第 06 章中间件链里的 `SkillActivationMiddleware` 干的事：

```
用户输入：/data-analysis 分析这个 CSV
       │
       ▼
SkillActivationMiddleware.before_model()
   │
   ├─ ① 检测：消息是否以 /skill-name 开头？
   │     是 → 提取出 skill-name = "data-analysis"
   │
   ├─ ② 校验：这个技能存在吗？启用了吗？
   │     在当前 Agent 的白名单里吗？
   │
   ├─ ③ 加载：从磁盘读取 data-analysis/SKILL.md 的完整正文
   │
   └─ ④ 注入：把完整正文作为「隐藏上下文」注入当前这一轮
              （只在当前轮生效，不污染后续）
       │
       ▼
LLM 看到：用户任务 + data-analysis 的完整方法论
       → 按照 SKILL.md 的流程执行
```

### 斜杠激活 vs 自动激活

| 方式 | 触发者 | 优点 |
|------|--------|------|
| **自动激活** | Agent 自己判断 | 用户不用记技能名 |
| **斜杠激活** `/skill-name` | 用户手动 | ✅ 强制生效；✅ 省去 Agent 判断；✅ 只在当前轮加载（更省 token） |

斜杠激活适合「**我知道我要干什么，直接告诉 Agent 用哪套方法**」的场景。

### 保留命令

有些 `/` 开头的是系统命令，不是技能，`SkillActivationMiddleware` 会排除它们：

| 命令 | 含义 |
|------|------|
| `/new` | 新对话 |
| `/help` | 帮助 |
| `/status` | 状态 |
| `/models` | 模型列表 |
| `/memory` | 记忆 |

所以你不用担心你的技能名和这些冲突。

## 11.6 技能的存储与发现

### 目录结构

```
skills/
├── public/                          ← 自带技能（跟着仓库走）
│   ├── deep-research/
│   │   └── SKILL.md
│   ├── ppt-generation/
│   │   └── SKILL.md
│   ├── data-analysis/
│   │   └── SKILL.md
│   └── ...（20+ 个）
│
└── custom/                          ← 你的自定义技能（gitignore，不入库）
    └── my-skill/
        └── SKILL.md
```

- `public/`：deer-flow 官方自带的，随代码更新
- `custom/`：你自己的，不入 git（可以放私有/定制技能）

### 发现机制

`load_skills()` 会**递归扫描** `skills/{public,custom}` 目录，找到所有 `SKILL.md`：

```python
# 简化示意：skills/storage.py
def load_skills(enabled_only=True):
    """扫描技能目录，加载所有 SKILL.md。"""
    skills = []
    for category_dir in [public_dir, custom_dir]:
        for skill_dir in category_dir.iterdir():
            skill_md = skill_dir / "SKILL.md"
            if skill_md.exists():
                skill = parse_skill(skill_md)
                if enabled_only and not is_enabled(skill.name):
                    continue
                skills.append(skill)
    return skills
```

### 启用状态

技能的「启用/禁用」状态存在 `extensions_config.json`（第 04 章讲过）：

```json
{
  "skills": {
    "deep-research": {"enabled": true},
    "ppt-generation": {"enabled": false},
    "data-analysis": {"enabled": true}
  }
}
```

你在界面 Settings 里开关技能，改的就是这个文件。禁用的技能不会被加载，也不会出现在系统提示词的清单里。

## 11.8 动手实验：写一个自己的技能

我们来写一个实用的技能：「会议纪要整理」。

### 步骤 1：创建技能目录

在 `skills/custom/` 下建目录（如果没有 `custom/` 就建一个）：

```
skills/custom/meeting-notes/
└── SKILL.md
```

### 步骤 2：写 SKILL.md

```markdown
---
name: meeting-notes
description: 整理会议记录、对话录音转写为结构化的会议纪要。
  当用户提供会议内容、对话记录、或希望整理讨论要点时使用。
  输出包含议题、讨论要点、决议、待办事项的结构化纪要。
---

# Meeting Notes Skill（会议纪要整理技能）

## 何时使用
- 用户提供了一段会议/讨论的记录（文字或转写）
- 用户说「帮我整理一下这次会议」
- 用户想从杂乱的对话里提取要点

## 工作流程

### 第 1 步：通读理解
仔细阅读用户提供的全部会议内容，理解：
- 这是什么会议（主题、参与者）
- 主要讨论了哪些议题

### 第 2 步：结构化提取
按以下结构整理（如果没有某项就省略）：

#### 会议基本信息
- 主题、时间、参与人

#### 讨论议题与要点
按议题分组，每个议题下列出：
- 关键观点（谁说了什么）
- 分歧点（如有）

#### 决议
会议达成的明确结论

#### 待办事项（Action Items）
- 任务 | 负责人 | 截止时间（如提及）

### 第 3 步：格式化输出
用清晰的 Markdown 格式输出，使用：
- 二级/三级标题分节
- 列表罗列要点
- **加粗**关键决策和负责人
- 待办事项用 `- [ ]` 复选框格式

## 注意事项
- 忠于原文，不要编造没讨论过的内容
- 区分「明确决议」和「倾向性意见」
- 不确定的地方标注「（需确认）」
- 如果原文太杂乱，可以按时间顺序先理清脉络
```

### 步骤 3：启用技能

在界面的 Settings > Skills 里找到 `meeting-notes`，打开开关（或在 `extensions_config.json` 加 `"meeting-notes": {"enabled": true}`）。

### 步骤 4：测试

新开对话，输入：
> /meeting-notes 帮我整理这段会议：
> 张三说我们下个月要上线新功能，李四问人手够不够，王五说他可以借调两个人。最后决定下周三再开会定排期。

Agent 会读取你的 SKILL.md，按照你定义的结构化流程输出一份漂亮的会议纪要。

### 这个实验的意义

你刚刚用**一个 Markdown 文件**，就给 Agent 加了一种全新的能力——完全没写代码！这就是技能系统的威力。第 14 章实战会结合工具 + 技能做更完整的扩展。

## 11.9 技能与提示词工程

写好技能，本质上就是**提示词工程（Prompt Engineering）**。几个写好技能的秘诀：

### 秘诀 1：description 要写清「触发条件」

```yaml
# ❌ 差
description: 帮助做研究

# ✅ 好
description: 当用户问"什么是X""对比X和Y""调研X"，或任何需要
  网络信息的任务时使用。提供系统化的多角度研究方法。
```

把「**什么情况下该用**」写具体，LLM 才能准确触发。

### 秘诀 2：流程要分阶段、可执行

```markdown
# ❌ 差（太笼统）
做好研究，多搜搜，然后总结。

# ✅ 好（分阶段、具体）
### 阶段 1：广度探索
- 搜索主题，了解全貌
- 识别 3-5 个子话题

### 阶段 2：深度挖掘
- 每个子话题搜 2-3 次具体查询
- 阅读排名靠前的结果
```

### 秘诀 3：给示例

LLM 看示例学得最快。在技能里多写「**Example**」：

```markdown
### 示例
主题：「AI 在医疗的应用」
搜索查询：
- "AI 医疗应用 2024"
- "人工智能 医学诊断"
- "医疗 AI 市场趋势"
```

### 秘诀 4：写注意事项（边界）

告诉 LLM「**不要做什么」**和「什么情况要小心」：

```markdown
## 注意事项
- 不要只搜一次就下结论
- 多源交叉验证重要数据
- 标注信息的时效性
```

> 💡 这些秘诀不只适用于写技能，写任何 Agent 的系统提示词都适用。技能系统本质上就是「**把提示词工程模块化、可复用、可组合**」。

## 11.10 本章小结

- ✅ **技能（Skill）** = 一套方法论/工作流，用 **Markdown 文档**定义（不是代码），是 Agent 的「领域能力」
- ✅ 技能 vs 工具：**技能是「知识/方法论」，工具是「动作」**；技能告诉 Agent「怎么做一类任务」，工具是它执行时调用的动作
- ✅ SKILL.md = **YAML 元数据（name + description）+ Markdown 正文（方法论）**
- ✅ ★**渐进式加载**★：平时只注入技能清单（省 token），用到时才 read_file 加载完整正文
- ✅ **斜杠激活** `/skill-name`：用户手动强制激活，SkillActivationMiddleware 处理
- ✅ 技能存 `skills/{public,custom}/`，启用状态在 `extensions_config.json`
- ✅ deer-flow 自带 20+ 技能（研究、PPT、数据分析…），开箱即用
- ✅ 写好技能 = **提示词工程**：description 写清触发条件、流程分阶段可执行、多给示例、写注意事项
