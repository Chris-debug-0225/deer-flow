# 第 04 章 配置系统详解

> 🎯 本章目标：搞懂 `config.yaml` 这个「控制面板」。它是驱动整个 deer-flow 的中枢——改一个字段，就能换模型、加工具、开关记忆。本章我们把它的每个核心段落讲清楚，并理解背后的「加载优先级、热重载、环境变量替换」三大机制。

## 4.2 概念铺垫：为什么用「配置驱动」

先想一个问题：如果不用配置文件，要换一个模型，你会怎么做？

最直觉的方式是**改代码**：
```python
# 直接在代码里写死
model = ChatOpenAI(model="gpt-4o", api_key="sk-xxx")
```

这有三个大问题：
1. **不灵活**：换模型要改代码、重新部署
2. **不安全**：API Key 写死在代码里，容易泄露（提交到 git 就完了）
3. **不可复用**：你的代码只能给特定的人用，别人换不了

deer-flow 的解法是**配置驱动（Configuration-Driven）**：

> 把所有「可能变化的东西」（用什么模型、开哪些工具、怎么存记忆）都抽到一个 **YAML 文件**里，代码只负责「读配置 → 按配置执行」。

这样：
- 换模型？改 YAML，不改代码 ✅
- API Key？放在独立的 `.env` 文件，YAML 里用 `$OPENAI_API_KEY` 引用 ✅
- 不同人不同配置？每人一份 `config.yaml`，代码完全一样 ✅

> 💡 **核心认知**：`config.yaml` 是 deer-flow 的「大脑配置」。你 80% 的定制化工作，都是改这个文件，而不是改代码。

---

## 4.3 两个配置文件各管什么

deer-flow 有**两个**核心配置文件，分工明确：

| 文件 | 管什么 | 改动频率 |
|------|--------|---------|
| **`config.yaml`** | 模型、工具、沙箱、记忆、摘要等**系统行为** | 经常改 |
| **`extensions_config.json`** | MCP 服务器开关、技能启用状态 | 偶尔改（常通过界面改） |
| **`.env`** | 所有 API Key（密钥） | 设一次就少改 |

### 它们的关系

```
config.yaml（行为配置）
   │
   │ 引用（用 $ENV_VAR 语法）
   ▼
.env（密钥）            ← config.yaml 里的 $OPENAI_API_KEY 从这里读

extensions_config.json（扩展配置）  ← 独立的，管 MCP 和技能开关
```

举个具体例子，`config.yaml` 里这样写：
```yaml
models:
  - name: gpt-4o
    api_key: $OPENAI_API_KEY    # ← 不写真实 key，用变量引用
```

而真实的 key 在 `.env` 里：
```bash
OPENAI_API_KEY=sk-xxxxxxxxxxxx
```

这样代码和配置都不含密钥，`.env` 被 `.gitignore` 排除，永远不会泄露。

## 4.4 `config.yaml` 核心段落详解

下面我们把 `config.yaml` 的几个最重要段落逐一拆解。完整的字段参考项目根目录的 `config.example.yaml`（里面有详细注释）。

### 📦 段落 1：`models`（模型配置）★最核心★

这是你接触最多的段落。**每个模型是一段独立的配置**：

```yaml
models:
  # 一个模型 = 一段配置
  - name: gpt-4o                    # 内部标识（代码里用这个名字引用）
    display_name: GPT-4o            # 界面上显示的名字
    use: langchain_openai:ChatOpenAI   # ★用哪个 LangChain 适配器
    model: gpt-4o                    # 模型 ID（发给 API 的名字）
    api_key: $OPENAI_API_KEY        # 密钥（从 .env 读）
    request_timeout: 600.0          # 请求超时（秒）
    max_retries: 2                  # 失败重试次数
    max_tokens: 4096                # 最大输出 token
    temperature: 0.7                # 温度（创造性）
    supports_vision: true           # ★是否支持看图
    supports_thinking: false        # ★是否支持思考模式
```

#### `use` 字段：最关键的字段

`use` 是整个配置里最重要的字段，它告诉 deer-flow「**用哪个 Python 类来实例化这个模型**」，格式是 `模块路径:类名`。

| 你用的模型 | `use` 填什么 | 为什么 |
|-----------|-------------|--------|
| OpenAI 原版 | `langchain_openai:ChatOpenAI` | 官方适配器 |
| DeepSeek | `langchain_openai:ChatOpenAI` | DeepSeek 兼容 OpenAI 接口 |
| 豆包（火山） | `langchain_openai:ChatOpenAI` | 同上，但要配 `base_url` |
| Anthropic Claude | `langchain_anthropic:ChatAnthropic` | 专用接口 |
| Google Gemini | `langchain_google_genai:ChatGoogleGenerativeAI` | 专用接口 |
| Ollama 本地 | `langchain_ollama:ChatOllama` | Ollama 用原生接口 |
| vLLM 自部署 | `deerflow.models.vllm_provider:VllmChatModel` | deer-flow 自己的适配器 |

> 🔑 **关键规律**：只要某个模型**兼容 OpenAI 接口**（提供 `/v1/chat/completions`），就用 `langchain_openai:ChatOpenAI` + 配 `base_url`。这就是为什么 DeepSeek、豆包、Kimi 都能用同一个适配器。

#### `base_url`：指向哪个服务器

对于非 OpenAI 官方的模型，必须配 `base_url`：

```yaml
- name: deepseek-chat
  use: langchain_openai:ChatOpenAI
  model: deepseek-chat
  api_key: $DEEPSEEK_API_KEY
  base_url: https://api.deepseek.com/v1   # ★关键：DeepSeek 的地址
```

#### 思考模式（thinking）

部分模型（DeepSeek-R1、Claude、豆包思考版）支持「思考模式」，配置如下：

```yaml
- name: deepseek-reasoner
  use: deerflow.models.patched_deepseek:PatchedChatDeepSeek
  model: deepseek-reasoner
  supports_thinking: true              # ★必须声明支持
  when_thinking_enabled:               # 开启思考时，额外发这些参数
    extra_body:
      thinking:
        type: enabled
  when_thinking_disabled:              # 关闭思考时，发这些参数
    extra_body:
      thinking:
        type: disabled
```

`when_thinking_enabled` / `when_thinking_disabled` 是两套「开关参数」——用户在界面上切换思考模式时，deer-flow 会自动把对应的参数塞进去。这个机制第 05 章会结合代码细讲。

---

### 🔧 段落 2：`tools` 和 `tool_groups`（工具配置）

工具段落定义「Agent 有哪些工具可用」。

#### 工具分组（tool_groups）

先定义「工具组」（逻辑分类）：
```yaml
tool_groups:
  - name: web          # 网络类工具
  - name: file:read    # 文件读取类
  - name: file:write   # 文件写入类
  - name: bash         # 命令执行类
```

#### 工具定义（tools）

每个工具一段配置，关键字段也是 `use`（和模型一样，指向具体的 Python 实现）：

```yaml
tools:
  # 搜索工具（用 DuckDuckGo，免费不要 key）
  - name: web_search
    group: web                                      # 归属哪个组
    use: deerflow.community.ddg_search.tools:web_search_tool   # 实现在哪
    max_results: 5                                   # 工具自己的参数

  # 网页抓取工具（用 Jina AI）
  - name: web_fetch
    group: web
    use: deerflow.community.jina_ai.tools:web_fetch_tool
    timeout: 10

  # 文件读取工具
  - name: read_file
    group: file:read
    use: deerflow.sandbox.tools:read_file_tool       # ← 沙箱工具

  # 文件写入工具
  - name: write_file
    group: file:write
    use: deerflow.sandbox.tools:write_file_tool

  # bash 执行工具
  - name: bash
    group: bash
    use: deerflow.sandbox.tools:bash_tool
```

注意 `use` 的值——`deerflow.sandbox.tools:write_file_tool` 表示「在 `deerflow.sandbox.tools` 这个模块里，找 `write_file_tool` 这个对象」。这就是第 05 章要讲的**反射机制**。

> 💡 自定义 Agent（custom agents）可以通过 `tool_groups` 限制只用某些组的工具，实现权限隔离。比如一个「只读分析」Agent 只给 `file:read` 和 `web` 组。

### 📦 段落 3：`sandbox`（沙箱配置）

控制 Agent 的执行环境：

```yaml
sandbox:
  # 本地沙箱（默认，直接在你的机器上执行）
  use: deerflow.sandbox.local:LocalSandboxProvider
  # 或者用 Docker 隔离沙箱（更安全，需要 Docker）
  # use: deerflow.community.aio_sandbox:AioSandboxProvider

  allow_host_bash: false    # ★本地模式下是否允许直接用你的 bash（危险，默认关）
```

| 模式 | `use` 填什么 | 安全性 | 适合场景 |
|------|------------|--------|---------|
| 本地执行 | `deerflow.sandbox.local:LocalSandboxProvider` | 低（直接动你电脑） | 本地开发、完全可信 |
| Docker 隔离 | `deerflow.community.aio_sandbox:AioSandboxProvider` | 高（容器隔离） | 生产、不可信任务 |

### 📝 段落 4：`memory`（记忆配置）

```yaml
memory:
  enabled: true                # ★总开关：是否开启记忆抽取
  injection_enabled: true      # ★是否把记忆注入到 system prompt
  storage_path: null           # 存储路径（null 用默认的按用户隔离）
  debounce_seconds: 30         # 去抖时间（多久后开始抽取记忆）
  model_name: null             # 用哪个模型抽取记忆（null 用默认模型）
  max_facts: 100               # 最多存多少条事实
  fact_confidence_threshold: 0.7   # 事实置信度阈值（低于此值不存）
  max_injection_tokens: 2000   # 注入 prompt 的最大 token
  token_counting: tiktoken     # token 计数方式（tiktoken/char）
```

### 🎯 段落 5：`skills`（技能配置）

```yaml
skills:
  path: ./skills                    # 技能目录在硬盘上的位置
  container_path: /mnt/skills       # ★沙箱容器内看到的路径
```

注意这里有个「双路径」：`path` 是**真实硬盘路径**，`container_path` 是 **Agent 看到的虚拟路径**。这个映射第 08 章沙箱会详细讲。

技能的**启用/禁用**状态不在 `config.yaml` 里，而在 `extensions_config.json` 里（见 4.6 节）。

## 4.5 三大机制之一：配置加载优先级

deer-flow 找 `config.yaml` 时，有一套明确的优先级（从高到低）：

```
1. 命令行参数 / 代码里显式传的 config_path     ← 最高优先级
       ↓ 找不到才看
2. 环境变量 DEER_FLOW_CONFIG_PATH               ← 用这个指定自定义路径
       ↓ 找不到才看
3. 当前目录下的 config.yaml（比如 backend/）
       ↓ 找不到才看
4. 父目录下的 config.yaml（项目根目录）★推荐放这里★
```

> 💡 **实践建议**：把 `config.yaml` 放在**项目根目录**（`deer-flow/config.yaml`），这是最省心的位置。`make setup` 向导默认就放这里。

`extensions_config.json` 的查找逻辑完全一样，只是环境变量名换成 `DEER_FLOW_EXTENSIONS_CONFIG_PATH`。

## 4.6 三大机制之二：热重载 vs 需重启

这是新手最容易困惑的点：**改了配置到底要不要重启服务？**

deer-flow 的答案是：**「看字段」**。它把所有配置字段分成两类：

### 🟢 热重载字段（改了立即生效，不用重启）

这些是「**每次请求都会读**」的字段，改了 `config.yaml` 后，下一次对话就生效：

- `models[*]` 的内容（换模型参数、加新模型）
- `tools[*]`（加新工具）
- `summarization.*`、`title.*`、`memory.*`
- `subagents.*`、技能相关
- Agent 的 system prompt

> 这就是为什么第 02 章的实验里，你加个模型不用重启就能在界面看到——因为模型配置是热重载的。

### 🔴 需重启字段（改了必须重启服务）

这些是「**启动时初始化一次**」的基础设施字段：

- `database`（数据库连接）
- `checkpointer`（状态检查点存储）
- `sandbox`（沙箱类型——本地切 Docker 必须重启）
- `channels`（IM 渠道配置）
- `log_level`（日志级别）

> 为什么这些不能热重载？因为它们涉及**进程级别的资源初始化**。比如沙箱从本地切到 Docker，要重新建立容器连接，没法在运行中切换。

### 怎么判断？

deer-flow 用一个**注册表**明确列出了哪些是「需重启」字段，位置在 `reload_boundary.py` 里的 `STARTUP_ONLY_FIELDS`。而且在代码里，这些字段的说明都带了 `startup-only:` 前缀，你用 IDE 悬停就能看到提示。

**口诀**：业务行为（模型/工具/记忆）多半能热重载；基础设施（数据库/沙箱/渠道）多半要重启。拿不准就重启一下，最保险。

## 4.7 三大机制之三：`$ENV_VAR` 环境变量替换

还记得 `config.yaml` 里到处都是 `$OPENAI_API_KEY` 这种写法吗？这是 deer-flow 的**环境变量替换**机制。

### 工作原理

任何以 `$` 开头的配置值，deer-flow 都会把它当作**环境变量名**，去读取对应的值：

```yaml
# config.yaml 里写的
api_key: $OPENAI_API_KEY

# deer-flow 启动时，发现值以 $ 开头
# → 去读环境变量 OPENAI_API_KEY 的值
# → 假设 .env 里写了 OPENAI_API_KEY=sk-abc123
# → 最终等价于 api_key: sk-abc123
```

### 为什么这么设计

| 方式 | 优点 | 缺点 |
|------|------|------|
| 直接写 key 在 config.yaml | 简单 | ❌ 提交 git 会泄露 |
| **`$ENV_VAR` 引用**（deer-flow 用的） | ✅ 密钥在 .env 不入库；✅ 不同环境（开发/生产）用不同 .env | 要维护两个文件 |
| 运行时 `export` | 不落盘 | 每次开终端都要设 |

deer-flow 选了第二种，兼顾安全和灵活。

### `.env` 文件怎么管

`.env` 是一个简单的「键=值」文本文件：

```bash
# .env
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxx
DEEPSEEK_API_KEY=sk-yyyyyyyyyyyy
TAVILY_API_KEY=tvz-zzzzzzzzzzz
```

注意事项：
- ✅ `.env` **已经被 `.gitignore` 排除**，不会提交（deer-flow 默认配置好了）
- ✅ 你可以有多个 `.env`（开发用一个，生产用一个）
- ⚠️ 改了 `.env` **必须重启服务**才生效（环境变量是启动时加载的）

## 4.8 `extensions_config.json`：MCP 和技能的开关

这个文件和 `config.yaml` 平级，但管的是「**可热插拔的扩展**」：

```json
{
  "mcpServers": {
    "github": {
      "enabled": true,                    // ★是否启用这个 MCP 服务器
      "type": "stdio",                    // 传输类型（stdio/sse/http）
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {"GITHUB_TOKEN": "$GITHUB_TOKEN"}
    }
  },
  "skills": {
    "pdf-processing": {"enabled": true},   // ★技能的启用状态
    "deep-research": {"enabled": true}
  }
}
```

### 为什么不和 config.yaml 合并？

因为这个文件**改动频繁**且**经常通过界面操作**（在 Settings 里点开关）。把它独立出来，Gateway API 修改它时不会干扰主配置，也更安全。

> 你在 deer-flow 界面的「Settings > MCP」里增删 MCP 服务器，改的就是这个文件。

## 4.9 代码精读：配置是怎么加载的

光讲概念不过瘾，我们看一眼 deer-flow 加载配置的核心代码（`config/app_config.py`，简化版）：

```python
# 文件：packages/harness/deerflow/config/app_config.py（简化示意）

from pydantic import BaseModel
import yaml, os

class AppConfig(BaseModel):
    """整个应用的配置，对应 config.yaml 的结构"""
    models: list[ModelConfig]           # 模型列表
    tools: list[ToolConfig]             # 工具列表
    tool_groups: list                   # 工具组
    sandbox: SandboxConfig              # 沙箱配置
    memory: MemoryConfig                # 记忆配置
    skills: SkillsConfig                # 技能配置
    # ... 其他段落
    config_version: int = 0             # 配置版本号

    @classmethod
    def from_file(cls, path: str = None):
        """从 YAML 文件加载配置"""
        path = path or _resolve_config_path()   # 按优先级找到 config.yaml
        with open(path) as f:
            raw = yaml.safe_load(f)              # 解析 YAML → dict
        raw = _resolve_env_vars(raw)             # ★把 $VAR 替换成真实值
        return cls(**raw)                        # 用 Pydantic 校验并构造对象
```

### 配置缓存与失效

为了性能，deer-flow 会**缓存**解析后的配置，但会智能失效：

```python
# 伪代码：配置缓存逻辑
_cached_config = None
_cached_mtime = None

def get_app_config() -> AppConfig:
    global _cached_config, _cached_mtime
    path = _resolve_config_path()
    mtime = os.path.getmtime(path)          # 文件的修改时间
    if _cached_config and mtime == _cached_mtime:
        return _cached_config               # 没变，用缓存
    # 文件变了，重新加载！
    _cached_config = AppConfig.from_file(path)
    _cached_mtime = mtime
    return _cached_config
```

**这就是热重载的秘密**：每次请求都调 `get_app_config()`，它通过比对文件的修改时间（mtime）判断配置是否变了——变了就重新加载，没变就用缓存。所以你改了 `config.yaml` 保存的瞬间，下次请求就生效了。

### 配置版本号（config_version）

```yaml
config_version: 12
```

这是个防呆设计。`config.example.yaml`（模板）里有个版本号，你的 `config.yaml` 里也有。启动时 deer-flow 会比对两者：
- 你的版本 < 模板版本 → 弹警告「你的配置可能过时了」
- 运行 `make config-upgrade` 可以自动把模板里新增的字段合并到你的配置里

升级 deer-flow 后如果加了新配置项，跑一下 `make config-upgrade` 就行，不用手动对照。

## 4.10 动手实验：配置一个本地 Ollama 模型

理论讲完了，我们来实操。这个实验让你亲手走一遍「配模型 → 生效 → 使用」的全流程。

### 步骤 1：安装并启动 Ollama

去 [ollama.com](https://ollama.com) 下载安装，然后拉一个模型：

```bash
ollama pull qwen3:8b    # 拉一个 8B 的小模型（对显存要求低）
```

确认 Ollama 在跑：
```bash
ollama list    # 应该能看到 qwen3:8b
```

### 步骤 2：在 config.yaml 添加模型配置

打开 `config.yaml`，在 `models:` 段落下**加一段**：

```yaml
models:
  # ... 你原有的模型保留 ...

  # 新增：本地 Ollama 模型
  - name: qwen3-local
    display_name: Qwen3 8B (本地)
    use: langchain_ollama:ChatOllama
    model: qwen3:8b
    base_url: http://localhost:11434     # Ollama 默认地址，注意不要加 /v1
    temperature: 0.7
    supports_thinking: true
    supports_vision: false
```

## 4.11 本章小结

- ✅ deer-flow 是**配置驱动**的：改 `config.yaml` 就能定制大部分行为，不用改代码
- ✅ 两个核心配置文件：**`config.yaml`（系统行为）+ `extensions_config.json`（MCP/技能开关）**，密钥放 `.env`
- ✅ 配置加载有**优先级**：显式参数 > 环境变量 > 当前目录 > 父目录
- ✅ **热重载 vs 需重启**：业务字段（模型/工具/记忆）热重载，基础设施字段（沙箱/数据库/渠道）需重启
- ✅ `$ENV_VAR` 机制让密钥不入库，安全又灵活
- ✅ 热重载的秘密是**文件 mtime 比对 + 配置缓存**
- ✅ `use` 字段（`模块:类名`）是连接配置和代码的桥梁，背后是反射机制（第 05 章详解）
