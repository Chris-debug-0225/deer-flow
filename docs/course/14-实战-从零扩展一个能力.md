## 14.2 实战需求分析

我们要给 deer-flow 加这样一个能力：

> **用户问「今天北京天气怎么样，适合出门吗」时，Agent 能查真实天气，并给出出行建议（带伞/防晒/穿衣等）。**

### 需要什么

这个能力需要两个部件配合：

| 部件 | 作用 | 对应知识 |
|------|------|---------|
| **一个工具** | 能查询某个城市的真实天气 | 第 07 章 |
| **一个技能** | 定义「根据天气给出行建议」的方法论 | 第 11 章 |

### 为什么需要两个

- **工具**提供「能力」——能查到天气数据
- **技能**提供「智慧」——知道拿到数据后怎么分析和建议

光有工具，Agent 查到「15℃，小雨」可能只会干巴巴回复；加上技能，它会主动分析「下雨→带伞」「15℃→穿外套」「还要看风力」等。

## 14.3 第 1 步：写一个天气查询工具

### 14.3.1 创建工具文件

在 `backend/packages/harness/deerflow/community/` 下创建 `weather/` 目录，建 `tools.py`：

```python
# 文件：community/weather/tools.py

import urllib.request
import json

from langchain.tools import tool


@tool
def weather_tool(city: str) -> str:
    """查询指定城市的当前天气。

    当用户问"XX城市天气怎么样""今天冷不冷""要不要带伞"等
    与实时天气相关的问题时使用此工具。
    返回温度、天气状况、湿度、风力等信息。

    Args:
        city: 城市名称（中文或英文，如"北京"或"Beijing"）
    """
    try:
        # 使用 Open-Meteo 的免费天气 API（不需要 key）
        # 第 1 步：地理编码，把城市名转成经纬度
        geo_url = (
            "https://geocoding-api.open-meteo.com/v1/search"
            f"?name={urllib.parse.quote(city)}&count=1&language=zh"
        )
        with urllib.request.urlopen(geo_url, timeout=10) as resp:
            geo_data = json.loads(resp.read())

        if not geo_data.get("results"):
            return f"找不到城市：{city}"

        location = geo_data["results"][0]
        lat, lon = location["latitude"], location["longitude"]

        # 第 2 步：查天气
        weather_url = (
            "https://api.open-meteo.com/v1/forecast"
            f"?latitude={lat}&longitude={lon}"
            "&current=temperature_2m,relative_humidity_2m,"
            "weather_code,wind_speed_10m"
            "&timezone=auto"
        )
        with urllib.request.urlopen(weather_url, timeout=10) as resp:
            weather_data = json.loads(resp.read())

        current = weather_data["current"]
        # weather_code 是 WMO 编码，转成可读描述
        desc = _weather_code_to_desc(current["weather_code"])

        return (
            f"{location['name']}当前天气：\n"
            f"- 温度：{current['temperature_2m']}°C\n"
            f"- 天气：{desc}\n"
            f"- 湿度：{current['relative_humidity_2m']}%\n"
            f"- 风速：{current['wind_speed_10m']} km/h"
        )
    except Exception as e:
        return f"查询天气失败：{e}"


def _weather_code_to_desc(code: int) -> str:
    """WMO 天气编码转中文描述。"""
    mapping = {
        0: "晴", 1: "多云", 2: "局部多云", 3: "阴",
        45: "雾", 48: "雾凇",
        51: "毛毛雨", 53: "毛毛雨", 55: "毛毛雨",
        61: "小雨", 63: "中雨", 65: "大雨",
        71: "小雪", 73: "中雪", 75: "大雪",
        80: "阵雨", 81: "中阵雨", 82: "大阵雨",
        95: "雷暴", 96: "雷暴伴冰雹", 99: "强雷暴伴冰雹",
    }
    return mapping.get(code, f"未知（编码{code}）")
```

别忘了在文件顶部补一个 import：
```python
import urllib.parse   # 补上这个
```

### 14.3.2 工具设计要点回顾

对照第 07 章的知识检查这个工具：

| 要点 | 我们怎么做的 |
|------|------------|
| 用 `@tool` 装饰器 | ✅ |
| **description 写清楚「何时用」** | ✅ 写了「当用户问天气、冷不冷、带伞时」 |
| 参数有类型注解 | ✅ `city: str` |
| 错误处理 | ✅ try/except，失败返回友好消息而不是崩溃 |
| 不依赖付费 API | ✅ 用 Open-Meteo 免费 API |

### 14.3.3 注册到 config.yaml

打开项目根目录的 `config.yaml`，在 `tools:` 段落加一段：

```yaml
tools:
  # ... 原有工具保留 ...

  # 新增：天气查询工具
  - name: weather
    group: web
    use: deerflow.community.weather.tools:weather_tool
```

`use` 的值就是 `模块路径:对象名`——指向我们刚写的 `weather_tool` 函数。

### 14.4.2 写 SKILL.md

```markdown
---
name: travel-advice
description: 根据实时天气给出行的穿衣和装备建议。
  当用户问"今天穿什么""要不要带伞""适合出门吗"
  或任何结合天气做出行决策的问题时使用。
  先用 weather 工具查天气，再按本技能的方法论给建议。

# Travel Advice Skill（出行建议技能）

## 何时使用
- 用户问「今天穿什么」「要不要带伞」「冷不冷」
- 用户问「适不适合出门/运动/骑行」
- 任何需要结合实时天气做出行决策的问题

## 工作流程

### 第 1 步：查询天气
必须先调用 `weather` 工具获取实时天气数据，不要凭印象猜。

### 第 2 步：分析并给建议
根据天气数据，从以下维度给出**具体、可执行**的建议：

#### 🌧️ 降水维度
- 小雨/毛毛雨 → 建议带折伞
- 中雨/大雨 → 建议带伞 + 穿防水鞋，尽量减少外出
- 雷暴 → 建议避免户外活动，尤其不要在空旷地带

#### 🌡️ 温度维度
- < 0°C → 羽绒服、围巾、手套、帽子
- 0-10°C → 厚外套 + 毛衣
- 10-18°C → 外套 + 长袖
- 18-25°C → 舒适，单衣 + 薄外套
- 25-30°C → 短袖，注意防晒
- > 30°C → 轻薄透气衣物，强烈建议防晒 + 补水

#### 💨 风力维度
- 风速 > 30 km/h → 注意防风，骑行困难
- 风速 > 50 km/h → 避免户外活动

#### 💧 湿度维度
- 湿度 > 80% + 高温 → 体感更热，注意中暑
- 湿度 < 30% → 干燥，注意补水保湿

### 第 3 步：综合输出
把以上分析整合成一段自然的建议，格式参考：

```
📍 {城市} 现在 {温度}°C，{天气描述}。

👕 穿衣建议：...
☂️ 装备建议：...
⚠️ 注意事项：...
```

## 注意事项
- 一定要先查天气再说，不要凭空建议
- 建议要具体（"穿薄外套"比"穿暖点"好）
- 如果天气极端，优先强调安全
- 可以结合用户的具体场景（上班/运动/出游）调整建议
```

### 14.4.3 启用技能

打开 `extensions_config.json`，在 `skills` 里加一条（或通过界面开关）：

```json
{
  "skills": {
    "travel-advice": {"enabled": true}
  }
}
```

## 14.5 第 3 步：测试验证

两个文件都就位了，现在测试。

### 14.5.1 确认热加载

- 工具配置（config.yaml）：**热加载**，不用重启
- 技能（extensions_config.json）：**热加载**，不用重启

所以你**不用重启服务**，直接刷新浏览器就能测。

### 14.5.2 测试 1：自动触发

新开对话，问：
> 今天北京天气怎么样？适合出门吗？

预期行为：
1. Agent 判断需要天气信息 → 自动调用 `weather` 工具
2. 拿到天气数据 → 结合 `travel-advice` 技能的方法论分析
3. 输出结构化的穿衣 + 装备建议

如果 Agent **没调工具**而是瞎编天气 → 说明工具的 description 不够清楚，回去加强。
如果调了工具但**建议很泛泛** → 说明技能方法论没被加载，检查技能是否启用、description 是否写好。

### 14.5.3 测试 2：斜杠激活

如果自动触发不稳定，用斜杠强制激活：
> /travel-advice 上海今天适合骑行吗？

这次技能被**强制注入**当前轮，Agent 一定会按技能方法论走。

### 14.5.4 查看日志

在 `make dev` 的终端里，观察日志确认：
```
Total tools loaded: 13, built-in tools: 4   ← 应该比之前多 1 个（weather）
```
```
Including view_image_tool for model ...       ← 正常的中间件加载日志
```

### 14.5.5 调试技巧

如果不工作，按这个顺序排查：

| 症状 | 可能原因 | 排查方法 |
|------|---------|---------|
| Agent 说「我没有天气工具」 | 工具没加载 | 查 config.yaml 的 use 路径对不对；看日志 tools loaded 数量 |
| 工具调了但报错 | 代码有 bug | 看后端日志的 traceback；单独跑函数测试 |
| 有天气数据但没建议 | 技能没加载 | 查 extensions_config.json 技能是否 enabled |
| 技能加载了但 LLM 没遵守 | 技能正文不够清晰 | 加强 SKILL.md 的流程描述 |

> 💡 **最重要的调试技巧**：看后端日志。Gateway 的终端输出会告诉你「加载了几个工具」「Agent 在干什么」「哪里出错了」。养成看日志的习惯。

## 14.6 进阶：还可以怎么增强

基础版完成后，你可以往这些方向继续增强（作为练习）：

### 增强 1：多日预报
改 `weather_tool`，加个 `days` 参数，查未来 N 天的预报。

### 增强 2：自定义子代理
写一个 `travel-planner` 子代理（第 09 章），专门做旅行规划，调用天气工具 + 搜索景点 + 给完整行程。

### 增强 3：把建议写成文件
结合沙箱系统（第 08 章），让 Agent 把出行建议写成一个 `advice.md` 文件存到 outputs。

### 增强 4：接入 MCP
如果你找到了一个天气相关的 MCP 服务器，可以不用自己写工具，直接配 MCP（第 07 章）。
