## 7.2 概念铺垫：为什么 Agent 需要工具

第 01 章我们说过：LLM 只会「说话」，不会「动手」。工具就是给它的「手」。

### 工具的本质

一个工具，本质上就是**一个 LLM 可以调用的函数**：

```python
# 一个工具 = 一个函数 + 它的说明
def web_search(query: str) -> str:
    """搜索互联网并返回结果。"""
    return 实际搜索(query)
```

但 LLM 不能直接调用 Python 函数。它需要一个**约定**：模型输出「我要调用 web_search，参数是 query=xxx」，然后有个执行器帮它真正调用这个函数，把结果喂回去。

这个「约定」就是 **Tool 协议**。

### Tool 调用的完整流程

```
① LLM 收到：用户问题 + 可用工具列表（每个工具的名字、说明、参数）
       │
       ▼
② LLM 思考后输出：「我要调用 web_search，参数 query="LangGraph"」
   （这叫 tool_call，本质是结构化的 JSON）
       │
       ▼
③ 执行器（LangGraph 的 ToolNode）收到 tool_call
       → 找到对应的 Python 函数 web_search
       → 用参数 query="LangGraph" 调用它
       → 得到返回值 "LangGraph 是一个 Agent 编排框架..."
       │
       ▼
④ 返回值作为 ToolMessage 喂回给 LLM
       │
       ▼
⑤ LLM 看到结果，继续思考下一步
```

> 💡 **关键**：LLM 永远不直接执行代码，它只是「声明」要调哪个工具、传什么参数。真正的执行由框架（LangGraph）完成。这就是「工具调用」的安全边界——LLM 的输出是「意图」，不是「指令」。

## 7.3 LangChain Tool 协议

LangChain 定义了工具的标准格式。一个工具需要三样东西：

| 要素 | 作用 | 给谁看 |
|------|------|--------|
| **name** | 工具名（LLM 用它引用） | LLM |
| **description** | 工具说明（告诉 LLM 这工具干嘛的） | LLM |
| **args_schema** | 参数结构（告诉 LLM 要传什么参数） | LLM |

### 用 `@tool` 装饰器定义工具

最简单的方式是用 LangChain 的 `@tool` 装饰器：

```python
from langchain.tools import tool

@tool
def web_search(query: str, max_results: int = 5) -> str:
    """搜索互联网，返回相关网页摘要。

    Args:
        query: 搜索关键词
        max_results: 最大返回条数
    """
    # 实际的搜索逻辑
    results = 调用搜索API(query, max_results)
    return 格式化(results)
```

`@tool` 装饰器会自动：
- 把函数名 `web_search` 作为工具 **name**
- 把 docstring 作为 **description**（LLM 靠它判断「该不该用这个工具」）
- 把函数签名的类型注解（`query: str`）转成 **args_schema**

### description 为什么极其重要

LLM 决定「用哪个工具」时，**主要看 description**。如果你写得含糊，LLM 就会用错。

```python
# ❌ 差的 description：LLM 不知道什么时候该用它
@tool
def search(q: str) -> str:
    """搜索"""

# ✅ 好的 description：清楚说明用途、参数、适用场景
@tool
def web_search(query: str) -> str:
    """搜索互联网获取最新信息。当你需要查找实时数据、新闻、
    或训练数据里可能没有的新知识时使用。参数 query 是搜索词。"""
```

> 💡 **写工具描述是 Agent 开发的核心技能之一**。一个好的描述 = LLM 用对工具的一半保障。deer-flow 自带工具的描述都写得很细致，值得模仿。

## 7.4 代码精读：`get_available_tools()`

现在看 deer-flow 怎么把所有工具组装起来。这个函数是工具系统的核心（来自 `tools/tools.py`）。我贴简化版逐段讲解：

```python
# 文件：packages/harness/deerflow/tools/tools.py（简化）

def get_available_tools(
    groups=None,              # 只加载这些组的工具（权限控制）
    include_mcp=True,         # 是否包含 MCP 工具
    model_name=None,          # 当前模型（决定要不要加视觉工具）
    subagent_enabled=False,   # 是否加子代理工具
    *,
    app_config=None,
) -> list[BaseTool]:
    """获取所有可用工具。"""
    config = app_config or get_app_config()
```

### 第 1 步：从配置加载工具

```python
    # 从 config.yaml 的 tools 段落读工具配置
    # 如果指定了 groups，只留这些组的（自定义 Agent 的权限隔离）
    tool_configs = [
        tool for tool in config.tools
        if groups is None or tool.group in groups
    ]

    # ★安全：本地沙箱默认禁用 bash★
    if not is_host_bash_allowed(config):
        tool_configs = [t for t in tool_configs if not _is_host_bash_tool(t)]

    # 用反射把每个 config 的 use 字段（字符串）变成真正的工具对象
    # use 示例："deerflow.sandbox.tools:write_file_tool"
    loaded_tools = [resolve_variable(cfg.use, BaseTool) for cfg in tool_configs]
```

这里又用到了第 05 章讲的反射机制——`resolve_variable(cfg.use)` 把字符串变成对象。这就是为什么加工具不用改代码，只要在 `config.yaml` 写 `use` 路径。

### 第 2 步：加内置工具

```python
    builtin_tools = [present_file_tool, ask_clarification_tool]

    # 如果开了子代理，加 task 工具
    if subagent_enabled:
        builtin_tools.append(task_tool)

    # 如果模型支持视觉，加 view_image 工具
    model_config = config.get_model_config(model_name) if model_name else None
    if model_config and model_config.supports_vision:
        builtin_tools.append(view_image_tool)
```

注意 `view_image_tool` 是**条件添加**的——纯文本模型不需要它。这和第 06 章的 `ViewImageMiddleware` 条件加载是配套的。

### 第 3 步：加 MCP 工具

```python
    mcp_tools = []
    if include_mcp:
        from deerflow.mcp.cache import get_cached_mcp_tools
        extensions_config = ExtensionsConfig.from_file()
        if extensions_config.get_enabled_mcp_servers():
            mcp_tools = get_cached_mcp_tools()   # 拿到所有 MCP 工具
            for t in mcp_tools:
                tag_mcp_tool(t)   # 打标记，方便延迟加载识别
```

MCP（Model Context Protocol）是一个标准协议，允许外部服务器暴露工具给 Agent。比如你可以接一个「GitHub MCP 服务器」，它会把「创建 issue」「查 PR」等操作变成工具。deer-flow 支持任意 MCP 服务器，第 4.8 节讲过在 `extensions_config.json` 配置。

### 第 4 步：加 ACP 工具（可选）

```python
    acp_tools = []
    acp_agents = getattr(config, "acp_agents", {}) or {}
    if acp_agents:
        acp_tools.append(build_invoke_acp_agent_tool(acp_agents))
```

ACP（Agent Communication Protocol）是另一种协议，允许调用**外部的 Agent**（比如 Codex CLI、Claude Code）作为工具。这是高级功能，新手可以忽略。

### 第 5 步：合并 + 去重

```python
    # 合并所有来源的工具
    all_tools = loaded_tools + builtin_tools + mcp_tools + acp_tools

    # ★去重：按工具名去重，避免 LLM 收到重名工具混淆
    seen_names = set()
    unique_tools = []
    for t in all_tools:
        if t.name not in seen_names:
            unique_tools.append(t)
            seen_names.add(t.name)
        else:
            logger.warning(f"重复的工具名 {t.name}，已跳过")

    return unique_tools
```

去重很重要——如果两个工具同名，LLM 收到的工具 schema 会冲突，出现「我要调 search，但有两个 search，不知道调哪个」的混乱。deer-flow 显式去重并打日志。

### 完整流程图

```
get_available_tools()
       │
       ├─ 配置工具（config.yaml 里 tools 段落定义的）
       │    └─ 反射加载：resolve_variable(use) → 工具对象
       │
       ├─ 内置工具
       │    ├─ present_file（展示文件给用户）
       │    ├─ ask_clarification（问用户澄清）
       │    ├─ task（子代理，条件）
       │    └─ view_image（看图，条件）
       │
       ├─ MCP 工具（来自 MCP 服务器）
       │
       ├─ ACP 工具（来自外部 Agent，可选）
       │
       └─ 合并 + 按名去重 → 最终工具列表
              │
              └─ 传给 create_agent(tools=...)
```

## 7.5 四大类工具详解

### 类型 1：配置工具（来自 config.yaml）

这是最常见的一类，在 `config.yaml` 的 `tools:` 段落定义。典型代表：

| 工具名 | 用途 | `use` 实现 |
|--------|------|-----------|
| `web_search` | 网页搜索 | DuckDuckGo / Tavily / Serper 等 |
| `web_fetch` | 抓取网页内容 | Jina AI / Firecrawl 等 |
| `image_search` | 搜图片 | DuckDuckGo 图片 |
| `read_file` | 读文件 | 沙箱工具（第 08 章） |
| `write_file` | 写文件 | 沙箱工具 |
| `bash` | 执行命令 | 沙箱工具 |
| `ls` / `glob` / `grep` | 文件查找 | 沙箱工具 |
| `str_replace` | 字符串替换 | 沙箱工具 |

注意：**同一类工具有多个实现可选**。比如 `web_search` 可以用 DuckDuckGo（免费）或 Tavily（要 key），你选哪个就在 `use` 里填对应的实现路径。这就是「**同一接口，多种实现，配置切换**」的灵活性。

### 类型 2：内置工具（代码里写死的）

这些不在 `config.yaml` 里，而是代码里直接定义的，因为它们是 deer-flow 框架运转必需的：

| 工具 | 用途 | 特点 |
|------|------|------|
| `present_files` | 把生成的文件「展示」给用户 | 只允许展示 outputs 目录的文件 |
| `ask_clarification` | 问用户澄清问题 | 被 ClarificationMiddleware 拦截（第 06 章） |
| `view_image` | 把图片读成 base64 | 只有视觉模型才有 |
| `task` | 派发任务给子代理 | 只有开了子代理才有（第 09 章） |
| `write_todos` | 更新待办清单 | 只有计划模式才有 |

> 💡 内置工具和中间件是**配套**的。比如 `ask_clarification` 工具 + `ClarificationMiddleware` 中间件 = 完整的「澄清」功能。工具负责「声明意图」，中间件负责「拦截处理」。这种「工具 + 中间件」的配合模式在 deer-flow 里很常见。

### 类型 3：MCP 工具（最强大的扩展方式）

MCP（Model Context Protocol）是 Anthropic 提出的标准协议，允许**任何外部程序**把自己的能力暴露成工具。

举个例子，接一个 GitHub MCP 服务器后，Agent 自动获得这些工具：
- `create_issue`（创建 issue）
- `search_repositories`（搜索仓库）
- `get_file_contents`（读文件）
- ……几十个 GitHub 操作

而你**完全不用写代码**——只要在 `extensions_config.json` 里配：

```json
{
  "mcpServers": {
    "github": {
      "enabled": true,
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {"GITHUB_TOKEN": "$GITHUB_TOKEN"}
    }
  }
}
```

deer-flow 会自动连接这个服务器，把它的工具加载进来。

**MCP 的价值**：它把「工具开发」变成了「装插件」。社区有大量现成的 MCP 服务器（GitHub、数据库、Slack、文件系统…），你装上就能用。这是目前 Agent 生态最主流的扩展方式。

### 类型 4：ACP 工具（调用外部 Agent）

ACP 允许把一个**完整的外部 Agent**（如 Codex CLI、Claude Code）当成一个工具来调用。这是高级用法，新手了解即可。

## 7.6 工具的两个高级特性

### 特性 1：延迟加载（Deferred Tool Loading）

当 MCP 工具很多时（比如接了 5 个 MCP 服务器，共 100 个工具），全部塞进 LLM 的上下文会**浪费大量 token**（工具 schema 也很占空间）。

deer-flow 的解法是**延迟加载**（`tool_search` 配置）：
- 启动时只把工具**名字**列在提示词里（省空间）
- 加一个 `tool_search` 工具，Agent 用它「搜索」需要的工具
- 找到后才把那个工具的完整 schema 暴露出来

```
默认：100 个工具的完整 schema 全塞给 LLM（贵）
延迟：只给 100 个名字 + 1 个 tool_search 工具（省）
      Agent 用 tool_search("我要操作 GitHub") → 暴露 GitHub 工具
```

这就像查字典——你不需要把整本字典背下来，知道怎么查就行。第 06 章的 `DeferredToolFilterMiddleware` 就是干这个的。

### 特性 2：工具输出预算（Tool Output Budget）

有些工具返回超长内容（比如 `read_file` 读了一个 10 万字的文件），直接塞回 LLM 会撑爆上下文。

deer-flow 的 `tool_output` 配置解决这个问题：
- 工具结果超过阈值（默认 12000 字符）→ **存到磁盘**，只给 LLM 一个「预览 + 文件引用」
- LLM 需要完整内容时，用 `read_file` 去读那个文件

```
工具返回 50000 字 → 超过 12000 阈值
  → 存到 .tool-results/xxx.txt
  → 给 LLM：前 2000 字 + "...（完整内容见 .tool-results/xxx.txt）" + 后 1000 字
```

这个设计避免了「一次工具调用就撑爆上下文」的常见坑。

## 7.7 动手实验：写一个自己的工具

理论够了，我们来实战。假设你想给 Agent 加一个「获取当前时间」的工具。

### 步骤 1：写工具代码

在 `backend/packages/harness/deerflow/community/` 下建一个 `clock/` 目录，创建 `tools.py`：

```python
# 文件：community/clock/tools.py

from datetime import datetime

from langchain.tools import tool


@tool
def current_time_tool(timezone: str = "UTC") -> str:
    """获取当前时间。

    当用户问"现在几点""今天日期"等时间相关问题时使用。
    返回格式化的日期和时间字符串。

    Args:
        timezone: 时区名称（目前仅支持 UTC 和 local，默认 UTC）。
    """
    now = datetime.now() if timezone == "local" else datetime.utcnow()
    return now.strftime("%Y-%m-%d %H:%M:%S")
```

关键点：
- 用 `@tool` 装饰器
- **description 要写清楚**「什么时候用」
- 参数要有类型注解

### 步骤 2：在 config.yaml 注册

```yaml
tools:
  # ... 原有工具 ...

  - name: current_time
    group: web                      # 归到 web 组（也可以新建组）
    use: deerflow.community.clock.tools:current_time_tool
```

`use` 的值就是 `模块路径:对象名`，指向你刚写的函数。

### 步骤 3：验证

不用重启（工具配置热加载），刷新浏览器。问 Agent：

> 现在几点了？

Agent 应该会调用 `current_time` 工具并告诉你时间。如果它没用工具而是瞎编一个时间，说明你的 **description 写得不够清楚**——回去加强「当用户问时间问题时使用」这类指引。

### 这个实验的意义

你刚刚完成了 deer-flow 最核心的扩展方式：**写一个函数 → 配置 use → Agent 自动获得新能力**。不需要改 Agent 代码、不需要改中间件——这就是「配置驱动 + 反射加载」的威力。第 14 章的实战会在这个基础上做更复杂的扩展。

## 7.8 工具与中间件的协作

回顾一下，工具不是孤立工作的，它和中间件紧密配合。我们用「ask_clarification」这个例子看完整的协作链：

```
用户：「帮我分析这个数据集」
       │
       ▼
LLM 思考：数据集？哪个？不清楚 → 决定调 ask_clarification
       │
       ▼
LLM 输出 tool_call: ask_clarification(question="你指的是哪个数据集？")
       │
       ▼
★ ClarificationMiddleware 拦截（wrap_tool_call 钩子）★
       │  → 不执行工具，而是把问题格式化
       │  → 返回 Command(goto=END) 中断 Agent
       │
       ▼
用户看到：「❓ 你指的是哪个数据集？」并回答
       │
       ▼
新一轮对话开始，Agent 带着用户的回答继续工作
```

看明白了吗？**工具是"意图"，中间件是"处理"**。有些工具（如 `ask_clarification`、`present_files`、`task`）本身不干实事，它们更像「信号」——真正的逻辑在对应的中间件里。这种「**工具声明 + 中间件实现**」的分离，让系统非常灵活。

## 7.9 本章小结

✅ **工具** = LLM 可调用的函数，是 Agent 的「手」；LLM 只声明「调用意图」，框架负责真正执行
✅ LangChain Tool 协议三要素：**name（名字）、description（说明，极重要）、args_schema（参数结构）**
✅ `get_available_tools()` 把**四类工具**（配置/内置/MCP/ACP）统一组装、按名去重
✅ 加工具的流程：**写函数（@tool）→ config.yaml 配 use → 反射加载 → 自动生效**
✅ **MCP** 是最强大的扩展方式，把「工具开发」变成「装插件」
✅ 高级特性：**延迟加载**（省 token）、**输出预算**（防爆上下文）
✅ 工具和中间件常常**配套协作**：工具声明意图，中间件实现逻辑
