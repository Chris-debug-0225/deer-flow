## 13.2 为什么把 Agent 接入 IM

到目前为止，我们和 deer-flow 交互都是通过它的**网页界面**。但很多时候，用户更希望在**日常用的聊天软件**里直接用：

- 在**飞书**里 @机器人：「帮我查一下这周的销售数据」
- 在 **Telegram** 里发：「总结一下这个 PDF」
- 在 **Slack** 里说：「调研竞品 X」

把 Agent 接入 IM 的价值：
- ✅ **降低使用门槛**：用户不用打开专门网页，在哪聊天就在哪用
- ✅ **融入工作流**：IM 是很多人工作的地方，Agent 在那里最方便
- ✅ **主动触达**：结合群聊，Agent 能服务整个团队

deer-flow 支持多种 IM：**飞书/Lark、Slack、Telegram、Discord、钉钉、企业微信、微信**。

> 💡 这一章是「了解性质」的，不需要你立刻去配 IM。重点是理解「**一套 Agent 能力，多渠道触达**」的架构思想。

## 13.3 架构：消息总线（MessageBus）

deer-flow 的 IM 集成核心是一个**消息总线（MessageBus）**——一个异步的「发布-订阅」中心。

```
┌─────────────┐     入站消息      ┌──────────────┐
│  飞书渠道   │ ───────────────▶ │              │
├─────────────┤                  │              │
│  Telegram   │ ───────────────▶ │  MessageBus  │
├─────────────┤                  │  (消息总线)   │
│  Slack      │ ───────────────▶ │              │
├─────────────┤                  │              │
│  钉钉       │ ───────────────▶ │              │
└─────────────┘                  └──────┬───────┘
                                        │
                                        │ 分发
                                        ▼
                                 ┌──────────────┐
                                 │ ChannelManager│
                                 │ (核心调度器)  │
                                 └──────┬───────┘
                                        │
                                        │ 调 Gateway API 跑 Agent
                                        ▼
                                 ┌──────────────┐
                                 │  Gateway API │
                                 │  (Lead Agent)│
                                 └──────┬───────┘
                                        │
                                        │ 出站消息（回复）
                                        ▼
                                 ┌──────────────┐
                                 │  MessageBus  │
                                 └──────┬───────┘
                                        │ 分发到对应渠道
                                        ▼
                                 对应的 IM 渠道回复用户
```

### 为什么要消息总线

如果不加总线，每个渠道都要直接和 Agent 交互，会变成「多对多」的混乱耦合：

```
❌ 没有总线（混乱）：
飞书 ──→ Agent    Agent ──→ 飞书
Telegram ──→ Agent   Agent ──→ Telegram
Slack ──→ Agent      Agent ──→ Slack
（N 个渠道 × 2 方向 = 2N 条连接）
```

加了总线，变成「星型」解耦：

```
✅ 有总线（清晰）：
飞书 ──→ 总线 ──→ Agent
Telegram ──→ 总线 ──→ Agent
Slack ──→ 总线 ──→ Agent
（每个渠道只连总线一次）
```

新增一个渠道？只要写「渠道 ↔ 总线」的适配，不用动 Agent 代码。这就是**发布-订阅模式**的价值。

## 13.4 一条消息的完整旅行

我们追踪「用户在飞书发了一条消息」的完整流程：

```
① 用户在飞书发：「帮我总结这个 PDF」
       │
       ▼
② 飞书渠道（feishu.py）收到消息
   → 飞书通过 WebSocket 长连接把消息推过来
   → 封装成 InboundMessage（包含文本、用户、聊天 ID）
       │
       ▼
③ 发布到 MessageBus
   bus.publish_inbound(InboundMessage(...))
       │
       ▼
④ ChannelManager 消费消息（_dispatch_loop）
   → 查找/创建对应的 thread（对话）
   → 判断是命令还是普通消息
       │
       ├── 如果是命令（如 /new）→ 本地处理，直接回复
       │
       └── 如果是普通消息 → 继续
       │
       ▼
⑤ 调 Gateway API 跑 Agent
   client.runs.stream(thread_id, input={"messages":[...]})
   → 和前端发消息走的是同一条路！（第 12 章）
       │
       ▼
⑥ Agent 执行，流式返回
   → 用 runs.stream() 拿到 SSE 流
   → 每收到一段文字，就发布出站更新（is_final=False）
       │
       ▼
⑦ 出站消息回到 MessageBus
   bus.publish_outbound(OutboundMessage(..., is_final=False))
       │
       ▼
⑧ 飞书渠道收到出站消息
   → ★流式更新飞书卡片★（打字机效果）
   → 不是发新消息，而是「编辑」同一条卡片消息
       │
       ▼
⑨ Agent 跑完，发布最终消息（is_final=True）
   → 飞书渠道做最后一次卡片更新
   → 用户看到完整回答
```

### 关键设计：复用 Gateway API

注意第 ⑤ 步——IM 渠道**不是自己跑 Agent**，而是**调 Gateway API**（和前端一样的接口）。

这意味着：
- Agent 运行逻辑只有一份（在 Gateway 里）
- 网页、IM、任何渠道，都通过同一个 Gateway API 用 Agent
- **一套 Agent 能力，多渠道触达**

这是非常优雅的「**单一服务源 + 多前端**」架构。加一个新渠道，不用复制 Agent 代码，只要写「渠道适配」就行。

## 13.5 流式卡片更新：IM 里的打字机效果

第 12 章我们说网页用 SSE 实现打字机效果。但 IM 软件不支持 SSE——你不能往飞书/Telegram 推流。那怎么实现「边生成边显示」？

deer-flow 的解法是：**反复编辑同一条消息**。

### 飞书的实现

```
Agent 开始生成
   │
   ▼
飞书渠道先发一条「占位卡片」：「正在思考...」
   → 拿到这条消息的 message_id
   │
   ▼ （Agent 生成中，每隔一段推一次更新）
   │
收到 "你好" → PATCH 同一张卡片（用 message_id）：「你好」
收到 "，世界" → PATCH 同一张卡片：「你好，世界」
收到 "！" → PATCH 同一张卡片：「你好，世界！」
   │
   ▼
Agent 完成 → 最终 PATCH：「你好，世界！（完整回答）」
```

关键：用 **PATCH（部分更新）** 同一条消息，而不是发新消息。这样用户看到的是「一条消息在变长」，和网页的打字机效果类似。

### 各渠道的差异

不同 IM 的实现细节不同：

| 渠道 | 流式方式 | 特点 |
|------|---------|------|
| **飞书** | PATCH 卡片消息 | 需要 `update_multi=true` |
| **Telegram** | editMessageText | 有限速（群聊 20 条/分钟） |
| **钉钉** | AI Card streaming | 需要 card_template_id |
| **Slack/Discord** | 等 Agent 跑完一次性回复 | 不流式（API 限制） |

> 💡 Slack/Discord 因为 API 限制，做不到反复编辑消息，所以是「等 Agent 全跑完，一次性回复」。这也是为什么飞书/Telegram 体验更好——它们支持消息编辑。

### 限流处理

Telegram 群聊有「每分钟 20 条消息」的限制。deer-flow 的 Telegram 渠道做了**节流（throttle）**：

```python
# 简化示意
# 私聊：每 1 秒最多更新一次
# 群聊：每 3 秒最多更新一次（因为群聊限制更严）
if now - last_update < throttle_interval:
    skip_this_update()   # 丢弃这次更新，等下次
```

这种「**按平台限制自适应节流**」是 IM 集成的必备功夫——不懂平台规则，机器人会被封。

## 13.6 命令系统

除了普通聊天，IM 渠道还支持 `/` 开头的命令：

| 命令 | 作用 |
|------|------|
| `/new` | 开启新对话 |
| `/status` | 查看当前对话状态 |
| `/models` | 列出可用模型 |
| `/memory` | 查看记忆 |
| `/help` | 显示帮助 |

这些命令**不走 Agent**，由 `ChannelManager` 直接处理（快，不花 token）：

```
用户：/new
   │
   ▼
ChannelManager 判断：是 / 开头的命令
   → 直接创建新 thread，回复「已开启新对话」
   → 不调 Agent（省时省钱）

用户：帮我查天气
   │
   ▼
ChannelManager 判断：不是命令，是普通消息
   → 调 Gateway API 跑 Agent
   → 流式返回
```

## 13.7 IM 渠道的配置

IM 渠道在 `config.yaml` 的 `channels` 段落配置。以飞书为例：

```yaml
channels:
  langgraph_url: http://localhost:8001/api   # Gateway 地址
  gateway_url: http://localhost:8001

  feishu:
    enabled: true
    app_id: $FEISHU_APP_ID          # 从 .env 读
    app_secret: $FEISHU_APP_SECRET
    # domain: https://open.feishu.cn      # 国内（默认）
    # domain: https://open.larksuite.com  # 国际版 Lark

  telegram:
    enabled: true
    bot_token: $TELEGRAM_BOT_TOKEN
```

`.env` 里放凭证：
```bash
FEISHU_APP_ID=cli_xxxx
FEISHU_APP_SECRET=your_secret
TELEGRAM_BOT_TOKEN=123456:ABCdef...
```

> ⚠️ 注意：改 `channels` 配置**需要重启服务**（属于第 04 章说的「基础设施字段」）。

## 13.8 用户绑定的渠道连接

deer-flow 还有个高级功能「**用户自有的渠道连接（channel_connections）**」：

- 默认情况下，一个 IM 机器人服务所有用户（共用一个 Agent 实例）
- 开启 `channel_connections` 后，**每个用户可以绑定自己的 IM 账号**到自己的 DeerFlow 用户

这适合「**多租户**」场景——一个部署，多个用户各自独立用，记忆、文件都隔离。

绑定流程（以 Telegram 为例）：
1. 用户在网页生成一个绑定码
2. 在 Telegram 给机器人发 `/start 绑定码`
3. 系统把「这个 Telegram 用户」绑定到「这个 DeerFlow 用户」
4. 之后这个 Telegram 用户的消息，都跑在绑定的 DeerFlow 用户身份下

## 13.9 本章小结

✅ IM 集成让 Agent「**无处不在**」——在飞书/Telegram/Slack 等日常工具里直接用
✅ 核心是 **MessageBus（消息总线）** 的发布订阅架构，解耦「渠道」和「Agent」
✅ 一条消息的旅行：**IM 渠道 → 总线 → ChannelManager → Gateway API → Agent → 流式回 → 总线 → 渠道回复**
✅ 关键设计：**IM 渠道复用 Gateway API**，一套 Agent 能力多渠道触达
✅ 流式卡片：**反复 PATCH 同一条消息**实现打字机效果（受平台 API 限制）
✅ `/new`、`/help` 等命令**不走 Agent**，本地直接处理（省 token）
✅ 配置在 `config.yaml` 的 `channels` 段落，改了需重启
