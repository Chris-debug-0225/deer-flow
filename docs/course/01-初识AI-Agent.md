# 第 01 章 初识 AI Agent

> 🎯 本章目标：搞懂「LLM、Agent、框架」这三个概念，理解 deer-flow 到底在 LLM 之上多做了哪些事。本章几乎不写代码，重在建立认知地基。

---

## 1.1 从聊天到干活：LLM 是什么

**LLM（Large Language Model，大语言模型）** 是 deer-flow 的「大脑」。

你已经用过 ChatGPT、DeepSeek、豆包这类产品了，它们的「内核」就是一个 LLM。用最朴素的话说：

> LLM 是一个**接收一段文字、返回一段文字**的超强函数。

```python
# 这就是 LLM 的本质：一个「文字进、文字出」的函数
回答 = 大模型("请用一句话解释什么是递归")
# → "递归是一个函数在内部调用自己来解决问题。"
```

它之所以「智能」，是因为它在海量文本上训练过，所以能回答问题、写文章、写代码、做翻译。

但请注意一个关键限制：

> ⚠️ **LLM 本身只能「说话」，不能「动手」。**

你问 ChatGPT：「帮我查一下今天北京的天气。」它会**告诉你怎么查**（比如建议你打开天气 App），但它**没法真的去查**——因为它只是一个「文字进、文字出」的函数，没有眼睛、没有手、不能联网、不能操作电脑。

这就是 LLM 和 Agent 的分界线。

---

## 1.2 什么是 Agent：给大脑装上身体

**Agent（智能体）** 的核心思想很简单：

> 把 LLM 这个「只会说话的大脑」，装进一个「有手有脚、能感知环境、能采取行动」的身体里，让它**自己决定下一步做什么**。

我们用一个非常生活化的类比来理解。

### 🧠 类比：老板与员工

想象你是一家公司的老板，你招了一个**非常聪明但什么都不会自己动手**的员工（这就是 LLM）：

- 你说：「帮我调研一下竞品。」
- 这个员工脑子转得飞快，知道该去查哪些资料、怎么对比，**但他不会上网、不会开电脑、不会打电话**。

这个员工再聪明也没用，对吧？

于是你给他配了一套装备：

| 装备 | 对应 Agent 概念 | 让他能做的事 |
|------|----------------|------------|
| 一台能上网的电脑 | **搜索工具** | 自己去查资料 |
| 一个文件柜 | **文件读写工具** | 存取资料 |
| 一个计算器 | **代码执行工具** | 算数据 |
| 一个工位 | **沙箱（执行环境）** | 有自己的工作空间 |
| 几个实习生 | **子代理** | 把杂活外包出去 |
| 一个笔记本 | **记忆系统** | 记住你的偏好 |

装备配齐后，这个员工就**不只是「告诉你怎么做」，而是「直接帮你做完」**了。

> 💡 **LLM + 一套能动手的装备 + 自己决定怎么用这些装备 = Agent**

这里最关键的一点是「**自己决定**」。Agent 不是按固定流程执行脚本的程序，而是 LLM 在每一步都**自己思考、自己决定**下一步该用哪个工具、传什么参数。这就是「智能」的体现。

---

## 1.3 Agent 的核心循环：ReAct 范式

那 Agent 具体是怎么「自己决定」的呢？业界最经典的模式叫 **ReAct（Reasoning + Acting，推理 + 行动）**。

我们通过一个具体例子来理解。假设用户对 Agent 说：

> 「帮我查一下北京今天的天气，如果下雨就提醒我带伞。」

一个 ReAct Agent 会这样循环：

```
第 1 轮：
  🤔 思考（Reasoning）：用户想知道北京天气并决定是否带伞。
                     我需要先查天气，这要用到「搜索工具」。
  🔧 行动（Acting）：  调用工具 search("北京今天天气")
  👀 观察（Observing）：工具返回 → "北京今天小雨，气温 15-22℃"

第 2 轮：
  🤔 思考：已经拿到天气信息，北京今天有小雨。
          根据用户要求，下雨要提醒带伞。我可以直接回答了。
  🗣️ 回答：用户你好！北京今天小雨（15-22℃），记得带伞哦 ☔
```

看出来了吗？Agent 在不停地重复一个循环：

```
        ┌───────────────────────────┐
        │                           │
        ▼                           │
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ 思考    │ →  │ 行动    │ →  │ 观察    │
   │ Think   │    │ Act     │    │ Observe │
   └─────────┘    └─────────┘    └─────────┘
        │                              │
        │      （工具返回结果）          │
        └──────────────────────────────┘
                    循环，直到
                  觉得可以回答用户
```

这个「**思考 → 行动 → 观察**」的循环，就是 Agent 的心脏。循环的每一圈，都由 LLM 来决定「下一步干嘛」。

deer-flow 用的 **LangGraph** 框架，本质上就是帮你实现和管理这个循环的「骨架」。我们后面会看到，deer-flow 的 Lead Agent 就是一个被 LangGraph 编排起来的、不断 ReAct 循环的 Agent。

## 1.4 为什么需要「框架」

你可能会想：ReAct 循环听起来不难啊，我自己写个 `while` 循环不就行了？

```python
# 伪代码：自己手写一个最简 Agent 循环
messages = [用户的问题]
while True:
    回复 = 大模型调用(messages, 可用工具列表)
    if 回复.想调用工具:
        结果 = 执行工具(回复.工具调用)
        messages.append(回复)
        messages.append(结果)
    else:
        print(回复.最终答案)   # 不用工具了，就是最终答案
        break
```

确实，一个「玩具级」的 Agent 用几十行代码就能写出来。但当你要做一个**真正能用的生产级 Agent 系统**时，会立刻撞上一堆问题：

### 🤕 痛点 1：状态管理
对话进行到第 10 轮时，Agent 记得第 1 轮说了什么吗？多轮对话的「历史」怎么存？用户刷新页面后能接着聊吗？多个用户同时用怎么隔离？

### 🤕 痛点 2：上下文太长
聊了 50 轮后，历史消息可能已经有几万个 token，每次都全部发给 LLM 既慢又贵，还可能超出模型的上下文窗口。怎么自动压缩、总结？

### 🤕 痛点 3：工具太多
一个 Agent 可能要接搜索、文件、代码执行、数据库……几十个工具。怎么统一管理？工具调用出错了怎么办？怎么让 Agent 知道「现在该用哪个工具」？

### 🤕 痛点 4：执行隔离
Agent 要跑用户给的代码、写文件，万一它执行了 `rm -rf /` 怎么办？怎么给它一个安全的隔离环境？

### 🤕 痛点 5：协作
一个任务太复杂，一个 Agent 搞不定，怎么派几个「子 Agent」分工合作？

### 🤕 痛点 6：记忆
怎么让 Agent 跨对话记住「这个用户喜欢简洁风格」「这个用户是做前端的」？

### 🤕 痛点 7：可控性
Agent 陷入死循环怎么办？它一直调错工具怎么办？怎么在它「跑偏」时介入纠正？

……这些问题，**就是框架要帮你解决的**。

> 💡 **框架的价值**：让你不用重新发明轮子，把精力花在「我的 Agent 该有什么能力」上，而不是「怎么管理对话状态、怎么做上下文压缩」这些基础设施上。

deer-flow 正是这样一个框架——它把上面这些痛点**全都解决了**，并且打包成一套开箱即用的系统。我们这份课件，就是带你逐一拆解它是怎么解决这些问题的。

## 1.5 deer-flow 在生态中的位置

最后，我们用一张图把 deer-flow 放进整个 AI 开发生态里，让你看清它的定位：

```
┌─────────────────────────────────────────────────────────────┐
│  你的应用 / 你的业务                                        │
│  （你要做的具体事情：研究、做 PPT、数据分析…）              │
└────────────────────────────┬────────────────────────────────┘
                             │ 你在这层工作
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  deer-flow（Super Agent Harness）                           │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ • 完整的 Agent 运行时（沙箱、记忆、技能、子代理）    │    │
│  │ • Gateway API + 前端 + IM 渠道                       │    │
│  │ • 开箱即用的工具和技能                               │    │
│  └─────────────────────────────────────────────────────┘    │
└────────────────────────────┬────────────────────────────────┘
                             │ deer-flow 建立在下面两层之上
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  LangGraph + LangChain（Agent 编排框架）                    │
│  ┌──────────────────┐  ┌────────────────────────────────┐   │
│  │ LangGraph        │  │ LangChain                      │   │
│  │ 图式的 Agent 流程 │  │ LLM 抽象、Tool 协议、消息类型  │   │
│  │ 状态机、检查点   │  │ 各种 Provider 适配器            │   │
│  └──────────────────┘  └────────────────────────────────┘   │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  LLM Provider（大模型提供方）                               │
│  OpenAI、DeepSeek、豆包、Anthropic、Ollama、vLLM …          │
└─────────────────────────────────────────────────────────────┘
```

### 各层职责

- **LLM Provider**（最底层）：真正的大模型，提供「文字进、文字出」的能力。通过 API 调用。
- **LangChain / LangGraph**（中间层）：把各种不同的 LLM 统一成一套接口（这样你换模型不用改代码），定义了「工具」「消息」「Agent」的标准格式。LangGraph 在此基础上提供了「用图来编排 Agent 流程」的能力（状态机、循环、检查点）。
- **deer-flow**（上层）：在 LangGraph 之上，搭建了一个**完整的、生产可用的 Agent 运行系统**——加上沙箱、记忆、技能、子代理、Web 界面、IM 接入等等。你不用自己搭这些，开箱即用。
- **你的应用**（最上层）：在 deer-flow 之上，你做的具体定制——加自己的工具、写自己的技能、接自己的业务。

> 🔑 **关键认知**：学 deer-flow，你同时学会了 LangGraph 和 LangChain 的实战用法。这三个是「层层叠加」的关系，不是互相替代。

## 1.6 deer-flow 的演进：从「深度研究」到「超级 Agent」

了解一点历史，有助于你理解 deer-flow 为什么长现在这样。

deer-flow 经历了一次重大的版本演进：

### v1.0 时代：Deep Research Framework（深度研究框架）
最初，deer-flow 是一个**专门做「深度研究」**的框架——你给它一个研究课题，它会自动搜索、阅读、总结，最后产出一份研究报告。它的核心是一个固定的研究流程图（计划 → 搜索 → 阅读 → 总结 → 写报告）。

### v2.0 时代：Super Agent Harness（超级 Agent 运行平台）
开发者在实际使用中发现：这套系统能干的事远不止「研究」。有人拿它做数据流水线、有人拿它做内容自动化、有人拿它做 PPT 生成……**它本质上是一个「给 Agent 提供基础设施的运行平台」**，研究只是它的一个应用场景。

于是字节团队**从零重写了 v2.0**（和我们现在的代码完全不同，v1 在 `main-1.x` 分支）。v2.0 的核心理念变成了：

> 不再是「一个固定流程的研究框架」，而是「一个**可任意扩展的 Agent 运行平台**」。研究、做 PPT、分析数据……都只是它装载的「技能（Skill）」之一。

这就像从「一个只能做红烧肉的菜谱 App」进化成了「一个能做任何菜的智能厨房」。

### 这对你学习意味着什么
你学习的 deer-flow 2.0 是**架构最清晰、扩展性最强**的版本。它的设计哲学是「**万物皆可替换**」：

- 不喜欢某个 LLM？换。
- 不喜欢某个工具？换。
- 不喜欢某个沙箱实现？换。
- 想加个新能力？写个 Skill 或 Tool。

理解了这个「可替换、可扩展」的设计哲学，你就抓住了 deer-flow 的灵魂。

## 1.7 本章小结

让我们用几句话浓缩这一章：

- ✅ **LLM** 是「文字进、文字出」的大脑，**本身不能动手**
- ✅ **Agent** = LLM + 一套能动手的「装备」（工具、沙箱、记忆…）+ 自己决定怎么用这些装备
- ✅ Agent 的核心是 **ReAct 循环**：思考 → 行动 → 观察，不断循环直到完成任务
- ✅ 生产级 Agent 会撞上**状态管理、上下文压缩、工具管理、执行隔离、协作、记忆、可控性**等一堆问题——**这就是框架存在的意义**
- ✅ 生态层次：**LLM Provider → LangChain/LangGraph → deer-flow → 你的应用**
- ✅ deer-flow 从「深度研究框架」演进成了「可任意扩展的 Agent 运行平台」，核心哲学是**万物皆可替换**
