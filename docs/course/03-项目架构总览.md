## 3.2 宏观架构：三个服务 + 一个大脑

deer-flow 运行起来后，你的电脑上其实跑着**三个独立的服务进程**，它们通过 HTTP 互相通信：

```
┌─────────────────────────────────────────────────────────────────┐
│                        你的浏览器                                │
└──────────────────────────────┬──────────────────────────────────┘
                               │  访问 http://localhost:2026
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│              ① Nginx（反向代理，端口 2026）                     │
│                                                                  │
│   它是「大堂经理」，负责把请求分拣给不同的服务：                │
│   • /api/langgraph/* → 改写成 /api/* → 转给 ② Gateway            │
│   • /api/*（其他）   → 原样转给 ② Gateway                        │
│   • /（其他所有）    → 转给 ③ Frontend                           │
└──────────────┬───────────────────────────────────┬──────────────┘
               │                                    │
               ▼                                    ▼
┌──────────────────────────────┐       ┌──────────────────────────┐
│ ② Gateway API（端口 8001）   │       │ ③ Frontend（端口 3000）  │
│                              │       │                          │
│ FastAPI 后端，里面藏着：     │       │ Next.js 网页界面         │
│ ┌──────────────────────────┐ │       │ （你看到的聊天框）       │
│ │     Lead Agent ★大脑★    │ │       └──────────────────────────┘
│ │  (Agent 运行时)          │ │
│ └──────────────────────────┘ │
│ + REST API（模型/技能/记忆） │
└──────────────────────────────┘
```

### 为什么要分三个服务？

你可能会问：为什么要这么麻烦，一个进程搞定不行吗？这是经典的**关注点分离**思想：

| 服务 | 职责 | 为什么独立 |
|------|------|-----------|
| **Nginx** | 反向代理、统一入口 | 把「前端页面」和「后端 API」统一在同一个域名端口下（2026），避免浏览器的跨域问题 |
| **Gateway API** | 业务逻辑 + Agent 运行 | Python 写的，专注于 AI 逻辑，不用关心页面渲染 |
| **Frontend** | 用户界面 | Node.js 写的，专注于交互体验，不用关心 AI 细节 |

> 💡 **类比**：Nginx 是公司前台，Gateway 是后端技术部，Frontend 是前端设计部。访客（浏览器）只跟前台打交道，前台根据访客的需求分流到不同部门。

### 关键端口记忆

- **2026** = 你唯一需要记住的端口（Nginx，统一入口）
- 8001 = Gateway（后端，一般不直接访问）
- 3000 = Frontend（前端，一般不直接访问）

## 3.3 请求流转全链路：一次对话的旅行

这是本章最核心的内容。我们追踪「**用户在浏览器发了一条消息**」这个请求，看它怎么一路流转，最终返回 AI 的回答。

```
用户输入："帮我写个 hello.txt"
    │
    ▼ ① 浏览器发 HTTP POST 请求
    │   目标：http://localhost:2026/api/langgraph/threads/{id}/runs/stream
    │   方式：SSE（Server-Sent Events，服务端流式推送）
    │
    ▼ ② Nginx 收到请求
    │   匹配规则：/api/langgraph/* → 改写成 /api/* → 转发给 Gateway:8001
    │
    ▼ ③ Gateway API 收到请求
    │   路由到「Thread Runs」处理器
    │   创建一个后台「Run」（一次 Agent 执行任务）
    │
    ▼ ④ 调用 Agent 运行时
    │   RunManager 启动 run_agent()
    │   → 调用 make_lead_agent(config) 构造 Lead Agent
    │
    ▼ ⑤ Lead Agent 开始 ReAct 循环
    │   ┌─ 中间件链依次执行（记忆注入、上下文处理、工具过滤…）
    │   ├─ 调用 LLM（带上 system prompt + 历史 + 工具列表）
    │   ├─ LLM 决定调用 write_file 工具
    │   ├─ 中间件链拦截工具调用（沙箱审计、错误处理…）
    │   ├─ 执行 write_file（通过沙箱写到硬盘）
    │   ├─ 工具结果返回给 LLM
    │   └─ LLM 生成最终回答（不再调工具，循环结束）
    │
    ▼ ⑥ 中间件链「收尾」执行
    │   标题生成、记忆抽取入队、token 统计…
    │
    ▼ ⑦ 流式返回
    │   整个过程通过 SSE 实时把「思考」「工具调用」「文字」推给前端
    │   Gateway → Nginx → 浏览器
    │
    ▼ ⑧ 前端渲染
        浏览器收到 SSE 事件流，实时更新界面
        （打字机效果、工具调用卡片、思考过程展示）
```

### 流式响应（SSE）为什么重要

注意第 ⑦ 步用的是 **SSE（Server-Sent Events）**，不是普通的 HTTP 请求-响应。

普通 HTTP：你发请求 → 服务器算完 → **一次性**返回整个结果。
SSE：你发请求 → 服务器**边算边推** → 你实时看到每个字、每次工具调用。

这就是为什么你在 deer-flow 界面里能看到「打字机效果」和「实时工具调用卡片」——它不是等 Agent 全干完才显示，而是 Agent 每产生一个动作就立刻推给你。

> 这个流式机制我们在第 12 章「Gateway API」会结合代码细讲。

## 3.4 代码分层：Harness / App 双层架构

现在我们钻进代码看怎么组织的。deer-flow 的后端代码严格分成了两层，这是它架构上最核心的设计决策之一。

### 两层是什么

```
backend/
├── packages/harness/deerflow/    ← ① Harness 层（Agent 框架）
│   ├── agents/                      Agent 系统
│   ├── tools/                       工具系统
│   ├── sandbox/                     沙箱系统
│   ├── subagents/                   子代理
│   ├── memory/                      记忆（在 agents/memory/）
│   ├── skills/                      技能
│   ├── models/                      模型工厂
│   ├── mcp/                         MCP 集成
│   └── config/                      配置系统
│
└── app/                          ← ② App 层（应用外壳）
    ├── gateway/                    FastAPI Gateway API
    │   ├── app.py                    FastAPI 应用入口
    │   └── routers/                  各个 REST 路由
    └── channels/                   IM 渠道（飞书/Telegram…）
```

### 依赖规则：单向依赖

这两层之间有一个**铁律**：

```
   app（应用层）  ────可以 import────▶  deerflow（框架层）
   deerflow（框架层） ────禁止 import────▶  app（应用层）
```

即：**App 可以调用 Harness，但 Harness 绝对不能调用 App。**

这个规则由一个测试 `tests/test_harness_boundary.py` 在 CI 里强制保证。如果你在 deerflow 里写了 `from app.xxx import yyy`，CI 会直接报错。

### 为什么要这么分？

用一个问题来理解：**「如果字节明天把 deer-flow 打包成 PyPI 上的一个库发布，App 层的东西还需要吗？」**

答案是：**不需要**。因为：

- **Harness 层（`deerflow.*`）** = 通用的 Agent 框架能力。它发布成了一个独立的包 `deerflow-harness`。任何人都可以拿它来搭自己的 Agent 系统，不一定要用 deer-flow 的网页界面。
- **App 层（`app.*`）** = deer-flow 这个**具体产品**的外壳——它的 FastAPI 网关、它的 IM 渠道接入。这些是「deer-flow 这个应用」特有的，不是通用的。

> 💡 **类比**：Harness 是「引擎」，App 是「装了引擎的一辆整车」。引擎可以被装到任何车上（通用），但整车换不了（特定）。

### 这对你学习意味着什么

我们后面 8 章（第 04-11 章）学的**全都是 Harness 层**——Agent 的核心能力都在这里。第 12-13 章才涉及 App 层（API 和 IM 渠道）。

记住这个分层，后面看任何代码，先问自己：**「它在 Harness 层还是 App 层？」**这能帮你快速定位。

## 3.6 核心数据结构：ThreadState

在深入各个模块之前，我们必须认识一个贯穿全局的数据结构——**`ThreadState`（线程状态）**。

还记得第 01 章说的「状态管理」痛点吗？Agent 在多轮对话中需要记住很多东西：历史消息、当前用的沙箱、生成的文件、待办事项……**`ThreadState` 就是存放这一切的「大背包」**。

我们直接看代码（来自 `thread_state.py`）：

```python
# 文件：packages/harness/deerflow/agents/thread_state.py（简化版）

from typing import Annotated, NotRequired, TypedDict
from langchain.agents import AgentState   # LangChain 提供的基础状态

# ThreadState 继承自 LangChain 的 AgentState
# AgentState 已经包含了最核心的：messages（消息历史）
class ThreadState(AgentState):
    # sandbox：当前对话用的沙箱信息
    sandbox: NotRequired[SandboxState | None]           # 第 08 章

    # thread_data：这个对话的工作目录路径
    thread_data: NotRequired[ThreadDataState | None]    # 第 08 章

    # title：对话标题（自动生成）
    title: NotRequired[str | None]                      # 第 06 章

    # artifacts：Agent 生成的成品文件列表
    artifacts: Annotated[list[str], merge_artifacts]    # 第 07 章

    # todos：计划模式下的待办事项
    todos: Annotated[list | None, merge_todos]          # 第 06 章

    # uploaded_files：用户上传的文件
    uploaded_files: NotRequired[list[dict] | None]       # 第 12 章

    # viewed_images：Agent 看过的图片（视觉模型用）
    viewed_images: Annotated[dict[str, ViewedImageData], merge_viewed_images]  # 第 06 章

    # promoted：延迟加载的工具（MCP 工具搜索用）
    promoted: Annotated[PromotedTools | None, merge_promoted]  # 第 07 章
```

### 逐字段解读

| 字段 | 类型 | 干什么 | 哪章讲 |
|------|------|--------|--------|
| `messages` | list | **消息历史**（继承自 AgentState，最核心） | 全章 |
| `sandbox` | dict | 当前沙箱的 id | 第 08 章 |
| `thread_data` | dict | 工作区/上传/输出三个目录的路径 | 第 08 章 |
| `title` | str | 对话标题（第一轮对话后自动生成） | 第 06 章 |
| `artifacts` | list | Agent 生成的「成品」文件路径列表 | 第 07 章 |
| `todos` | list | 计划模式下的任务清单 | 第 06 章 |
| `uploaded_files` | list | 用户上传的文件信息 | 第 12 章 |
| `viewed_images` | dict | Agent「看」过的图片（base64） | 第 06 章 |
| `promoted` | dict | 被激活的延迟工具 | 第 07 章 |

### 神奇的 `Annotated[..., reducer]` 是什么？

你注意到有些字段写成了 `Annotated[list[str], merge_artifacts]` 这种形式。这里的 `merge_artifacts` 是一个 **reducer（归并函数）**，它告诉 LangGraph：**「当多个节点同时更新这个字段时，怎么合并它们」**。

举个例子，`merge_artifacts` 的作用是「合并并去重」：

```python
def merge_artifacts(existing: list[str] | None, new: list[str] | None) -> list[str]:
    """合并成品列表 - 合并并去重"""
    if existing is None:
        return new or []
    if new is None:
        return existing
    # 用 dict.fromkeys 去重，同时保持顺序
    return list(dict.fromkeys(existing + new))
```

为什么要这个？因为 LangGraph 支持并行执行——比如两个子代理同时往 `artifacts` 里塞文件，如果没有 reducer，后写的会覆盖先写的；有了 reducer，它们会被正确合并。

> 这个机制叫 **「Channel Reducer」**，是 LangGraph 的核心概念。现在你只要知道「`Annotated[类型, 合并函数]`」这种写法是用来定义「冲突时怎么合并」的就行，后面会在具体场景里反复看到。

## 3.8 动手实验：用日志验证请求链路

光看图不直观，我们来做个实验，亲眼看到请求是怎么流转的。

1. **开两个终端窗口**：
   - 终端 A：运行 `make dev`（看服务日志）
   - 终端 B：待命

2. **在浏览器里发一条消息**（比如「你好」）

3. **观察终端 A 的日志**，你会看到类似这样的输出：
   ```
   [gateway] INFO:     127.0.0.1 - "POST /api/threads/xxx/runs/stream HTTP/1.1" 200
   [gateway] Create Agent(default) -> thinking_enabled: True, model_name: gpt-4o...
   [gateway] Including view_image_tool for model 'gpt-4o' (supports_vision=True)
   [gateway] Total tools loaded: 12, built-in tools: 4, MCP tools: 0
   ```

   这些日志正好对应了我们前面讲的链路：
   - 第一行：Gateway 收到 `/api/.../runs/stream` 请求
   - 第二行：构造 Lead Agent（`Create Agent`）
   - 第三行：加载工具（`Total tools loaded`）

4. **试试在终端 B 直接访问 Gateway**：
   ```bash
   curl http://localhost:8001/health
   # → {"status":"ok"}   说明 Gateway 独立存活

   curl http://localhost:8001/api/models
   # → {"models":[...]}  说明 REST API 可以直接调
   ```

通过这个实验，你应该能直观感受到「**Nginx 转发 → Gateway 处理 → Agent 执行**」这条链路是真实存在的，而不只是纸上画的图。

## 3.9 本章小结

✅ deer-flow 是**三个服务**：Nginx(:2026 统一入口) → Gateway(:8001 后端+Agent) → Frontend(:3000 网页)
✅ 一次对话请求经过：**浏览器 → Nginx → Gateway → Lead Agent（中间件链 + ReAct 循环）→ 流式返回**
✅ 代码严格分两层：**Harness（通用框架，可独立发布）+ App（deer-flow 专属外壳）**，依赖单向（App→Harness）
