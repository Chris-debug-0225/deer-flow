简介

Sandbox 是 Agent 的安全保护子。当 Agent 执行代码、调用外部服务或访问文件系统时，Sandbox 会检查这些操作是否安全、是否符合权限。本章讲解 Sandbox 的设计原理、权限控制機制、代码执行的安全性。

学习目标：
  • 理解 Sandbox 的重要性
  • 掌握 Sandbox 的设计原理
  • 学会实现权限控制
  • 了解代码执行的安全性最佳实践

为什么需要 Sandbox

想象一个场景：Agent 接收了一个恶意的用户输入，不幸的是 LLM 不幸中了提示注入攻击，调用了一个应该删除数据的 Skill。没有 Sandbox 的情况下，数据会被立即删除。有了 Sandbox，即使 LLM 中了攻击，也会先检查这个操作是否安全、是否符合权限。

没有 Sandbox 的风险

风险类型 描述 后果
提示注入 LLM 中了提示注入攻击 执行恶意操作
权限推溌 Agent 执行超出权限的操作 数据泄露或系统破坡
资源浪費 Agent 执行超量计算或存储 系统资源耗尽
侧道攻击 通过时序或其他侧道推溌信息 信息泄露
关键点： Sandbox 是 Agent 系统中不可或缺的一部分。它不是可选的优化，而是必需的安全措施。

Sandbox 的设计

Sandbox 的核心思想是：不信任任何事物。每一个操作都需要经过严格的验证。

Sandbox 的分层设计

操作请求
→
身份验证
→
权限检查
→
安全上下文
→
执行

Sandbox 的实现

class Sandbox: """Sandbox 安全执行环境""" def __init__(self, config: SandboxConfig): self.config = config self.permissions = PermissionManager(config) self.audit_log = AuditLog() def execute(self, operation: Operation, context: ExecutionContext): """\u5728 Sandbox 中执行操作""" try: # 1. 身份验证 self.verify_identity(context) # 2. 权限检查 self.permissions.check(context.user, operation) # 3. 安全上下文检查 self.verify_context(context, operation) # 4. 执行操作 result = operation.execute() # 5. 记录审计日志 self.audit_log.log(context, operation, result) return result except PermissionDenied as e: self.audit_log.log_denied(context, operation, e) raise except Exception as e: self.audit_log.log_error(context, operation, e) raise

权限控制机制

权限控制是 Sandbox 的核心。它定义了不同的 Agent 或用户可以执行哪些操作。

权限模型

DeerFlow 使用基于角色的权限控制 (RBAC) 模型：

class Role: """\u6743限角色""" def __init__(self, name: str, permissions: List[str]): self.name = name self.permissions = set(permissions) # 预定义的角色 ROLE_ADMIN = Role( "admin", ["read", "write", "delete", "execute_code"] ) ROLE_USER = Role( "user", ["read", "write"] ) ROLE_GUEST = Role( "guest", ["read"] )

权限检查的实现

class PermissionManager: def check(self, user: User, operation: Operation) -> bool: """\u68c0查用户是否有权限执行操作""" required_permission = operation.required_permission # 检查用户的角色是否有所需权限 if required_permission not in user.role.permissions: raise PermissionDenied( f"User {user.id} lacks permission: {required_permission}" ) return True
示例： 权限检查的实际应用
# 创建用户 admin_user = User(id="admin1", role=ROLE_ADMIN) guest_user = User(id="guest1", role=ROLE_GUEST) # 删除操作（需要 delete 权限） delete_op = Operation( type="delete", required_permission="delete" ) # Admin 可以执行 sandbox.execute(delete_op, ExecutionContext(user=admin_user)) # Guest 无法执行，会抛出 PermissionDenied 异常 sandbox.execute(delete_op, ExecutionContext(user=guest_user))

代码执行的安全性

当 Agent 需要执行代码（比如执行 Python 脚本）时，Sandbox 需要提供一个安全的执行环境。

代码执行的安全需求

安全需求 说明 实现方案
资源限制 限制 CPU、内存、磁盘使用 使用 cgroups 或 Docker
文件系统隔离 限制文件访问范围 使用 chroot 或 Docker
网络隔离 控制网络访问 使用防火墙或网络策略
时间限制 限制执行时间 使用 timeout 或信号

安全代码执行的实现

class CodeExecutor: def execute(self, code: str, timeout: int = 30) -> str: """\u5b89全执行代码""" try: # 1. 代码静态分析（检查危险的操作） self.validate_code(code) # 2. 在隔离的环境中执行 with IsolatedEnvironment() as env: result = env.execute(code, timeout=timeout) return result except DangerousCodeDetected as e: raise SecurityError(f"Dangerous code detected: {e}") except TimeoutError: raise ExecutionTimeout("Code execution timeout") def validate_code(self, code: str): """\u68c0查代码是否包含危险操作""" dangerous_patterns = [ "os.system", "subprocess", "__import__", "eval", "exec" ] for pattern in dangerous_patterns: if pattern in code: raise DangerousCodeDetected( f"Dangerous pattern found: {pattern}" )
警告： 不要信任任何代码。即使是你自己的代码，也应该经过 Sandbox 执行。不要直接使用 eval() 或 exec() 执行用户提供的代码。

隔离技术

Sandbox 可以使用多种隔离技术来实现安全执行环境。不同的技术有不同的权衡。

隔离技术对比

技术 隔离程度 性能开销 适用场景
进程隔离 中等 低 轻量级任务
容器隔离 高 中等 中等复杂度任务
虚拟机隔离 非常高 高 高风险任务
语言级隔离 低 很低 受信任的代码

Docker 容器隔离示例

使用 Docker 容器提供强大的隔离：

import docker from typing import Dict class DockerSandbox: """使用 Docker 容器的 Sandbox""" def __init__(self, image: str = "python:3.9"): self.client = docker.DockerClient() self.image = image def execute(self, code: str, timeout: 30) -> str: """在 Docker 容器中执行代码""" try: # 创建容器 container = self.client.containers.run( self.image, f"python -c {code}", detach=True, mem_limit="512m", # 限制内存 cpus="1", # 限制 CPU network_disabled=True # 禁用网络 ) # 等待执行完成 exit_code = container.wait(timeout=timeout) # 获取输出 output = container.logs().decode() # 清理容器 container.remove() return output except Exception as e: raise SandboxError(f"Execution failed: {e}")

资源限制配置

Sandbox 应该配置合理的资源限制：

class ResourceLimits: """资源限制配置""" def __init__(self): self.max_memory = "512m" # 最大内存 self.max_cpu = "1" # 最大 CPU 核心数 self.max_timeout = 30 # 最大执行时间（秒） self.max_file_size = 10 * 1024 * 1024 # 最大文件大小（字节） self.max_processes = 10 # 最大进程数

监控与审计

Sandbox 应该提供完整的监控和审计能力，记录所有的操作，便于事后分析和安全审查。

审计日志的内容

审计日志应该记录以下信息：

class AuditLog: """审计日志""" def log(self, event: AuditEvent): """记录审计事件""" log_entry = { "timestamp": time.time(), "user_id": event.user_id, "action": event.action, "resource": event.resource, "result": event.result, # success / denied / error "reason": event.reason, "details": event.details } # 存储到数据库或日志文件 self.store(log_entry) class AuditEvent: def __init__( self, user_id: str, action: str, resource: str, result: str, reason: str = "", details: Dict = None ): self.user_id = user_id self.action = action self.resource = resource self.result = result self.reason = reason self.details = details or {}

实时监控

除了审计日志，还应该进行实时监控，检测异常行为：

class SandboxMonitor: """Sandbox 监控器""" def detect_anomalies(self, logs: List[AuditEvent]): """检测异常行为""" anomalies = [] # 检测频繁的权限拒绝 denied_count = sum(1 for log in logs if log.result == "denied") if denied_count > 10: anomalies.append({ "type": "frequent_denials", "count": denied_count }) # 检测异常的资源使用 for log in logs: if log.details.get("memory_usage", 0) > 400: # MB anomalies.append({ "type": "high_memory_usage", "usage": log.details["memory_usage"] }) return anomalies
监控建议： 设置告警阈值，当检测到异常行为时立即告警。同时定期分析审计日志，发现潜在的安全问题。

最佳实践

1. 最小权限原则

仅给予 Agent 它真正需要的权限。不要给予过广泛的权限。

2. 定期审计

定期检查 Sandbox 的审计日志，发现不正常的活动。

3. 分层防御

不要仅依赖一个安全措施。应该使用多个安全层次。

4. 定期更新

定期更新 Sandbox 的依赖项目和安全补丁。

总结

Sandbox 是 Agent 系统中的安全保护层。它确保了 Agent 的操作不会对系统造成危害。本章深入讲解了 Sandbox 的设计、权限控制、代码执行安全、隔离技术和监控审计。

核心要点总结：
  • 安全重要性： Sandbox 是必需的，不是可选的优化
  • 分层设计： 身份验证 → 权限检查 → 安全上下文 → 执行 → 审计
  • 权限控制： 使用 RBAC 模型，遵守最小权限原则
  • 代码执行安全： 静态分析 + 资源限制 + 时间限制
  • 隔离技术： 进程隔离、容器隔离、虚拟机隔离等多种选择
  • 监控审计： 完整的审计日志 + 实时异常检测
  • 最佳实践： 最小权限、定期审计、分层防御、定期更新
安全建议： Sandbox 的安全性取决于多个因素的综合。不要仅依赖一个安全措施，而应该使用多层防御。同时，要定期审查和更新安全策略，应对新的威胁。
