Sandbox与安全执行环境

DeerFlow 学习系列 - 第 12 章

本章讲的是 DeerFlow 如何把“文件与命令执行”收进受控边界

DeerFlow 的 sandbox 不是抽象上的“安全盒子”概念，而是一组真实的路径映射、provider 选择、延迟获取、命令约束和文件工具实现。理解这一章的关键，是分清虚拟路径契约、provider 差异以及 shell 能力的安全边界，而不是把 Local sandbox 误解成强隔离容器。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 需要让 agent 读写文件、搜索目录、执行命令、产出工件，但又不能把宿主机文件系统和 shell 直接裸露给模型。sandbox 层正是为这个矛盾服务：一方面给 agent 一个稳定、可推理的工作空间视图，另一方面把真实宿主路径、权限范围和命令访问方式收敛到受控工具里。

在 DeerFlow 里，sandbox 的职责不只是“执行 bash”。它更重要的职责是建立统一的虚拟路径契约，例如 /mnt/user-data/workspace、/mnt/user-data/uploads、/mnt/user-data/outputs 与 /mnt/acp-workspace，让 prompt、tool、artifact API 与不同 provider 都围绕同一套路径语义工作。

最重要的认知边界： Local sandbox 是主机侧便利模式，不是强隔离 shell 边界。它解决的是 DeerFlow 的统一执行接口问题，不等于容器级安全隔离。

真实运行链路或数据链路

DeerFlow 默认不是“run 开始就先启动 sandbox”。它采用 lazy sandbox acquisition：只有当运行真的触碰到文件/命令工具时，sandbox 才通过 middleware 和 provider 被获取。这一点在 backend/packages/harness/deerflow/sandbox/middleware.py 与 backend/packages/harness/deerflow/agents/factory.py 的 lazy init 组合里非常明确。

lead agent / subagent runtime -> sandbox middleware (lazy_init=True) -> first sandbox-backed tool call -> backend/packages/harness/deerflow/sandbox/tools.py -> get_sandbox_provider().acquire(thread_id) -> LocalSandboxProvider 或 AioSandboxProvider -> path mapping + bounds check + tool execution -> 返回虚拟路径视角下的结果

如果当前 provider 是 backend/packages/harness/deerflow/sandbox/local/local_sandbox_provider.py，那么它会为每个线程建立独立的 path mappings，把真实宿主目录映射成统一的 /mnt/user-data/workspace、/mnt/user-data/uploads、/mnt/user-data/outputs 和 /mnt/acp-workspace。如果当前 provider 是 AioSandboxProvider，这些路径则会在容器内以 bind mount 形式呈现。

必须记住的运行语义： 对 agent 来说，看到的是稳定的虚拟路径；对 DeerFlow 来说，真正的隔离能力取决于 provider。路径契约是一致的，隔离强度并不一致。

配置面与使用方式

从配置和使用层看，sandbox 至少有四个关键面：

  • backend/docs/CONFIGURATION.md 里的 sandbox.use 决定当前采用本地 provider 还是 AioSandboxProvider。
  • backend/docs/CONFIGURATION.md 里的 allow_host_bash: false 默认关闭宿主机 bash；只有显式启用后，本地模式才允许 host bash。
  • backend/packages/harness/deerflow/sandbox/local/local_sandbox_provider.py 会根据线程和用户上下文建立 /mnt/user-data/... 映射，这意味着路径不是全局固定磁盘目录，而是和 thread/user 绑定。
  • sandbox.mounts 可以额外挂载自定义目录，但不能覆盖保留前缀，例如 /mnt/user-data 与 /mnt/acp-workspace。

这也是为什么 prompt、artifact API、上传系统和文件工具都坚持使用虚拟路径。只要契约不变，provider 可以更换，部署方式也可以变化。

关键实现细节与权衡

  • backend/packages/harness/deerflow/sandbox/tools.py 不只是执行工具，还负责把 /mnt/user-data/* 虚拟路径解析成线程级真实路径，并校验是否越界。
  • lazy sandbox acquisition 避免了每次 run 都预先创建 provider 资源，尤其对容器化 provider 更重要；代价是第一次工具调用时才真正拿到 sandbox，读源码时要把这一步考虑进去。
  • LocalSandboxProvider 通过每线程 sandbox 和 LRU 缓存保持路径一致性与一定的性能，但它本质上仍然是在宿主机文件系统上工作。
  • AioSandboxProvider 提供更接近强隔离的容器化边界，因此 DeerFlow 文档明确把它作为需要隔离 shell 访问时的优先方案。
  • allow_host_bash: false 默认值不是保守过度，而是显式承认“本地模式不是安全 shell 边界”的设计现实。
核心取舍： DeerFlow 优先保证统一的虚拟路径与工具接口，再通过 provider 分层提供不同隔离强度，而不是假装所有运行模式都具有相同安全属性。

当前限制、边界与非目标

  • Local sandbox 不是容器安全边界。即使它对 agent 暴露的是受控工具接口，也不应被当成不可信代码执行的强隔离环境。
  • 设置 allow_host_bash: true 只适用于完全受信任的本地单用户场景；这不是 DeerFlow 推荐的默认部署模式。
  • /mnt/user-data/... 是 DeerFlow 对 agent 暴露的逻辑契约，不意味着宿主机上真实目录结构必须和它逐字对应。
  • 本章不展开每个文件工具的全部参数，也不替代 artifact API 文档；重点是 sandbox 语义、provider 边界和路径模型。
如果你的目标是“让 agent 能安全跑 shell”，正确的问题不是“怎么把 Local sandbox 配得更像容器”，而是“为什么这里不直接使用 AioSandboxProvider”。

它与 DeerFlow 其他能力的关系

sandbox 是 DeerFlow 运行时的基础设施层，不是孤立能力：

  • 它和第 10 章执行引擎相连，因为 sandbox 通过 runtime middleware 进入主执行链，而且采用 lazy acquisition，而不是 eager boot。
  • 它和第 13 章长期记忆相连，因为 memory、uploads、artifacts 等很多数据最终都依赖线程级路径与用户隔离设计，而 backend/docs/AUTH_DESIGN.md 正是这层语义的背景文档。
  • 它和第 14 章 sub-agents 相连，因为 subagent 也会继承父运行时的 sandbox_state、thread_data 和虚拟路径语义，但 ACP agent 又明确不在 /mnt/user-data 工作。
  • 它和 Gateway/API/文件展示链路相连，因为 artifact URL 暴露的是 DeerFlow 对虚拟路径工件的 HTTP 访问面，而不是直接公开宿主机真实路径。
建议把 backend/packages/harness/deerflow/sandbox/tools.py、backend/packages/harness/deerflow/sandbox/local/local_sandbox_provider.py、backend/packages/harness/deerflow/sandbox/middleware.py、backend/docs/CONFIGURATION.md 和 backend/docs/AUTH_DESIGN.md 连起来读。这样最容易看清 DeerFlow 的路径契约、provider 分层与安全边界。
