简介

在 DeerFlow 中，Skill 和 Tool 是两个容易混淆但本质完全不同的概念。 理解它们的区别是掌握 DeerFlow 能力扩展体系的关键。

一句话概括： Tool 是可执行的 Python 函数（如搜索网页、执行代码）， Skill 是 Markdown 格式的工作流指令文档（如"如何做深度研究"的方法论）。 Agent 通过 Tool 执行动作，通过 Skill 获取"做事的方法论"。

学习目标：
  • 理解 Skill 和 Tool 的本质区别
  • 掌握 SKILL.md 的结构和 YAML frontmatter 元数据
  • 理解 Skill 如何通过系统提示注入影响 Agent 行为
  • 了解 Skill 的加载、启用和自我进化机制

Skill vs Tool：核心区别

这是本章最重要的部分。很多人（包括之前的版本的本文档）把 Skill 描述为"包装了 Python 函数的能力单元"—— 那其实是 Tool 的定义，不是 Skill。

常见误解：Skill 就是 Tool 的另一个名字，都是可执行的函数。
事实：Skill 不是可执行代码。Skill 是 Markdown 文档， 它不包含任何 Python 函数，不能被"调用"或"执行"。

对比表

维度 Tool（工具） Skill（技能）
本质 Python 函数（langchain.tools.BaseTool） Markdown 文档（SKILL.md）
是否可执行 ✅ 是，LLM 直接调用执行 ❌ 否，是一段指令文本
配置位置 config.yaml 中的 tools: 列表 skills/ 目录下的 SKILL.md 文件
如何加载 Python 动态导入（resolve_variable） 文件系统扫描 + YAML frontmatter 解析
如何影响 Agent 绑定到 Agent，LLM 可直接调用 注入系统提示，Agent 按需 read_file 读取
典型示例 bash_tool（执行 shell 命令）
web_search（搜索网页）		 deep-research（深度研究方法论）
skill-creator（创建新 Skill 的指南）
数据模型 langchain_core.tools.BaseTool @dataclass Skill（见 skills/types.py）
比喻 手中的工具（锤子、搜索引擎） 脑中的方法论（"如何做调研"的 SOPS）

用类比理解

想象你在写一篇学术论文：

  • Tool = 你使用的工具：Google Scholar（搜索文献）、Python（分析数据）、Word（排版）
  • Skill = 你掌握的方法论："如何系统地做文献综述"——这是一份写在纸上的指南，告诉你先搜什么、再搜什么、怎么筛选、怎么组织，但它本身不能帮你搜文献

Agent 有了 Tool（搜索工具），但不知道"先搜什么后搜什么、搜到之后怎么交叉验证"。 Skill（方法论文档）告诉它这些步骤，然后 Agent 一步步地调用 Tool 来执行。

源码定义对比

# ━━━ Tool：可执行函数（backend/.../tools/tools.py）━━━ # Tool 是 LangChain BaseTool 实例，有 _run 方法，可以被 LLM 直接调用 # config.yaml 中配置： tools: - use: deerflow.community.ddg_search.tools:web_search_tool # ← Python 可导入路径 - use: deerflow.sandbox.tools:bash_tool # 加载时通过 resolve_variable 动态导入这些 Python 函数对象 # ━━━ Skill：Markdown 文档（backend/.../skills/types.py）━━━ # Skill 是 dataclass，没有 func 属性，不可执行 @dataclass class Skill: name: str # 从 SKILL.md frontmatter 读取 description: str # 从 SKILL.md frontmatter 读取 license: str | None # 许可证 skill_dir: Path # SKILL.md 所在目录 skill_file: Path # SKILL.md 文件路径 relative_path: Path # 相对于 skills 目录的路径 category: SkillCategory # 'public' 或 'custom' allowed_tools: list[str] | None = None # 可选：限制可用工具 enabled: bool = False # 是否启用 # 注意：没有 func、没有 execute 方法！

Skill 的磁盘结构

Skill 以文件夹为单位，每个 Skill 是 skills/ 目录下的一个子目录， 必须包含一个 SKILL.md 文件。这是你在项目 skills/ 目录中实际看到的结构：

skills/ # ← Skill 根目录 ├── public/ # 内置 Skill（只读，随平台分发） │ ├── deep-research/ │ │ └── SKILL.md # 深度研究方法论（最简结构） │ │ │ ├── bootstrap/ │ │ ├── SKILL.md # 核心标识文件 │ │ ├── templates/SOUL.template.md # 模板文件 │ │ └── references/conversation-guide.md # 参考文档 │ │ │ ├── skill-creator/ # 最复杂的 Skill（创建新 Skill 的指南） │ │ ├── SKILL.md # 32KB 的核心指令文档 │ │ ├── agents/ # 子 Agent 角色定义 │ │ │ ├── analyzer.md # 分析角色 │ │ │ ├── comparator.md # 比较角色 │ │ │ └── grader.md # 评分角色 │ │ ├── references/ # 参考文档 │ │ │ ├── output-patterns.md │ │ │ ├── schemas.md │ │ │ └── workflows.md │ │ ├── scripts/ # 辅助脚本 │ │ │ ├── init_skill.py │ │ │ ├── package_skill.py │ │ │ └── quick_validate.py │ │ ├── assets/ # 静态资源 │ │ │ └── eval_review.html │ │ └── LICENSE.txt │ │ │ └── ... (共 21 个内置 Skill) │ └── custom/ # 用户自定义 Skill（可编辑、可删除） ├── .history/ # 变更历史（JSONL 格式） │ └── my-skill.jsonl └── my-skill/ └── SKILL.md

支撑文件的允许目录：Skill 内部除了 SKILL.md， 还可以包含以下子目录中的资源（见 skill_storage.py）：

  • references/ — 参考文档，Agent 可按需 read_file 读取
  • templates/ — 模板文件
  • scripts/ — 辅助脚本（Agent 可通过 bash_tool 执行）
  • assets/ — 静态资源（HTML、图片等）

SKILL.md 是唯一必需的文件，其他都是可选的。 这正是你平时看到的 Skill 结构——一个 .md 文档做标识， 其他资源文件作为使用过程中的辅助材料。

SKILL.md 详解

SKILL.md 是每个 Skill 的核心。它由两部分组成： YAML frontmatter（元数据）和 Markdown 正文（工作流指令）。

实际示例：deep-research 的 SKILL.md

# ━━━ skills/public/deep-research/SKILL.md ━━━ --- # YAML frontmatter（元数据） name: deep-research # ← Skill 的唯一标识 description: > # ← LLM 看到的描述（决定何时使用此 Skill） Use this skill instead of WebSearch for ANY question requiring web research. Trigger on queries like "what is X", "explain X", "compare X and Y", "research X". Provides systematic multi-angle research methodology. --- # Deep Research Skill # Markdown 正文（工作流指令） ## Overview # ← 这部分不被解析，由 Agent 运行时 read_file 读取 This skill provides a systematic methodology for conducting thorough web research. Load this skill BEFORE starting any content generation task. ## When to Use This Skill **Always load this skill when:** - User asks "what is X", "explain X", "research X" - The question requires current, comprehensive information ## Core Principle 1. Break the research question into sub-questions 2. Search each sub-question from multiple angles 3. Cross-validate findings across sources 4. Synthesize into a coherent answer ...（几百行的方法论指令）

YAML frontmatter 字段

frontmatter 中 --- 之间的部分被 skills/parser.py 解析。 允许的字段（见 skills/validation.py）：

字段 必需 说明
name ✅ Skill 的唯一标识，也是目录名
description ✅ 功能描述，注入到系统提示中供 LLM 判断何时使用
license ❌ 开源许可证
allowed-tools ❌ 限制此 Skill 启用时 Agent 可用的 Tool 列表
version ❌ 版本号
author ❌ 作者
metadata ❌ 自定义元数据
compatibility ❌ 兼容性信息

description 是最重要的字段。 它被注入到 Agent 的系统提示中，LLM 根据它判断"当前用户问题是否需要加载这个 Skill"。 写得越精确，Agent 判断越准确。

Skill 的加载机制

DeerFlow 在创建 Agent 时加载 Skills。整个过程涉及扫描目录、解析 frontmatter、 注入系统提示。以下是完整的加载流程：

扫描 skills/ 目录
→
解析 SKILL.md
→
构建 Skill 对象
→
过滤 enabled
→
注入系统提示

关键源码路径

文件 职责
skills/storage/local_skill_storage.py 扫描 skills/public/ 和 skills/custom/ 目录
skills/parser.py 解析 SKILL.md 的 YAML frontmatter，构建 Skill 对象
skills/validation.py 验证 frontmatter 字段是否合法
agents/lead_agent/prompt.py 将启用的 Skills 注入 Agent 的系统提示
skills/tool_policy.py 根据 Skill 的 allowed-tools 过滤可用 Tool

Skill 如何影响 Agent 行为

Skill 不是被"调用"的，而是通过两个机制影响 Agent 的行为：

机制一：注入系统提示

当 Agent 启动时，prompt.py 中的 get_skills_prompt_section() 会生成一段 XML，列出所有启用的 Skill，注入到 Agent 的系统提示中：

# 来源：agents/lead_agent/prompt.py 第 609-623 行 # 这段 XML 被拼接到 Agent 的 system_prompt 中 <skill_system> You have access to skills that provide optimized workflows for specific tasks. Each skill contains best practices, frameworks, and references. **Progressive Loading Pattern:** 1. When a user query matches a skill's use case, immediately call `read_file` on the skill's main file using the path attribute provided below 2. Read and understand the skill's workflow and instructions <available_skills> <skill> <name>deep-research</name> <description>Use this skill for ANY question requiring web research... [built-in]</description> <location>/mnt/skills/public/deep-research/SKILL.md</location> </skill> <skill> <name>skill-creator</name> <description>Guide for creating effective skills... [built-in]</description> <location>/mnt/skills/public/skill-creator/SKILL.md</location> </skill> </available_skills> </skill_system>

注意：系统提示中不包含 SKILL.md 的正文内容， 只包含 name、description 和文件路径。Agent 看到匹配的场景时， 会主动用 read_file Tool 去读取完整的 SKILL.md。 这就是"渐进式加载"——避免一次性把所有 Skill 的正文塞满上下文。

机制二：过滤可用 Tool

如果 Skill 的 frontmatter 中声明了 allowed-tools，当该 Skill 启用时， Agent 的可用 Tool 列表会被限制为这些工具：

# 来源：skills/tool_policy.py # 核心逻辑只有几行 def filter_tools_by_skill_allowed_tools(tools, skills): allowed = allowed_tool_names_for_skills(skills) if allowed is None: return tools # 没有 Skill 声明 allowed-tools → 不过滤 return [tool for tool in tools if tool.name in allowed] # SKILL.md 中的用法： # --- # name: my-restricted-skill # allowed-tools: # - web_search # - read_file # --- # 启用后，Agent 只能用 web_search 和 read_file，不能用 bash_tool 等

完整的运行时流程

# ━━━━━━━━━━━━ 一个真实场景：用户问"研究一下 2024 年 AI 发展趋势" ━━━━━━━━━━━━ # 步骤 1：Agent 启动时 # 系统提示中包含了 <skill_system>...</skill_system> XML # Agent 知道有 deep-research 这个 Skill 可用 # 步骤 2：用户提问 → LLM 推理 # LLM 看到 description 匹配："研究" → 需要 web research # LLM 决定："先加载 deep-research Skill" # 步骤 3：Agent 调用 read_file Tool（注意：read_file 是 Tool，不是 Skill！） # read_file("/mnt/skills/public/deep-research/SKILL.md") # → 返回完整的 Markdown 正文（几百行方法论） # 步骤 4：Agent 按照 SKILL.md 中的指令一步步执行 # - 拆解子问题 # - 用 web_search Tool 搜索每个子问题 # - 交叉验证 # - 综合成答案 # ★ 关键理解： # Skill 提供了"做什么、怎么做的步骤" # Tool 提供了"实际执行动作的能力" # Agent 读 Skill（获取方法论）→ 调 Tool（执行动作）→ 完成任务

总结：Skill 和 Tool 是互补的，不是替代关系。
Skill 告诉 Agent "遇到 X 类问题，应该按什么步骤处理"；
Tool 让 Agent 有能力实际执行每一步（搜索、读文件、执行代码等）。

Skill 的自我进化

DeerFlow 的一个独特设计是：Agent 可以自己创建和修改 Skill。 这是通过 skill_manage_tool（一个 Tool）实现的—— 注意这里的有趣关系：用 Tool 来管理 Skill。

# 来源：tools/skill_manage_tool.py # skill_manage_tool 是一个 LangChain Tool，用来管理 Skill（Markdown 文件） # Agent 可以执行的操作： { "operation": "create", # 创建新 Skill "operation": "edit", # 编辑现有 Skill "operation": "delete", # 删除 Skill "operation": "patch", # 局部修改 Skill } # Agent 发现某种工作流反复有效 → 写成新的 SKILL.md → 下次遇到同类问题自动加载 # 这就是"自我进化"——Agent 的方法论库会越来越丰富

自定义 Skill 的变更历史保存在 skills/custom/.history/<name>.jsonl 中， 以 JSONL 格式记录每次编辑，支持回溯。

Python 测试基础（附 assertRaises 详解）

虽然 Skill 本身是 Markdown 文档不需要测试，但 DeerFlow 的 Tool 和其他 Python 模块 需要单元测试。这里讲解一个常见的测试模式：assertRaises。

import unittest class TestWeatherTool(unittest.TestCase): def test_invalid_city(self): """测试无效的城市""" with self.assertRaises(ValueError): self.tool._run(city="不存在的城市")

assertRaises 的逻辑和普通断言相反：
普通 assertEqual(a, b) 验证的是"结果应该等于预期"——不报错才算通过。
assertRaises(ValueError) 验证的是"代码应该报错"—— 抛出了指定异常才算通过；如果代码正常跑完没报错，测试反而失败。

# assertRaises 的两种用法 # ━━━ 用法 1：上下文管理器（推荐）━━━ with self.assertRaises(ValueError): self.tool._run(city="不存在的城市") # ✅ _run 抛出 ValueError → 测试通过 # ❌ _run 正常返回 → 测试失败 # ━━━ 用法 2：直接传 callable ━━━ self.assertRaises(ValueError, self.tool._run, city="火星") # ━━━━━━━━━━━━ 用实际场景理解 ━━━━━━━━━━━━ # 假设 weather_tool 的 _run 内部有参数校验： def _run(self, city): valid_cities = ["北京", "上海", "广州"] if city not in valid_cities: raise ValueError(f"不支持的城市: {city}") return query_weather(city) # 测试用例 2：无效城市 → 必须抛 ValueError def test_invalid_city(self): with self.assertRaises(ValueError): self.tool._run(city="不存在的城市") # → 触发 ValueError → 测试通过

总结

Skill 和 Tool 是 DeerFlow 能力扩展体系的两大支柱，它们互补而非等同。

核心要点总结：
  • Tool = 可执行 Python 函数（如 web_search、bash_tool），LLM 直接调用执行
  • Skill = Markdown 工作流文档（如 deep-research），注入系统提示，Agent 按需 read_file 读取
  • SKILL.md 是核心文件：YAML frontmatter（元数据）+ Markdown 正文（方法论指令）
  • Skill 不被"调用"，而是通过注入系统提示和过滤 Tool 两个机制影响 Agent 行为
  • 渐进式加载：系统提示只含 name + description + 路径，正文由 Agent 按需读取
  • 自我进化：Agent 通过 skill_manage_tool 创建/编辑/删除 Skill
一句话记住： Tool 是 Agent 的"手"（执行动作），Skill 是 Agent 的"脑"（做事的方法论）。 Agent 读 Skill 获取步骤 → 调 Tool 执行每一步 → 完成任务。
