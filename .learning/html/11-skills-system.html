Skills系统：Agent的能力扩展

DeerFlow 学习系列 - 第 11 章

本章讲的是 DeerFlow 怎样管理“可注入能力面”

在 DeerFlow 里，Skill 不是一个抽象概念标签，而是一套可安装、可启停、可扫描、可注入到 system prompt 的能力包。真正要理解的是：Skill 如何从磁盘被发现，如何通过 Gateway 管理，如何进入 lead agent prompt，以及为什么它和普通 tool、以及第 30 章的“自定义 Skill 编写流程”不是一回事。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 需要一种比“再加一个 tool”更高层的能力扩展机制。原因很直接：很多能力不只是新增某个函数调用，还需要给 agent 补充操作约定、工作流、注意事项、参考模板，甚至告诉它什么时候应该触发某类工具搜索与执行策略。Skill 系统就是为这个问题设计的。

因此，本章中的 Skill 主要解决三件事：把一组能力说明和辅助资源组织成可发现目录；让这些目录能够被安装、启停、编辑和安全扫描；把启用中的 Skill 摘要注入到 lead agent 的系统提示中，让 DeerFlow 在运行前就拥有“该如何工作”的额外能力面。

一个容易混淆的边界： Skill 不是 Tool 的别名。Tool 是运行时可调用接口；Skill 更像运行前注入的能力包与工作说明。第 30 章 30-custom-skills-guide.html 讲“用户如何定义和构建自定义 skills”，本章讲“DeerFlow 的 skills 系统本身如何工作”。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/packages/harness/deerflow/skills/ Skill 类型、校验、安装、存储、slash 激活等核心目录 理解 DeerFlow skills 系统不能只看某一个文件，这个目录才是主战场
backend/packages/harness/deerflow/skills/installer.py 安装归档解包、安全提取、扫描和落盘 说明 Skill 不是随便把 ZIP 解开，而是经过路径与内容安全约束
backend/app/gateway/routers/skills.py Gateway 暴露的 skills 列表、安装、编辑、启停、历史与回滚接口 解释外部控制面如何进入 DeerFlow 的 skills 生命周期
backend/packages/harness/deerflow/agents/lead_agent/prompt.py 缓存启用中的 skills，并把它们转成 system prompt 片段 这是 Skill 真正影响 agent 行为的入口
backend/packages/harness/deerflow/tools/builtins/tool_search.py 延迟工具发现机制 它不是 Skill 实现，但经常与 Skill 协同，让 agent 先拥有工作方法，再按需发现 MCP 工具

真实运行链路或数据链路

一个典型的 Skill 生效链路可以分成“管理面”和“运行面”两段：

管理面： backend/app/gateway/routers/skills.py -> get_or_new_skill_storage(...) -> backend/packages/harness/deerflow/skills/installer.py -> scan_skill_content / storage write / history -> refresh_skills_system_prompt_cache_async() 运行面： backend/packages/harness/deerflow/agents/lead_agent/prompt.py -> get_or_new_skill_storage().load_skills(enabled_only=True) -> cache enabled skills -> render skills prompt section -> lead agent system prompt receives skill instructions

这条链说明，Skill 的主要作用不是在运行过程中“像 tool 一样被直接调用”，而是先通过 Gateway 或本地存储进入 skills 仓库，再由 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 读取启用态并注入 prompt。只有当 agent 开始思考和规划时，这些 Skill 才真正改变它的行为。

另外一个经常一起出现的链路，是 Skill 与延迟工具发现的配合。DeerFlow 可以通过 backend/packages/harness/deerflow/tools/builtins/tool_search.py 先只暴露 MCP 工具名称，再让 agent 按需搜索工具 schema。很多 Skill 实际上是在告诉 agent：什么时候应该去搜索哪类工具，而不是替代这些工具本身。

配置面与使用方式

从使用者视角看，DeerFlow 的 Skill 系统至少有四个配置和操作面：

  • 通过 backend/app/gateway/routers/skills.py 的 API 列出全部 skills、安装归档、启停单个 skill、读取自定义 skill 内容、查看历史并回滚。
  • 通过 ExtensionsConfig 与 skills enabled 状态控制某个 skill 是否进入当前运行时的 prompt 注入。
  • 通过 skills storage 决定从哪里加载 skills，本地默认会落到 DeerFlow 配置指定的 skills 根目录，再映射到容器内路径。
  • 通过第 30 章中的自定义 Skill 构建流程新增内容，但那一章关注的是“怎么写一个 skill 包”，不是本章的系统运行逻辑。
读代码时要特别注意： `enable/disable` 并不是只改一个前端勾选状态。它会影响 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 的缓存内容，进而改变后续新建 run 时 agent 能看到的系统提示。

关键实现细节与权衡

  • DeerFlow 选择把 Skill 作为目录与文档资源集合，而不是数据库里的一段纯文本配置，这让 Skill 可以携带脚本、模板、references 等支持文件。
  • backend/packages/harness/deerflow/skills/installer.py 明确加入了 ZIP 路径穿越检测、符号链接跳过、解压体积限制和内容扫描，说明 Skill 安装被当作一个安全边界处理。
  • backend/packages/harness/deerflow/agents/lead_agent/prompt.py 采用后台线程预热与缓存失效机制，而不是每次请求都同步扫描 skills 目录，这是为了避免把磁盘 I/O 和 prompt 组装直接绑死在请求路径上。
  • Skill 与工具发现分离。backend/packages/harness/deerflow/tools/builtins/tool_search.py 只负责延迟暴露 MCP 工具 schema；Skill 负责改变 agent 的工作方法。分开设计让 DeerFlow 不必把“能力说明”和“工具协议”混成一层。
核心取舍： DeerFlow 并没有把 Skill 设计成“万能插件接口”。它更像“提示词级能力包 + 受控资源目录 + 管理 API”，这样既能扩展 agent 行为，又不必让每个 Skill 都直接嵌入执行引擎。

当前限制、边界与非目标

  • Skill 系统当前的主要生效方式仍然是 prompt 注入与资源组织，而不是一个统一的运行时插件总线。
  • 安装与编辑路径受安全扫描和存储后端约束，所以不是所有任意文件都能被当成 Skill 支持文件直接执行。
  • Skill 不能替代 Tool、Sub-Agent 或 Middleware。它可以影响 agent 的策略，但不会天然创建新的 runtime orchestration 机制。
  • 本章不教你从零写一个自定义 Skill 包。那是 30-custom-skills-guide.html 的职责；本章的重点是 DeerFlow 自己怎样发现、管理和注入 skills。

它与 DeerFlow 其他能力的关系

Skill 系统在 DeerFlow 里处在一个“承上启下”的位置：

  • 它和第 2 章的模型与工具装配相连，因为 Skill 经常决定 agent 该如何理解工具面，而 backend/packages/harness/deerflow/tools/builtins/tool_search.py 又让这种理解能转成按需工具发现。
  • 它和第 10 章执行引擎相连，因为 Skill 最终是通过 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 进入 lead agent，再在真实运行链路中影响决策。
  • 它和第 20 章上下文工程相连，因为 Skill 本质上也是一种结构化上下文注入机制，只不过来源是可管理的能力包，而不是一次性 prompt 拼接。
  • 它和第 30 章自定义 Skill 指南互补：本章解释 DeerFlow skills 系统怎么工作，第 30 章再教用户怎样编写、打包和维护属于自己的 skills。
如果你要追踪一条最短理解路径，建议把 backend/app/gateway/routers/skills.py、backend/packages/harness/deerflow/skills/installer.py、backend/packages/harness/deerflow/skills/、backend/packages/harness/deerflow/agents/lead_agent/prompt.py 和 backend/packages/harness/deerflow/tools/builtins/tool_search.py 连起来读。这样能同时看到管理面、存储面、prompt 注入面和工具协同面。
