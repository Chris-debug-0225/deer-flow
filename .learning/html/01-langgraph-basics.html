1. 什么是LangGraph？

LangGraph是由LangChain团队开发的一个框架， 用于构建具有明确控制流的Agent应用。 它提供了一种声明式的方式来定义Agent的执行逻辑。

LangGraph的核心目标：
提供一个清晰、可控、可扩展的方式来构建复杂的Agent系统， 特别是那些需要多步骤推理和决策的系统。

LangGraph的主要特点

  • 显式控制流： 你可以清晰地定义Agent在不同状态下的行为
  • 状态管理： 系统自动管理Agent的状态变化
  • 可视化： 可以生成Graph的可视化表示，便于理解和调试
  • 持久化： 支持检查点（Checkpoint），可以保存和恢复执行状态
  • 可扩展性： 易于添加新的节点和边，扩展功能
  • LangSmith集成： 与LangSmith无缝集成，便于监控和调试

LangGraph解决的问题

问题1：控制流不清晰
在没有LangGraph的情况下，Agent的执行逻辑散落在各个函数中， 很难理解整个流程。
问题2：状态管理复杂
需要手动管理Agent的各种状态，容易出错。
问题3：难以调试
当Agent出现问题时，很难追踪执行过程。
问题4：难以扩展
添加新的功能需要修改核心代码。

2. 为什么DeerFlow选择LangGraph？

DeerFlow 2.0是一个完全的重写，选择LangGraph作为核心框架有几个重要原因。

1. 成熟的生态

LangGraph由LangChain团队维护，有完整的文档、活跃的社区和持续的更新。 这确保了框架的稳定性和可靠性。

2. 强大的抽象能力

LangGraph提供了Graph、Node、Edge等抽象， 这些抽象非常适合表示Agent的执行流程。

3. 天然支持多Agent

LangGraph的设计天然支持多Agent系统。 DeerFlow中的Sub-Agents就是基于LangGraph的Graph嵌套实现的。

4. 优秀的可观测性

LangGraph与LangSmith和Langfuse的集成， 使得Agent的执行过程可以被完整地追踪和监控。

5. 灵活的状态管理

LangGraph的状态管理机制非常灵活， 可以轻松处理复杂的状态转移。

总结： LangGraph不仅提供了技术上的优势， 更重要的是它的设计哲学与DeerFlow的目标完全一致： 构建清晰、可控、可扩展的Agent系统。

3. 核心概念：Graph、Node、Edge、State

理解这四个核心概念是掌握LangGraph的基础。

概念1：Graph（图）

Graph是LangGraph的最基本单位， 它代表了Agent的整个执行流程。

Graph = 节点 + 边 + 状态
一个Graph由多个Node组成，Node之间通过Edge连接， 整个Graph共享一个State。

Graph的特点：

  • 有明确的入口点（START）和出口点（END）
  • 可以有条件分支（conditional edges）
  • 可以有循环（cycles）
  • 可以嵌套（一个Graph可以包含另一个Graph）

概念2：Node（节点）

Node代表执行流程中的一个步骤。 每个Node是一个可调用的函数或对象。

Node的例子：
- 调用LLM的Node
- 执行工具的Node
- 处理结果的Node
- 做出决策的Node

Node的特点：

  • 接收当前的State作为输入
  • 执行某些操作
  • 返回更新后的State
  • 可以是同步的或异步的

概念3：Edge（边）

Edge定义了Node之间的连接关系。 它指定了从一个Node到另一个Node的转移。

Edge的类型：

类型 说明 例子
普通Edge 无条件转移，总是执行 从Node A到Node B
条件Edge 根据条件决定转移 如果结果为"continue"，转到Node B；否则转到Node C
循环Edge 可以返回到之前的Node 如果需要重试，返回到"调用LLM"的Node

概念4：State（状态）

State是Graph中所有Node共享的数据结构。 它记录了Agent执行过程中的所有信息。

State的作用：
1. 存储Agent的输入和输出
2. 记录中间结果
3. 追踪Agent的执行历史
4. 在Node之间传递数据

State的特点：

  • 通常是一个字典或TypedDict
  • 所有Node都可以读写State
  • State的变化被自动追踪
  • 可以被持久化（保存和恢复）
State的例子：
state_example.py :: 一个最基础的 AgentState 长什么样
1{
2 "messages": [...], # 对话历史，通常是整条消息序列
3 "current_step": "thinking", # 当前执行阶段，用来驱动路由
4 "tools_used": [...], # 已调用过哪些工具
5 "intermediate_results": {...}, # 工具返回、中间草稿、路由标记等
6 "final_output": None, # 最终输出通常在最后节点填充
7}

5. LangGraph vs LangChain

LangGraph和LangChain都是LangChain生态的一部分， 但它们的用途不同。

方面 LangChain LangGraph
主要用途 构建LLM应用的通用框架 构建具有明确控制流的Agent
抽象层级 高层抽象（Chain、Runnable等） 低层抽象（Graph、Node、Edge）
控制流 隐式的，通过Chain的组合 显式的，通过Graph定义
适用场景 简单的顺序流程 复杂的多步骤推理
可视化 有限的可视化支持 完整的Graph可视化
状态管理 隐式的 显式的

何时使用LangChain，何时使用LangGraph

使用LangChain的场景：
✓ 简单的顺序流程
✓ 快速原型开发
✓ 不需要复杂的控制流
✓ 需要使用LangChain的其他组件（如RAG）
使用LangGraph的场景：
✓ 复杂的多步骤推理
✓ 需要明确的控制流
✓ 需要循环和条件分支
✓ 需要完整的可观测性
✓ 构建生产级应用
重要： LangGraph和LangChain不是竞争关系， 而是互补关系。LangGraph通常会使用LangChain的组件（如Runnable）。

6. 简单示例：Hello World Graph

让我们看一个最简单的LangGraph应用。

hello_graph.py :: 最小 LangGraph 链路
1from typing import TypedDict
2from langgraph.graph import StateGraph, START, END
3
4class SimpleState(TypedDict):
5 message: str
6 trace: list[str]
7
8def node_a(state: SimpleState) -> SimpleState:
9 state["trace"].append("enter:node_a")
10 state["message"] += " -> A"
11 state["trace"].append("leave:node_a")
12 return state
13
14def node_b(state: SimpleState) -> SimpleState:
15 state["trace"].append("enter:node_b")
16 state["message"] += " -> B"
17 state["trace"].append("leave:node_b")
18 return state
19
20graph = StateGraph(SimpleState)
21graph.add_node("node_a", node_a)
22graph.add_node("node_b", node_b)
23graph.add_edge(START, "node_a")
24graph.add_edge("node_a", "node_b")
25graph.add_edge("node_b", END)
26
27app = graph.compile()
28result = app.invoke({"message": "Hello", "trace": []})
29print(result["message"]) # Hello -> A -> B
30print(result["trace"]) # ['enter:node_a', 'leave:node_a', 'enter:node_b', 'leave:node_b']

代码解析

  • 这个示例虽然简单，但已经包含 LangGraph 最重要的执行事实：State 不是“参数传进去一次”，而是在节点之间持续被修改、传递、带到最后。
  • 我这里特意加了 trace 字段，是为了让你看到 Node 不只是改业务结果，也常常顺手写入调试轨迹、执行元数据、日志线索。
  • 第 20 到 25 行的图结构完全是声明式的。先把节点注册进去，再把边连起来，最后统一 compile，这是 LangGraph 和“手写 if/else 流程函数”的关键差别。
  • 如果你已经能看懂这一段，后面 DeerFlow 那些更复杂的源码，其实只是把 message 扩展成了更丰富的 ThreadState，把 node_a/node_b 换成模型、中间件、工具、记忆、子代理节点。

7. 复杂示例：Agent Graph

现在让我们看一个更复杂的例子， 展示一个简单的Agent如何使用LangGraph实现。

agent_graph.py :: LLM -> route -> tool -> LLM -> finalize
1from typing import TypedDict, List, Any
2from langgraph.graph import StateGraph, START, END
3from langchain_openai import ChatOpenAI
4
5class AgentState(TypedDict):
6 messages: List[dict[str, Any]]

11def call_llm(state: AgentState) -> AgentState:
12 llm = ChatOpenAI(model="gpt-4o-mini")
13 response = llm.invoke(state["messages"])
14 content = response.content
15 state["messages"].append({"role": "assistant", "content": content})
16 state["current_step"] = "check_response"
17 return state
18

23 else:
24 state["current_step"] = "finalize"
25 return state
26

28 query = state["messages"][0]["content"]
29 state["messages"].append({
30 "role": "tool",
31 "content": f"搜索结果: 关于 {query} 的资料已经找到",
32 })
33 state["search_rounds"] += 1
34 state["current_step"] = "call_llm"
35 return state
36
37def finalize(state: AgentState) -> AgentState:
38 state["final_answer"] = state["messages"][-1]["content"]
39 return state
40
41def route(state: AgentState) -> str:
42 if state["current_step"] == "search":
43 return "search"
44 return "finalize"
45

58initial_state = {

这个例子展示了什么

  • 这段代码比 Hello World 更接近真正的 Agent：先调用模型，再检查响应，再决定要不要进工具节点，工具跑完还能回到模型节点继续推理。
  • 第 19 到 25 行把“判断是否调用工具”单独拆成 check_response 节点，这是生产系统里很常见的做法。因为路由逻辑一旦复杂，就值得单独成为一个节点，而不是塞进模型节点内部。
  • 第 27 到 35 行的 search 节点展示了循环图的本质：工具结果会被重新写回 messages，然后图沿着 search -> call_llm 这条边回到推理节点继续下一轮。
  • 第 41 到 55 行把线性流程升级成“带条件和回环的状态机”。这也是 DeerFlow 选择 LangGraph 的核心原因之一，因为很多真实 Agent 根本不是一条直线跑到底。

  • 所以对 DeerFlow 来说，LangGraph 不是“可选框架”，而是 Agent 执行骨架本身。后面讲的 memory、sandbox、subagent，本质上都只是往这个骨架里继续挂节点行为和状态更新规则。

核心源码：Graph 建好以后，Gateway worker 又如何真正执行这张图

节选自 backend/packages/harness/deerflow/runtime/runs/worker.py。这一段补上了 LangGraph 基础课最关键的最后一环：图不是 compile 完就结束，它还要被运行时包装、注入上下文、流式执行。

  • 这一段把 LangGraph 基础真正往前推进了一步：图对象不是 compile 完就自己运行，而是被 Gateway worker 包起来，补上 run 状态、事件桥、runtime 上下文、callbacks 和 tracing 再执行。

  • 也就是说，DeerFlow 真正的生产图执行，不只是“节点和边”，还包括运行时容器、可观测性和状态恢复。这也是教学示例和实际系统的最大差别。
