DeerFlow 学习系列 - 第 40 章

本章讲的是读 DeerFlow 源码时应该先抓哪些目录、再抓哪些链路

DeerFlow 不是一个靠单文件就能看懂的项目。最有效的阅读方式，是先把 Gateway、agents、runtime、sandbox、skills、channels 这几个核心目录放到同一张地图里，再按“外部入口 -> 运行主线 -> 支撑基础设施 -> 扩展点”逐层下钻。

这项能力在 DeerFlow 中解决什么问题

如果你第一次打开 DeerFlow 仓库就从某个深层模块开始，很容易陷进局部细节。源码导航这一章要解决的问题是：帮助你建立最短阅读路线，让你先看到 DeerFlow 的真实主线，再回头理解每个扩展点。

这项能力在代码库中的位置

路径 职责 阅读建议
backend/app/gateway/ 外部 API、auth、routers、service 生命周期 先看外部入口和默认运行时承载面
backend/packages/harness/deerflow/agents/ lead agent、factory、middlewares、memory、thread state 再看真正的 agent 构建与执行主线
backend/packages/harness/deerflow/runtime/ run worker、stream bridge、serialization、records 理解运行期状态和流式协议
backend/packages/harness/deerflow/sandbox/ 虚拟路径、provider、文件工具、边界控制 理解文件、命令与 artifact 是怎么落地的
backend/packages/harness/deerflow/skills/ skills 类型、安装、存储、安全扫描 看能力包如何进入系统
backend/app/channels/ IM adapter、message bus、channel manager 最后看外部渠道如何复用 Gateway runtime

真实运行链路或数据链路

recommended reading path -> backend/docs/ARCHITECTURE.md -> backend/app/gateway/ -> backend/packages/harness/deerflow/agents/ -> backend/packages/harness/deerflow/runtime/ -> backend/packages/harness/deerflow/sandbox/ -> backend/packages/harness/deerflow/skills/ -> backend/app/channels/

如果你的目标是排查“用户发一句话以后系统怎么走”，就优先看 Gateway -> agents -> runtime；如果目标是排查“文件、上传、artifact 为什么表现成这样”，就从 sandbox 和 uploads/artifacts routers 下钻；如果目标是扩展能力，则优先看 skills、custom agents 和 channels。

配置面与使用方式

  • 阅读源码前，最好先配合 backend/docs/ARCHITECTURE.md 和 backend/docs/API.md 建立目录地图。
  • 读 backend/app/gateway/ 时，要同步关注 auth、uploads、artifacts、thread_runs 这些 routers 的关系，而不是单看一个 endpoint。
  • 读 backend/packages/harness/deerflow/agents/ 时，要把 factory、lead agent、middlewares、memory 放在一起理解。
  • 读 backend/app/channels/ 时，要始终记得它是在复用 Gateway runtime，而不是单独一套 agent 宿主。

关键实现细节与权衡

  • 按目录分层阅读能更快建立全局心智，但会暂时忽略一些跨层 helper；这是值得接受的阅读权衡。
  • 如果你先看 runtime/ 再看 gateway/，会更快理解 streaming 与 run ownership，但可能一开始不理解请求怎么进入系统。
  • 如果你先看 skills/ 或 channels/，容易误把扩展点当成系统主线，所以通常不建议作为第一站。

当前限制、边界与非目标

  • 本章提供的是阅读地图，不是每个目录的完整 API 手册。
  • 源码导航只能帮你决定“先看哪里”，真正理解仍要回到前面章节对应的实现解释。
  • 随着 DeerFlow 演进，某些目录职责会扩展，但当前主线仍然围绕这里列出的几个核心区域。
