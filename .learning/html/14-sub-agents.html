Sub-Agents：多Agent协作

DeerFlow 学习系列 - 第 14 章

本章讲的是 DeerFlow 如何做结构化委派，而不是群聊式多智能体

DeerFlow 的 subagent 不是共享黑板式群体智能系统，也不是多个 agent 同时自由讨论。它的核心机制是：主 agent 通过 task tool 发起结构化委派，由独立的 SubagentExecutor 在隔离上下文中执行，再通过轮询、状态管理、超时和清理机制把结果带回父运行时。

这项能力在 DeerFlow 中解决什么问题

主 agent 在长任务中经常会遇到一个问题：某个子任务既复杂又啰嗦，如果继续在同一个上下文里展开，会污染当前推理轨迹、推高 token 成本，还会让后续决策更混乱。DeerFlow 的 subagent 机制正是为了解决这种“需要委派但又不想共享全部上下文”的场景。

因此，DeerFlow 的 subagent 本质上是结构化委派与隔离执行机制。它不是多个 agent 共用一块黑板持续协商，也不是一套自治社会模拟。父 agent 明确发出任务，subagent 在自己的上下文里完成执行，结果再以状态化方式返回。

一句话边界： DeerFlow 的 subagent 是“通过 task tool 触发的独立执行单元”，而不是“自动自组织的多智能体群体系统”。

真实运行链路或数据链路

DeerFlow 的 subagent 委派链很清晰，入口固定是 backend/packages/harness/deerflow/tools/builtins/task_tool.py：

parent agent decides to delegate -> task tool call -> backend/packages/harness/deerflow/tools/builtins/task_tool.py -> registry/config resolve subagent type -> create SubagentExecutor(...) -> execute_async(prompt, task_id=tool_call_id) -> backend/packages/harness/deerflow/subagents/executor.py -> background execution + polling -> terminal status / usage summary / cleanup -> result returned to parent agent

这条链有几个非常 DeerFlow-specific 的细节。第一，父运行时会把 sandbox_state、thread_data、thread_id、trace_id、父模型名和可用 skill/tool 限制传给 subagent。第二，task_tool.py 会持续轮询后台结果并向流式 writer 发出 task_started、task_running、task_completed 等事件。第三，token usage 会在 subagent 侧收集，再回写到父运行时的 usage 记录与 cache 中。

这里不是消息总线群聊： subagent 并不会和主 agent 长时间共享同一消息列表。它是在 backend/packages/harness/deerflow/subagents/executor.py 中构建自己的初始 state、技能消息和工具集合，然后独立执行。

配置面与使用方式

subagent 的配置面主要集中在类型解析和执行参数上：

  • backend/packages/harness/deerflow/subagents/registry.py 负责把名称映射到内置或自定义配置。
  • backend/packages/harness/deerflow/config/subagents_config.py 允许在配置层定义 custom agents，并覆盖 system prompt、tools、skills、timeout、model 等参数。
  • backend/packages/harness/deerflow/subagents/config.py 中的 SubagentConfig 与 resolve_subagent_model_name 决定 subagent 是否继承父模型、最多运行多少轮以及允许哪些工具。
  • backend/packages/harness/deerflow/tools/builtins/task_tool.py 会继承父级的 tool groups 与部分 skill allowlist，并显式关闭 subagent 再次拥有 subagent tools，避免递归嵌套失控。

换句话说，DeerFlow 并不是让任何 agent 随便“spawn 另一个 agent”。委派能力受注册表、配置、父级策略和执行器规则共同限制。

关键实现细节与权衡

  • backend/packages/harness/deerflow/tools/builtins/task_tool.py 采用后台执行加轮询，而不是让模型自己反复调用“查询任务状态”工具，这能减少 prompt 污染和控制复杂度。
  • backend/packages/harness/deerflow/subagents/executor.py 维护独立的状态机，包含 PENDING、RUNNING、COMPLETED、FAILED、CANCELLED、TIMED_OUT，让清理与恢复逻辑有明确落点。
  • 超时不是单点处理。既有执行超时，也有父侧轮询超时；如果后台线程卡住，还会通过 cooperative cancellation 与 deferred cleanup 收尾。
  • token usage attribution 是这套设计很重要的细节：subagent 的模型调用成本不会消失，而是会被收集、汇总并归因回父级运行。
  • subagent 自己也会重新装配 skills、tools 和 deferred tool search，这使它是独立执行单元，而不是父 agent 的一个“函数调用回调”。
核心取舍： DeerFlow 用显式委派、后台状态机、轮询与清理机制换来了上下文隔离和更可控的长任务执行，但它并不追求“无限递归、自发协商”的多智能体幻想。

当前限制、边界与非目标

  • subagent 不是共享黑板系统，也不是多个 agent 同时读写一个公共工作记忆的群体智能框架。
  • DeerFlow 明确禁止 subagent 再拥有 subagent tools，避免递归委派无界增长。
  • 超时、polling 和 cleanup 机制说明后台执行并非“发出去就不管”，也说明这套能力仍依赖当前进程内任务管理模型，而不是分布式任务队列。
  • 虽然 subagent 可以继承父级 sandbox 与 thread data，但 ACP agent 的工作空间又是独立的，因此不能把所有“子执行单元”都等同为同一种路径语义。

它与 DeerFlow 其他能力的关系

subagent 机制处在 DeerFlow 多个能力的交汇点：

  • 它和第 10 章执行引擎相连，因为 subagent 自己也会重新构建 agent、middlewares、state 和 tool 面，只是运行在独立执行器中。
  • 它和第 11 章 skills 系统相连，因为 backend/packages/harness/deerflow/subagents/executor.py 会按配置重新加载 skill 内容，并在子上下文里注入。
  • 它和第 12 章 sandbox 相连，因为 subagent 会继承父级的 sandbox_state、thread_data 与 thread_id，从而共享 DeerFlow 的虚拟路径与工具边界。
  • 它和 token usage、streaming、Gateway 运行所有权相连，因为 subagent 的状态变化、结果和 usage 都要回流到父任务的事件流与统计体系中。
建议把 backend/packages/harness/deerflow/tools/builtins/task_tool.py、backend/packages/harness/deerflow/subagents/executor.py、backend/packages/harness/deerflow/subagents/registry.py、backend/packages/harness/deerflow/config/subagents_config.py 和 backend/packages/harness/deerflow/subagents/config.py 一起读。这样最容易同时看到委派入口、执行器、注册表、配置模型和 timeout / cleanup / usage attribution 的完整闭环。
