Gateway API：与外界的接口

DeerFlow 学习系列 - 第 15 章

本章讲的不是“薄代理层”

DeerFlow 当前默认运行时并不是独立 LangGraph Server 再外面套一层轻 API。真正默认入口是 FastAPI Gateway：它持有路由、认证、线程语义、run lifecycle、SSE streaming、uploads、artifacts，以及面向前端和 IM 渠道的兼容接口。理解这一章的关键，是把 Gateway 看成 DeerFlow 的运行时承载体，而不是转发器。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 需要一个统一入口，把浏览器前端、LangGraph SDK 兼容客户端、IM channel worker、自定义脚本和本地文件生命周期都收进同一套运行时语义里。Gateway 正是用来解决这个问题的：它不仅暴露 REST API，还真正拥有线程、run、stream、上传、工件、认证和清理行为。

这也是为什么本章主题不是“怎么设计 API 风格”，而是“为什么 DeerFlow 把默认 runtime 放在 Gateway 里”。如果把 Gateway 误解成薄代理层，就会错过 DeerFlow 当前仓库最重要的架构事实。

必须明确的一点： /api/langgraph/* 是 nginx 重写后的 Gateway 兼容路径，不是 DeerFlow 核心架构里另起的一套运行时。根据 backend/docs/API.md 与 backend/docs/ARCHITECTURE.md，公开的 LangGraph-compatible API 仍然由 Gateway 自己承接。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/app/gateway/app.py Gateway 应用入口、middleware、lifespan、runtime 初始化 这是默认运行时真正启动的位置
backend/app/gateway/routers/thread_runs.py 线程级 runs、stream、wait、cancel、join 说明 run lifecycle 不在“外部 LangGraph 服务”里，而在 Gateway 自己手上
backend/app/gateway/routers/threads.py thread 创建、状态、history、删除与本地清理 说明 thread 语义同时覆盖 checkpointer 与 DeerFlow 本地目录
backend/app/gateway/routers/uploads.py 上传、列表、限制、虚拟路径与 sandbox 同步 Gateway 负责把文件接入 thread runtime，而不是让前端直接碰磁盘
backend/app/gateway/routers/artifacts.py 工件读取、虚拟路径解析、活跃内容下载保护 说明 artifacts 也是运行时的一部分，而不只是静态文件服务
backend/docs/API.md 公开 API 契约与兼容路径说明 这里直接写明了 `/api/langgraph/*` 如何映射到 Gateway
backend/docs/ARCHITECTURE.md 整体部署图与 Gateway embedded runtime 描述 帮助把路由文件放回真实架构图中理解

真实运行链路或数据链路

一次典型的 DeerFlow Web 运行链路可以压缩成下面这条主线：

nginx /api/langgraph/threads/{thread_id}/runs/stream -> rewrite to Gateway-owned /api/* compatibility path -> backend/app/gateway/app.py -> backend/app/gateway/routers/thread_runs.py -> app.gateway.services.start_run(...) -> deerflow runtime worker / stream bridge -> lead agent runtime -> SSE back through Gateway response

删除线程时，链路又会回到 backend/app/gateway/routers/threads.py：它不仅会删 checkpoint / thread meta，还会调用 DeerFlow 的路径管理删除线程目录。上传文件时，请求进入 backend/app/gateway/routers/uploads.py，生成 thread uploads、虚拟路径和 artifact URL；读取工件时，再由 backend/app/gateway/routers/artifacts.py 按 thread 虚拟路径解析与返回。

关键理解： Gateway 不是“请求进来转发给别的 DeerFlow 核心”。它自己就拥有 thread、run、upload、artifact 这些核心 runtime surface。

配置面与使用方式

对外使用时，调用方主要看到两组路径：

  • /api/langgraph/* 提供 LangGraph-compatible 线程与运行接口，方便前端和 SDK 沿用 LangGraph wire contract。
  • /api/* 提供 DeerFlow 自己的 models、skills、uploads、artifacts、channels 等 REST surface。

但从部署面看，这两组能力都在 backend/app/gateway/app.py 里被统一挂载。根据 backend/docs/API.md，nginx 是统一反向代理入口；根据 backend/docs/ARCHITECTURE.md，默认部署并不会额外跑一套独立 LangGraph runtime 来承接浏览器流量。

Gateway 还直接暴露了和浏览器安全相关的运行面：backend/app/gateway/app.py 会挂上 CORS、CSRFMiddleware 和 AuthMiddleware。这意味着 thread/run API 不只是“能调用”，还必须通过 DeerFlow 当前定义的 CSRF 和登录态保护。

实际使用时，前端会先用 thread API 创建会话，再用 run/stream API 发起执行；文件相关能力则通过 backend/app/gateway/routers/uploads.py 和 backend/app/gateway/routers/artifacts.py 进入同一个 thread 语义空间。IM channel 则通过内部认证 token 和匹配 CSRF 头进入同一 Gateway surface，而不是直接调用 runtime 私有函数。

关键实现细节与权衡

  • backend/app/gateway/app.py 在 lifespan 里初始化 LangGraph runtime、认证升级迁移和 channel service，说明 Gateway 拿着应用启动期的关键所有权。
  • backend/app/gateway/routers/thread_runs.py 直接实现 create / stream / wait / cancel / join，这比“反代到远端 API”更紧耦合，但也让 run manager、stream bridge 与鉴权能保持一致。
  • backend/app/gateway/routers/threads.py 把 LangGraph state 和 DeerFlow 本地 thread 目录一起管理，代价是 thread 删除不只是删一行元数据，而是多层清理。
  • backend/app/gateway/routers/uploads.py 与 backend/app/gateway/routers/artifacts.py 都坚持 thread 虚拟路径语义，这让 sandbox、前端下载和用户隔离可以共用同一套路径模型。
  • backend/docs/ARCHITECTURE.md 明确把默认 Gateway worker 约束在单 worker 语义上，原因不是部署偷懒，而是 run ownership、进程内 stream bridge 与部分 channel/service 生命周期目前都绑定在当前进程。
  • backend/packages/harness/deerflow/client.py 与 Gateway 共享同一 agent runtime 和模型/技能装配逻辑，但它故意保留同步、进程内的调用体验；这正是为什么 DeerFlow 既有 Gateway 路径，也有 embedded client 路径。
  • backend/docs/API.md 选择对外维持 LangGraph-compatible 路径，换来生态兼容性；代价是读代码时必须知道“公开路径名”和“内部原生 router 前缀”不是一回事。
核心取舍： DeerFlow 把默认运行时嵌进 Gateway，牺牲了一点“组件边界看起来更纯”的抽象感，换来了认证、线程、文件、streaming 和渠道接入的一致性。

当前限制、边界与非目标

  • 当前默认架构不是“Gateway 前面随便横向扩，多 worker 自动安全”。run ownership、stream bridge 和部分 channel 语义仍然依赖当前进程内状态。
  • /api/langgraph/* 是兼容 surface，不意味着 DeerFlow 内部所有实现都按 LangGraph Server 原生模块组织。
  • Gateway 受 CORS、CSRF、auth 和 internal channel 调用机制共同约束，因此它也不是一个“任何内部流量都可无保护绕过”的裸 API 面。
  • Gateway 负责 artifacts 和 uploads 的 HTTP 面，但它不是通用对象存储服务，也不是任意静态站点服务器。
  • 本章不覆盖所有 router；重点是默认 runtime 为什么落在 Gateway，以及 thread/run/uploads/artifacts 这几块如何体现这个事实。

它与 DeerFlow 其他能力的关系

Gateway 是 DeerFlow 多个能力的汇合点：

  • 它和第 10 章执行引擎相连，因为 run/stream 请求最终会驱动 lead agent runtime，而不是停留在 API 层。
  • 它和第 12、13 章的 sandbox、memory、files 相连，因为 thread 目录、uploads、artifacts 与用户隔离都通过 Gateway surface 暴露给外部。
  • 它和第 21 章模型集成相连，因为前端选择模型与 thinking 开关，最终是经由 Gateway request context 进入运行时。
  • 它和第 22 章 channels 集成相连，因为 IM channel manager 也是复用 Gateway runtime、auth 和 thread API，而不是绕过它另建执行通道。
建议把 backend/app/gateway/app.py、backend/app/gateway/routers/thread_runs.py、backend/app/gateway/routers/threads.py、backend/app/gateway/routers/uploads.py、backend/app/gateway/routers/artifacts.py、backend/docs/API.md 和 backend/docs/ARCHITECTURE.md 连起来读。这样最容易看清 DeerFlow 默认 runtime 的真实承载面。
