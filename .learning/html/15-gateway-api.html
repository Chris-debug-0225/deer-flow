Gateway API：与外界的接口

DeerFlow 学习系列 - 第 15 章

本章讲的不是“薄代理层”

DeerFlow 当前默认运行时并不是独立 LangGraph Server 再外面套一层轻 API。真正默认入口是 FastAPI Gateway：它持有路由、认证、线程语义、run lifecycle、SSE streaming、uploads、artifacts，以及面向前端和 IM 渠道的兼容接口。理解这一章的关键，是把 Gateway 看成 DeerFlow 的运行时承载体，而不是转发器。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 需要一个统一入口，把浏览器前端、LangGraph SDK 兼容客户端、IM channel worker、自定义脚本和本地文件生命周期都收进同一套运行时语义里。Gateway 正是用来解决这个问题的：它不仅暴露 REST API，还真正拥有线程、run、stream、上传、工件、认证和清理行为。

这也是为什么本章主题不是“怎么设计 API 风格”，而是“为什么 DeerFlow 把默认 runtime 放在 Gateway 里”。如果把 Gateway 误解成薄代理层，就会错过 DeerFlow 当前仓库最重要的架构事实。

必须明确的一点： /api/langgraph/* 是 nginx 重写后的 Gateway 兼容路径，不是 DeerFlow 核心架构里另起的一套运行时。根据 backend/docs/API.md 与 backend/docs/ARCHITECTURE.md，公开的 LangGraph-compatible API 仍然由 Gateway 自己承接。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/app/gateway/app.py Gateway 应用入口、middleware、lifespan、runtime 初始化 这是默认运行时真正启动的位置
backend/app/gateway/routers/thread_runs.py 线程级 runs、stream、wait、cancel、join 说明 run lifecycle 不在“外部 LangGraph 服务”里，而在 Gateway 自己手上
backend/app/gateway/routers/threads.py thread 创建、状态、history、删除与本地清理 说明 thread 语义同时覆盖 checkpointer 与 DeerFlow 本地目录
backend/app/gateway/routers/uploads.py 上传、列表、限制、虚拟路径与 sandbox 同步 Gateway 负责把文件接入 thread runtime，而不是让前端直接碰磁盘
backend/app/gateway/routers/artifacts.py 工件读取、虚拟路径解析、活跃内容下载保护 说明 artifacts 也是运行时的一部分，而不只是静态文件服务
backend/docs/API.md 公开 API 契约与兼容路径说明 这里直接写明了 `/api/langgraph/*` 如何映射到 Gateway
backend/docs/ARCHITECTURE.md 整体部署图与 Gateway embedded runtime 描述 帮助把路由文件放回真实架构图中理解

从源码阅读角度看，Gateway 最关键的不是 router 数量，而是 backend/app/gateway/app.py 怎样把它们装到同一个应用里。这个文件在 lifespan 阶段会加载配置、初始化 runtime、处理 admin/bootstrap 相关逻辑、启动 channel service；随后再通过一串 app.include_router(...) 把 models、memory、skills、uploads、threads、thread_runs、channels 等 surface 全挂在同一个 FastAPI 应用上。也正因为如此，Gateway 是 DeerFlow 的运行承载体，不是一个单独的接口层皮肤。

真实运行链路或数据链路

一次典型的 DeerFlow Web 运行链路可以压缩成下面这条主线：

gateway flow :: 一次 SSE run 请求怎样穿过 Gateway
1"nginx /api/langgraph/threads/{thread_id}/runs/stream"
2 -> "rewrite to Gateway-owned /api/* compatibility path"
3 -> "backend/app/gateway/app.py"
4 -> "backend/app/gateway/routers/thread_runs.py"
5 -> "app.gateway.services.start_run(...)"
6 -> "deerflow runtime worker / stream bridge"
7 -> "lead agent runtime"
8 -> "SSE back through Gateway response"

翻到实现层，会看到 thread_runs.py 的职责非常明确：它接收 body，交给 start_run(...) 建立 RunRecord，然后根据是 /runs、/runs/stream 还是 /runs/wait 选择立即返回、SSE StreamingResponse 或阻塞等待 checkpoint 最终状态。真正的 run 任务并不是在 router 里同步跑完，而是由 Gateway service 和 runtime worker 接手。这一点很重要，因为它解释了为什么取消、join、SSE、反馈统计都还能围绕同一条 run 继续工作。

删除线程时，链路又会回到 backend/app/gateway/routers/threads.py：它不仅会删 checkpoint / thread meta，还会调用 DeerFlow 的路径管理删除线程目录。上传文件时，请求进入 backend/app/gateway/routers/uploads.py，生成 thread uploads、虚拟路径和 artifact URL；读取工件时，再由 backend/app/gateway/routers/artifacts.py 按 thread 虚拟路径解析与返回。

核心源码 1：Gateway 路由是怎样被统一挂进应用的

节选自 backend/app/gateway/app.py。这段代码直接回答“为什么 Gateway 不是薄代理层”。

  • 这里把 models、memory、skills、uploads、threads、channels、runs 全都挂在同一个 FastAPI app 上，说明这些不是外围插件，而是 Gateway 原生承载的运行面。
  • thread_runs.router 和 runs.router 同时存在，意味着 DeerFlow 既支持线程态运行，也支持某些无预建线程的运行入口。
  • 课程里常说“Gateway owns the runtime”，看这段挂载代码就很好理解，因为整个外部 surface 都集中在这里。

核心源码 2：runs/stream endpoint 为什么一开始就能返回 SSE

节选自 backend/app/gateway/routers/thread_runs.py。这是外部请求进入执行主线的第一站。

gateway/routers/thread_runs.py :: stream_run(...)
1@router.post("/{thread_id}/runs/stream")
2@require_permission("runs", "create", owner_check=True, require_existing=True)
3async def stream_run(thread_id: str, body: RunCreateRequest, request: Request) -> StreamingResponse:
4 # 先从 request 依赖里拿到全局 stream bridge 和 run manager
5 bridge = get_stream_bridge(request)
6 run_mgr = get_run_manager(request)
7
8 # 真正创建 run record，并把后台执行任务启动起来
9 record = await start_run(body, thread_id, request)
10
11 # 这里不等待 run 完成，而是立刻把 SSE 消费器返回给浏览器
12 return StreamingResponse(
13 sse_consumer(bridge, record, request, run_mgr),
14 media_type="text/event-stream",
15 headers={
16 "Cache-Control": "no-cache",
17 "Connection": "keep-alive",
18 "X-Accel-Buffering": "no",
19 "Content-Location": f"/api/threads/{thread_id}/runs/{record.run_id}",
20 },
21 )
  • start_run(...) 先把后台任务建起来，router 自己并不阻塞等待 agent 跑完。
  • StreamingResponse(...) 立刻把 SSE 管道返回给前端，所以浏览器可以边跑边收事件。
  • Content-Location 把这次 run 的资源 URL 明确告诉前端，方便 SDK 后续继续追踪同一次运行。

核心源码 3：上传文件为什么会立刻进入 thread 语义，而不是裸落盘

节选自 backend/app/gateway/routers/uploads.py。这段代码能把 Gateway 和 sandbox / artifact 之间的连接看清楚。

gateway/routers/uploads.py :: upload_files(...)
1@router.post("", response_model=UploadResponse)
2@require_permission("threads", "write", owner_check=True, require_existing=False)
3async def upload_files(thread_id: str, request: Request, files: list[UploadFile] = File(...), config: AppConfig = Depends(get_config)) -> UploadResponse:
4 user_id = get_effective_user_id()
5 sandbox_provider = get_sandbox_provider(config)
6 paths = get_paths()
7 limits = _get_upload_limits(config)
8
9 # 这里不是写到某个全局 tmp 目录，而是按 thread / user 计算真正的 uploads 目录
10 uploads_dir = ensure_uploads_dir(get_uploads_dir(paths, thread_id, user_id))
11
12 for file in files:
13 display_filename = normalize_filename(file.filename or "upload")
14 file_path, file_size, total_size = await _write_upload_file_with_limits(file, uploads_dir=uploads_dir, display_filename=display_filename, max_single_file_size=limits.max_file_size, max_total_size=limits.max_total_size, total_size=total_size)
15
16 # 写完以后还要补 sandbox 可读/可写权限，说明上传不是只服务 Gateway 自己
17 _make_file_sandbox_readable(file_path)
18 if _uses_thread_data_mounts(sandbox_provider):
19 _make_file_sandbox_writable(file_path)
20
21 # 返回给前端的不是裸磁盘路径，而是 thread 语义下的 virtual_path + artifact_url
22 virtual_path = upload_virtual_path(thread_id, display_filename)
23 artifact_url = upload_artifact_url(thread_id, display_filename)
  • 上传入口虽然是 HTTP router，但一进来就先拿 user_id、sandbox_provider、paths 和上传限制，说明它服务的是整个运行时，不是单纯文件服务。
  • get_uploads_dir(...) 和 ensure_uploads_dir(...) 让文件直接落到 thread / user 绑定目录下，这就是 DeerFlow 后面能把上传文件和某个会话严格对应起来的原因。
  • 真正写文件的过程走的是 _write_upload_file_with_limits(...)，里面会边写边累计单文件大小和总大小，超限立刻失败，不会先全写完再检查。
  • 写完以后还要补 sandbox 可读可写权限，说明上传文件后续不是只给 Gateway 自己看，而是要进入 sandbox / agent 的执行链路。
  • 最后返回的是 virtual_path 和 artifact_url，不是“宿主机真实路径”。这正是 Gateway 把本地文件转换成 thread 语义资产的关键一步。

核心源码 4：Gateway service 层才是 router 和 runtime 之间的真正翻译层

节选自 backend/app/gateway/services.py。这段代码能把“为什么 router 很薄，但 Gateway 仍然不只是代理”讲透。

gateway/services.py :: normalize_input() / merge_run_context_overrides() / build_run_config()
1def normalize_input(raw_input: dict[str, Any] | None) -> dict[str, Any]:
2 if raw_input is None:

4 messages = raw_input.get("messages")
5 if messages and isinstance(messages, list):
6 converted: list[Any] = []

8 if isinstance(msg, BaseMessage):
9 converted.append(msg)
10 elif isinstance(msg, dict):
11 try:

13 except (ValueError, TypeError, NotImplementedError) as exc:
14 raise HTTPException(status_code=400, detail=f"Invalid message at input.messages[{index}]: {exc}") from exc
15 return {**raw_input, "messages": converted}
16 return raw_input

  • `normalize_input(...)` 把 HTTP body 里的消息字典转换成真正的 LangChain 消息对象，并且在边界层就把坏输入打成 400，而不是拖到 runtime 里炸 500。
  • `merge_run_context_overrides(...)` 很关键，因为 DeerFlow 现在要同时兼容 `configurable` 和 `context` 两套运行时读法，所以同一份覆盖要落到两个容器里。
  • `build_run_config(...)` 说明 Gateway service 不只是组一个字典，而是在这里决定 thread_id 怎样进入运行配置、custom agent 怎样映射成 `agent_name`、metadata 怎样并入 run。
  • 这正是 Gateway “拥有 runtime 语义”的证据: 它不是转发原始 HTTP body，而是会主动把外部请求翻译成 DeerFlow 内部能跑的那种配置形状。
关键理解： Gateway 不是“请求进来转发给别的 DeerFlow 核心”。它自己就拥有 thread、run、upload、artifact 这些核心 runtime surface。

配置面与使用方式

对外使用时，调用方主要看到两组路径：

  • /api/langgraph/* 提供 LangGraph-compatible 线程与运行接口，方便前端和 SDK 沿用 LangGraph wire contract。
  • /api/* 提供 DeerFlow 自己的 models、skills、uploads、artifacts、channels 等 REST surface。

但从部署面看，这两组能力都在 backend/app/gateway/app.py 里被统一挂载。根据 backend/docs/API.md，nginx 是统一反向代理入口；根据 backend/docs/ARCHITECTURE.md，默认部署并不会额外跑一套独立 LangGraph runtime 来承接浏览器流量。

另外一个很重要的源码事实是：请求期配置不是简单缓存死在应用启动时。Gateway 相关依赖会在请求路径上重新读取可热更新的配置，所以模型列表、memory 行为、summarization 等很多运行字段能在下一条消息时生效；但数据库、sandbox、stream bridge、channels 这类基础设施字段则属于 startup-only 边界。这种冷热分层，也是理解 Gateway 行为为什么“有些改配置后立刻生效，有些必须重启”的关键。

Gateway 还直接暴露了和浏览器安全相关的运行面：backend/app/gateway/app.py 会挂上 CORS、CSRFMiddleware 和 AuthMiddleware。这意味着 thread/run API 不只是“能调用”，还必须通过 DeerFlow 当前定义的 CSRF 和登录态保护。

实际使用时，前端会先用 thread API 创建会话，再用 run/stream API 发起执行；文件相关能力则通过 backend/app/gateway/routers/uploads.py 和 backend/app/gateway/routers/artifacts.py 进入同一个 thread 语义空间。IM channel 则通过内部认证 token 和匹配 CSRF 头进入同一 Gateway surface，而不是直接调用 runtime 私有函数。

关键实现细节与权衡

  • backend/app/gateway/app.py 在 lifespan 里初始化 LangGraph runtime、认证升级迁移和 channel service，说明 Gateway 拿着应用启动期的关键所有权。
  • backend/app/gateway/routers/thread_runs.py 直接实现 create / stream / wait / cancel / join，这比“反代到远端 API”更紧耦合，但也让 run manager、stream bridge 与鉴权能保持一致。
  • backend/app/gateway/routers/threads.py 把 LangGraph state 和 DeerFlow 本地 thread 目录一起管理，代价是 thread 删除不只是删一行元数据，而是多层清理。
  • backend/app/gateway/routers/uploads.py 与 backend/app/gateway/routers/artifacts.py 都坚持 thread 虚拟路径语义，这让 sandbox、前端下载和用户隔离可以共用同一套路径模型。
  • backend/docs/ARCHITECTURE.md 明确把默认 Gateway worker 约束在单 worker 语义上，原因不是部署偷懒，而是 run ownership、进程内 stream bridge 与部分 channel/service 生命周期目前都绑定在当前进程。
  • backend/packages/harness/deerflow/client.py 与 Gateway 共享同一 agent runtime 和模型/技能装配逻辑，但它故意保留同步、进程内的调用体验；这正是为什么 DeerFlow 既有 Gateway 路径，也有 embedded client 路径。
  • backend/docs/API.md 选择对外维持 LangGraph-compatible 路径，换来生态兼容性；代价是读代码时必须知道“公开路径名”和“内部原生 router 前缀”不是一回事。
核心取舍： DeerFlow 把默认运行时嵌进 Gateway，牺牲了一点“组件边界看起来更纯”的抽象感，换来了认证、线程、文件、streaming 和渠道接入的一致性。

当前限制、边界与非目标

  • 当前默认架构不是“Gateway 前面随便横向扩，多 worker 自动安全”。run ownership、stream bridge 和部分 channel 语义仍然依赖当前进程内状态。
  • /api/langgraph/* 是兼容 surface，不意味着 DeerFlow 内部所有实现都按 LangGraph Server 原生模块组织。
  • Gateway 受 CORS、CSRF、auth 和 internal channel 调用机制共同约束，因此它也不是一个“任何内部流量都可无保护绕过”的裸 API 面。
  • Gateway 负责 artifacts 和 uploads 的 HTTP 面，但它不是通用对象存储服务，也不是任意静态站点服务器。
  • 本章不覆盖所有 router；重点是默认 runtime 为什么落在 Gateway，以及 thread/run/uploads/artifacts 这几块如何体现这个事实。

它与 DeerFlow 其他能力的关系

Gateway 是 DeerFlow 多个能力的汇合点：

  • 它和第 10 章执行引擎相连，因为 run/stream 请求最终会驱动 lead agent runtime，而不是停留在 API 层。
  • 它和第 12、13 章的 sandbox、memory、files 相连，因为 thread 目录、uploads、artifacts 与用户隔离都通过 Gateway surface 暴露给外部。
  • 它和第 21 章模型集成相连，因为前端选择模型与 thinking 开关，最终是经由 Gateway request context 进入运行时。
  • 它和第 22 章 channels 集成相连，因为 IM channel manager 也是复用 Gateway runtime、auth 和 thread API，而不是绕过它另建执行通道。
建议把 backend/app/gateway/app.py、backend/app/gateway/routers/thread_runs.py、backend/app/gateway/routers/threads.py、backend/app/gateway/routers/uploads.py、backend/app/gateway/routers/artifacts.py、backend/docs/API.md 和 backend/docs/ARCHITECTURE.md 连起来读。这样最容易看清 DeerFlow 默认 runtime 的真实承载面。
