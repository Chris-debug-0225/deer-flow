上下文工程：提升Agent智能

DeerFlow 学习系列 - 第 20 章

本章讲的不是通用 prompt engineering 技巧

DeerFlow 里的上下文工程是一套运行时联动系统：lead agent prompt、skills 注入、memory 注入、deferred tool search、uploads、images、summarization，以及多个 middleware 共同决定模型在这一轮真正“看到什么、什么时候看到、以什么形式看到”。这一章要追踪的是仓库中的真实上下文组装链，而不是抽象原则。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 要解决的不是“怎么把一段系统提示写得更好看”，而是“在长任务和多能力运行时里，怎样让模型看到刚好够用、不会失控的上下文”。这里的上下文来源很多：基础 prompt、启用中的 skills、长期记忆、上传文件、图片查看结果、summarization 压缩内容，以及中间件动态插入的控制信息。

如果没有这套联动机制，DeerFlow 很快会遇到两类问题：第一，信息太少，模型不知道用户长期偏好、工具策略和线程资产；第二，信息太多，prompt 被历史、文件和工具 schema 淹没。所以 DeerFlow 的上下文工程本质上是 runtime composition，而不是静态模板编写。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/packages/harness/deerflow/agents/lead_agent/prompt.py lead agent system prompt、skills 缓存、subagent 与 deferred tools 提示片段 这是 DeerFlow 静态提示骨架与技能提示片段的主要入口
backend/packages/harness/deerflow/agents/factory.py middleware 装配顺序与功能特性组合 说明上下文不只来自 prompt，还来自 middleware 链
backend/packages/harness/deerflow/agents/middlewares/ dynamic context、memory、summarization、uploads、view-image、todo 等运行时注入点 这是 DeerFlow 把当轮上下文真正注入运行链的核心目录
backend/packages/harness/deerflow/agents/memory/prompt.py memory update prompt、token counting 与记忆文本格式化 长期记忆不是黑盒文件，而是经过专门 prompt 处理的上下文层
backend/packages/harness/deerflow/tools/builtins/tool_search.py deferred tool catalog 与 tool_search 工具 说明 DeerFlow 如何把工具 schema 也纳入“按需暴露”的上下文策略

真实运行链路或数据链路

一轮 DeerFlow 上下文组装，通常会沿着下面这条链发生：

backend/packages/harness/deerflow/agents/lead_agent/prompt.py -> build base system prompt -> inject enabled skills -> inject deferred tool names from backend/packages/harness/deerflow/tools/builtins/tool_search.py -> backend/packages/harness/deerflow/agents/factory.py -> mount middleware chain from backend/packages/harness/deerflow/agents/middlewares/ -> DynamicContextMiddleware injects current date + memory for this turn -> uploads / images / summarization / other middlewares modify runtime state -> final model invocation sees composed context

这里要特别分清两层责任。backend/packages/harness/deerflow/agents/lead_agent/prompt.py 主要提供静态 system prompt 骨架、skills 提示和 deferred tools 片段；而 memory 和当前日期这类“每轮都可能变化”的信息，是通过 backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py 在当轮消息里注入的。backend/packages/harness/deerflow/agents/memory/prompt.py 则负责长期记忆更新所需的提示与格式化，而不是主对话链里唯一的注入点。

上传文件、图片查看、summary 压缩也不是简单地“拼接到 prompt 末尾”，而是借助不同 middleware 在不同阶段注入 thread state 或额外说明。

一个很 DeerFlow 的点： 工具上下文也被分层处理。backend/packages/harness/deerflow/tools/builtins/tool_search.py 让 MCP 工具默认只以名字进入 prompt，真正 schema 要在运行中按需搜索，避免一次把全部工具细节塞进模型上下文。

配置面与使用方式

从使用面看，DeerFlow 的上下文工程至少有这些开关和入口：

  • 启用中的 skills 会被 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 缓存并注入，影响后续新 run 的 system prompt。
  • memory 配置与 backend/packages/harness/deerflow/agents/memory/prompt.py 的 token counting 策略决定记忆注入成本和回退方式。
  • backend/packages/harness/deerflow/agents/factory.py 里的 feature 组合和 middleware 顺序，决定是否挂入 dynamic_context、summarization、memory、uploads、view-image、todo 等上下文相关能力。
  • 上传文件与图片由对应 middleware 转成 thread 级上下文资产，而不是让 prompt 静态引用宿主机路径。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 的 deferred tool 策略，决定模型最初看到的是“工具名列表”还是完整 schema。

关键实现细节与权衡

  • backend/packages/harness/deerflow/agents/lead_agent/prompt.py 使用 enabled skills 缓存与失效刷新，避免每次请求同步扫盘；代价是理解 prompt 来源时要同时考虑缓存生命周期。
  • backend/packages/harness/deerflow/agents/factory.py 把上下文相关能力拆成 middleware 链，而不是一个超级 prompt builder，这样更组合化，但调试时需要跨文件追踪。
  • backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py 把 memory 与当前日期做成 per-turn 注入，而不是写死在静态 prompt 文件里，这更贴近实时状态，但也让“上下文从哪来”必须按静态提示和动态提示分开理解。
  • backend/packages/harness/deerflow/agents/memory/prompt.py 在 tiktoken 不可用时会退回字符估算，说明 DeerFlow 明确把“上下文预算可用性”看得比“绝对精确 token 数”更重要。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 采用 deferred tools，是为了控制上下文体积；代价是模型必须先学会何时搜索工具，而不是天然拿到完整 schema。
  • backend/packages/harness/deerflow/agents/middlewares/ 中多种 middleware 联动，说明 DeerFlow 的上下文工程是运行态注入系统，不是单一 prompt 模板。
核心取舍： DeerFlow 选择“少量静态 prompt + 大量运行时中间件联动”的方式管理上下文，这比纯 prompt 拼接更复杂，但更贴合真实 thread、files、images、memory 和 tools 的生命周期。

当前限制、边界与非目标

  • DeerFlow 的上下文工程不是通用 RAG 框架，也不是把所有外部知识统一索引后再检索注入。
  • skills、memory、summary、uploads、images 都会影响上下文，但它们不是同一种数据层，不能简单合并成一个“万能 context blob”。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 只解决 deferred tool schema 暴露，不负责所有工具策略判断。
  • 本章不展开每个 middleware 逐行源码；重点是理解它们如何共同塑造模型看到的上下文面。

它与 DeerFlow 其他能力的关系

上下文工程几乎连接 DeerFlow 的所有运行时能力：

  • 它和第 11 章 skills 系统相连，因为 skills 进入 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 后，会直接改变系统提示。
  • 它和第 13 章 memory 相连，因为长期记忆既依赖 backend/packages/harness/deerflow/agents/memory/prompt.py 的更新提示，也依赖 dynamic_context 这类 middleware 在当轮注入可见内容。
  • 它和第 12 章 files / sandbox 相连，因为 uploads、artifacts、images 最终都要变成模型可理解的 thread 资产。
  • 它和第 21、23 章模型集成与高级模式相连，因为上下文预算、thinking 模式、tool search 和 subagent 组合都会改变最优上下文策略。
建议把 backend/packages/harness/deerflow/agents/lead_agent/prompt.py、backend/packages/harness/deerflow/agents/factory.py、backend/packages/harness/deerflow/agents/middlewares/、backend/packages/harness/deerflow/agents/memory/prompt.py 和 backend/packages/harness/deerflow/tools/builtins/tool_search.py 连起来读。这样最容易看到 DeerFlow 上下文工程的真实联动机制。
