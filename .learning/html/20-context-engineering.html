上下文工程：提升Agent智能

DeerFlow 学习系列 - 第 20 章

本章讲的不是通用 prompt engineering 技巧

DeerFlow 里的上下文工程是一套运行时联动系统：lead agent prompt、skills 注入、memory 注入、deferred tool search、uploads、images、summarization，以及多个 middleware 共同决定模型在这一轮真正“看到什么、什么时候看到、以什么形式看到”。这一章要追踪的是仓库中的真实上下文组装链，而不是抽象原则。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 要解决的不是“怎么把一段系统提示写得更好看”，而是“在长任务和多能力运行时里，怎样让模型看到刚好够用、不会失控的上下文”。这里的上下文来源很多：基础 prompt、启用中的 skills、长期记忆、上传文件、图片查看结果、summarization 压缩内容，以及中间件动态插入的控制信息。

如果没有这套联动机制，DeerFlow 很快会遇到两类问题：第一，信息太少，模型不知道用户长期偏好、工具策略和线程资产；第二，信息太多，prompt 被历史、文件和工具 schema 淹没。所以 DeerFlow 的上下文工程本质上是 runtime composition，而不是静态模板编写。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/packages/harness/deerflow/agents/lead_agent/prompt.py lead agent system prompt、skills 缓存、subagent 与 deferred tools 提示片段 这是 DeerFlow 静态提示骨架与技能提示片段的主要入口
backend/packages/harness/deerflow/agents/factory.py middleware 装配顺序与功能特性组合 说明上下文不只来自 prompt，还来自 middleware 链
backend/packages/harness/deerflow/agents/middlewares/ dynamic context、memory、summarization、uploads、view-image、todo 等运行时注入点 这是 DeerFlow 把当轮上下文真正注入运行链的核心目录
backend/packages/harness/deerflow/agents/memory/prompt.py memory update prompt、token counting 与记忆文本格式化 长期记忆不是黑盒文件，而是经过专门 prompt 处理的上下文层
backend/packages/harness/deerflow/tools/builtins/tool_search.py deferred tool catalog 与 tool_search 工具 说明 DeerFlow 如何把工具 schema 也纳入“按需暴露”的上下文策略

这里最容易误解的一点，是把“上下文工程”看成只发生在 prompt 文件里。实际上 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 只负责系统提示骨架、skills 段落、subagent 说明和 deferred tool 提示片段；真正随每轮变化的上下文，大量是通过 backend/packages/harness/deerflow/agents/middlewares/ 注入的。换句话说，DeerFlow 的上下文工程同时存在于静态 prompt 代码和运行时中间件代码里。

真实运行链路或数据链路

一轮 DeerFlow 上下文组装，通常会沿着下面这条链发生：

01
先生成基础系统提示

这一层先定义 agent 的角色、固定规则和基础行为，还没放入当轮动态信息。

agents/lead_agent/prompt.py
02
把已启用的技能说明补进去

真正启用中的 Skill 会被渲染成工作方法提示，让模型知道这轮该按什么套路做事。

agents/lead_agent/prompt.py
03
只先告诉模型有哪些延迟工具可查

初始上下文不会塞满所有工具 schema，而是先暴露 deferred tool 名录，细节等需要时再搜索。

tools/builtins/tool_search.py + agents/lead_agent/prompt.py
04
工厂开始挂载中间件链

进入 agent 构建后，DeerFlow 才把上下文相关能力按顺序装进运行链，而不是全部写死在 prompt 中。

agents/factory.py + agents/middlewares/
05
动态提醒注入日期和长期记忆

DynamicContextMiddleware 会在每轮模型调用前补入当前日期、memory 提醒等实时信息。

dynamic_context_middleware.py
06
上传文件、图片和摘要继续改写运行态

文件、图片查看、summary、todo 等能力会继续补充 thread state，让上下文变成一份加工后的运行材料。

agents/middlewares/
07
模型最终看到的是组合完成后的上下文

所以模型真正读到的不是一段孤立 prompt，而是静态规则、技能提示和运行时提醒的合成结果。

final model invocation

真实源码里，这条链的关键节点是 DynamicContextMiddleware。它会在每轮模型调用前，把当前日期、长期记忆等提醒包装成特殊的人类消息插入上下文；这些消息还带有专门标记，后续标题生成和 summarization 中间件会识别并特殊处理。也就是说，DeerFlow 不是简单把一段“今天日期是...”写死进 system prompt，而是把实时上下文做成有生命周期的运行态消息。

这里要特别分清两层责任。backend/packages/harness/deerflow/agents/lead_agent/prompt.py 主要提供静态 system prompt 骨架、skills 提示和 deferred tools 片段；而 memory 和当前日期这类“每轮都可能变化”的信息，是通过 backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py 在当轮消息里注入的。backend/packages/harness/deerflow/agents/memory/prompt.py 则负责长期记忆更新所需的提示与格式化，而不是主对话链里唯一的注入点。

上传文件、图片查看、summary 压缩也不是简单地“拼接到 prompt 末尾”，而是借助不同 middleware 在不同阶段注入 thread state 或额外说明。

核心源码 1：DynamicContextMiddleware 怎样把日期和 memory 包成运行时提醒

节选自 backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py。这一段直接说明 DeerFlow 为什么不把动态上下文写死进 system prompt。

middlewares/dynamic_context_middleware.py :: _build_full_reminder()
1# DynamicContextMiddleware: 专门处理每轮变化的提醒，把日期和记忆注入到运行时消息而不是静态 prompt
1class DynamicContextMiddleware(AgentMiddleware):
2 # 它的职责不是改 system prompt 常量，而是把动态上下文包装成当前轮可注入的 reminder
3 # self: 中间件实例，会从内部 app_config 和 agent_name 里推导本轮完整 reminder
3 def _build_full_reminder(self) -> str:
4 from deerflow.agents.lead_agent.prompt import _get_memory_context
5
6 # memory 注入是可配置开关；日期提醒则始终保留
7 injection_enabled = self._app_config.memory.injection_enabled if self._app_config else True
8 memory_context = _get_memory_context(self._agent_name, app_config=self._app_config) if injection_enabled else ""
9 current_date = datetime.now().strftime("%Y-%m-%d, %A")
10
11 lines: list[str] = ["<system-reminder>"]
12 if memory_context:
13 lines.append(memory_context.strip())
14 lines.append("")
15 lines.append(f"<current_date>{current_date}</current_date>")
16 lines.append("</system-reminder>")
17
18 return "\n".join(lines)
  • 动态上下文被包成 <system-reminder>，说明它不是普通用户消息，也不是静态 prompt 常量。
  • memory.injection_enabled 控制长期记忆是否注入，但日期提醒始终存在，这反映了 DeerFlow 对时间上下文的重视。
  • _get_memory_context(...) 在这里被调用，说明长期记忆进入模型视野的真正时机是“本轮模型调用前”。

核心源码 2：tool_search 为什么能减少初始上下文体积

节选自 backend/packages/harness/deerflow/tools/builtins/tool_search.py。这段代码说明工具上下文也被当成上下文工程问题来处理。

tools/builtins/tool_search.py :: DeferredToolCatalog
1"""The agent sees deferred tool names in <available-deferred-tools> but cannot
2call them until it fetches their full schema via the tool_search tool."""
3
4@dataclass(frozen=True)
5# DeferredToolCatalog: 延迟工具目录本体，只暴露名字、指纹和检索能力，不负责直接执行工具
5class DeferredToolCatalog:
6 """不可变的 deferred tools 目录，只负责搜索，不负责修改。"""
7
8 tools: tuple[BaseTool, ...]
9
10 @cached_property
11 # self: 当前目录对象；这里先提炼所有工具名字，让 prompt 只暴露“可检索能力名单”
11 def names(self) -> frozenset[str]:
12 return frozenset(t.name for t in self.tools)
13
14 @cached_property
15 # hash: 为这一整份 deferred tool 集合做版本指纹，后面 state merge 才能判断 catalog 是否发生变化
15 def hash(self) -> str:
16 canon = [{"name": t.name, "schema": convert_to_openai_function(t)} for t in sorted(self.tools, key=lambda t: t.name)]
17 blob = json.dumps(canon, sort_keys=True, ensure_ascii=False, default=str)
18 return hashlib.sha256(blob.encode("utf-8")).hexdigest()[:16]
19
20 # query: 模型发来的检索表达式，既可能是自然语言，也可能是 select:/+ 这种结构化检索语法
20 def search(self, query: str) -> list[BaseTool]:
21 # 先做空查询和特殊前缀分流，后面才会进入模糊匹配、正则打分等真正的检索逻辑
21 query = query.strip()
22 if not query:
23 return []
24
25 if query.startswith("select:"):
26 wanted = {n.strip() for n in query[7:].split(",")}
27 return [t for t in self.tools if t.name in wanted][:MAX_RESULTS]
28
29 if query.startswith("+"):
30 parts = query[1:].split(None, 1)
31 if not parts:
32 return []
33 required = parts[0].lower()
34 candidates = [t for t in self.tools if required in t.name.lower()]
35 if len(parts) > 1:
36 candidates.sort(key=lambda t: _catalog_regex_score(parts[1], t), reverse=True)
37 return candidates[:MAX_RESULTS]
  • 文件开头的注释已经把设计目标说透了：模型初始只能看到 deferred tool 的名字，拿不到完整 schema，必须显式调用 tool_search 才能“解锁”细节。
  • DeferredToolCatalog 被声明成 frozen=True 的不可变数据类，说明这份目录在 agent build 阶段构造好之后，不希望执行中再被随手改写。
  • names 和 hash 都做了缓存，前者用于 prompt 层只暴露名字，后者用于在 graph state 里识别“当前 deferred tool 集合是不是同一份”。
  • search(...) 一开始先把空查询挡掉，再支持 select: 精确点名和 + 前缀的必含关键词搜索，这说明它不是简单字符串匹配，而是给模型准备了一套可推理的检索语法。
  • 这整段实现的真正价值是控制首轮 prompt 体积。完整工具 schema 不再一股脑塞进上下文，而是按需查询、按需提升。

核心源码 3：上下文工程真正的装配点其实在 lead agent 工厂

节选自 backend/packages/harness/deerflow/agents/lead_agent/agent.py。这段代码最适合回答“这些上下文能力到底是在哪一层被装进一轮对话里的”。

lead_agent/agent.py :: context-related middleware assembly
1middlewares = build_lead_runtime_middlewares(app_config=resolved_app_config, lazy_init=True)
2
3# 先把“每轮都可能变化”的提醒注入进来
4middlewares.append(DynamicContextMiddleware(agent_name=agent_name, app_config=resolved_app_config))

13cfg = _get_runtime_config(config)
14todo_list_middleware = _create_todo_list_middleware(cfg.get("is_plan_mode", False))

25# Clarification 总是放最后，让它能在所有上下文都就位后再判断是否必须先追问
26middlewares.append(ClarificationMiddleware())
  • 这一段最重要的价值，是把“上下文工程不只是 prompt 文件”的事实钉死了。真正决定本轮能看到哪些上下文信息的，是这里的 middleware 装配顺序。
  • `DynamicContextMiddleware` 和 `SkillActivationMiddleware` 靠前，说明 DeerFlow 认为“先让模型看到当前轮动态提醒和显式激活 skill”比“先统计 token”更重要。
  • `summarization` 和 `todo` 也是上下文工程的一部分，因为它们会直接改变保留下来的消息体积和结构。
  • `TitleMiddleware` 与 `MemoryMiddleware` 靠后，体现了一个很真实的执行顺序：先完成当前轮理解和回答，再做线程整理与长期记忆持久化。
  • 所以如果你想在 DeerFlow 里改上下文策略，通常不应该只改 prompt 文本，而要回到这条 middleware 链看“该在哪一阶段注入”。
一个很 DeerFlow 的点： 工具上下文也被分层处理。backend/packages/harness/deerflow/tools/builtins/tool_search.py 让 MCP 工具默认只以名字进入 prompt，真正 schema 要在运行中按需搜索，避免一次把全部工具细节塞进模型上下文。

配置面与使用方式

从使用面看，DeerFlow 的上下文工程至少有这些开关和入口：

  • 启用中的 skills 会被 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 缓存并注入，影响后续新 run 的 system prompt。
  • memory 配置与 backend/packages/harness/deerflow/agents/memory/prompt.py 的 token counting 策略决定记忆注入成本和回退方式。
  • backend/packages/harness/deerflow/agents/factory.py 里的 feature 组合和 middleware 顺序，决定是否挂入 dynamic_context、summarization、memory、uploads、view-image、todo 等上下文相关能力。
  • 上传文件与图片由对应 middleware 转成 thread 级上下文资产，而不是让 prompt 静态引用宿主机路径。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 的 deferred tool 策略，决定模型最初看到的是“工具名列表”还是完整 schema。

关键实现细节与权衡

  • backend/packages/harness/deerflow/agents/lead_agent/prompt.py 使用 enabled skills 缓存与失效刷新，避免每次请求同步扫盘；代价是理解 prompt 来源时要同时考虑缓存生命周期。
  • backend/packages/harness/deerflow/agents/factory.py 把上下文相关能力拆成 middleware 链，而不是一个超级 prompt builder，这样更组合化，但调试时需要跨文件追踪。
  • backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py 把 memory 与当前日期做成 per-turn 注入，而不是写死在静态 prompt 文件里，这更贴近实时状态，但也让“上下文从哪来”必须按静态提示和动态提示分开理解。
  • backend/packages/harness/deerflow/agents/memory/prompt.py 在 tiktoken 不可用时会退回字符估算，说明 DeerFlow 明确把“上下文预算可用性”看得比“绝对精确 token 数”更重要。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 采用 deferred tools，是为了控制上下文体积；代价是模型必须先学会何时搜索工具，而不是天然拿到完整 schema。
  • backend/packages/harness/deerflow/agents/middlewares/ 中多种 middleware 联动，说明 DeerFlow 的上下文工程是运行态注入系统，不是单一 prompt 模板。

从源码风格也能看出这个取向。很多上下文不是直接拼接大字符串，而是借助消息对象、state 字段和 middleware hook 渐进式加入。这种做法牺牲了“一眼看到完整 prompt”的简单性，但换来了更细粒度的生命周期控制，例如哪些提醒要被保留、哪些可以被 summary 压缩、哪些应该只影响当前轮。

核心取舍： DeerFlow 选择“少量静态 prompt + 大量运行时中间件联动”的方式管理上下文，这比纯 prompt 拼接更复杂，但更贴合真实 thread、files、images、memory 和 tools 的生命周期。

当前限制、边界与非目标

  • DeerFlow 的上下文工程不是通用 RAG 框架，也不是把所有外部知识统一索引后再检索注入。
  • skills、memory、summary、uploads、images 都会影响上下文，但它们不是同一种数据层，不能简单合并成一个“万能 context blob”。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 只解决 deferred tool schema 暴露，不负责所有工具策略判断。
  • 本章不展开每个 middleware 逐行源码；重点是理解它们如何共同塑造模型看到的上下文面。

它与 DeerFlow 其他能力的关系

上下文工程几乎连接 DeerFlow 的所有运行时能力：

  • 它和第 11 章 skills 系统相连，因为 skills 进入 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 后，会直接改变系统提示。
  • 它和第 13 章 memory 相连，因为长期记忆既依赖 backend/packages/harness/deerflow/agents/memory/prompt.py 的更新提示，也依赖 dynamic_context 这类 middleware 在当轮注入可见内容。
  • 它和第 12 章 files / sandbox 相连，因为 uploads、artifacts、images 最终都要变成模型可理解的 thread 资产。
  • 它和第 21、23 章模型集成与高级模式相连，因为上下文预算、thinking 模式、tool search 和 subagent 组合都会改变最优上下文策略。
建议把 backend/packages/harness/deerflow/agents/lead_agent/prompt.py、backend/packages/harness/deerflow/agents/factory.py、backend/packages/harness/deerflow/agents/middlewares/、backend/packages/harness/deerflow/agents/memory/prompt.py 和 backend/packages/harness/deerflow/tools/builtins/tool_search.py 连起来读。这样最容易看到 DeerFlow 上下文工程的真实联动机制。
