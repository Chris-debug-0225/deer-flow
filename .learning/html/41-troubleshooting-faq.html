DeerFlow 学习系列 - 第 41 章

本章讲的是遇到问题时，应该把 DeerFlow 故障先归到哪一层

排查 DeerFlow 最容易犯的错，是一上来就猜模型问题。更有效的方式是先按层分类：配置层、Gateway/API 层、streaming 层、auth 层、channels 层、sandbox/files 层。这个 FAQ 的目标，是让读者遇到问题时能快速回到真实代码和文档入口。

这项能力在 DeerFlow 中解决什么问题

当 Gateway 起不来、流式输出不对、登录态异常、channel 不回消息、uploads 或 artifacts 表现奇怪时，最难的往往不是修复本身，而是不知道问题属于哪一层。FAQ 章节解决的就是这个定位问题。

这项能力在代码库中的位置

路径 典型问题 为什么先看这里
backend/docs/CONFIGURATION.md 模型、sandbox、uploads、subagents、tool_search 配置错误 很多“系统坏了”其实是配置没有对齐
backend/docs/STREAMING.md 浏览器流式输出异常、client.stream 表现不一致 DeerFlow 明确有两条并行 streaming 路径
backend/docs/AUTH_DESIGN.md 登录、CSRF、default user bucket、internal channel 调用 权限与 identity 问题不能靠猜
backend/app/gateway/ threads、runs、uploads、artifacts、skills、models 路由问题 外部调用面的大多数问题都要回到这里
backend/app/channels/ IM 渠道不收消息、不回消息、附件处理异常 channel manager 和 provider 具体实现都在这里

真实运行链路或数据链路

symptom appears -> classify by layer -> config? auth? gateway? streaming? channels? files? -> open matching doc + code anchor -> verify runtime path -> only then inspect model/tool behavior

例如“前端不再流式输出”通常先看 backend/docs/STREAMING.md 与 Gateway run/stream 路由；“飞书不回消息”先看 backend/app/channels/ 和 internal auth；“上传文件后 agent 读不到”先看 uploads、artifacts 和 sandbox 路径契约。

配置面与使用方式

  • 先用 backend/docs/CONFIGURATION.md 对照当前配置，确认模型、sandbox、uploads、subagents、channels 和 tool_search 没有跑偏。
  • 涉及流式问题时，回到 backend/docs/STREAMING.md 分清是 Gateway SSE 路径还是 DeerFlowClient 同步路径。
  • 涉及权限或 channel 问题时，回到 backend/docs/AUTH_DESIGN.md 看 internal token、CSRF 和 default 用户桶语义。
  • 外部 API 行为异常时，优先看 backend/app/gateway/ 下的对应 router；IM 行为异常时，优先看 backend/app/channels/。

关键实现细节与权衡

  • DeerFlow 把很多能力都收在 Gateway runtime 中，优点是边界统一，代价是排障时要先分清“请求有没有真正进入 Gateway”。
  • 双 streaming 路径让不同消费者各得其所，但也意味着不能用浏览器症状去直接推断 embedded client 一定同样出错。
  • channel 复用 internal auth 与 default user bucket 很实用，但如果不理解这层语义，很容易把平台级 bot 行为误解成多用户隔离 bug。

当前限制、边界与非目标

  • FAQ 不是完整运维手册，它更像“先去哪一层找”的故障入口地图。
  • 很多问题最终仍要回到具体章节和源码，不会在 FAQ 页面里完整展开。
  • 如果问题是业务级 prompt 或模型行为偏差，这一章只能帮你先排除系统层故障。

它与 DeerFlow 其他能力的关系

  • 它和第 40 章源码导航相连，因为排障的第一步往往是知道该去哪个目录找线索。
  • 它和 Gateway、streaming、auth、channels、sandbox 等章节相连，因为 FAQ 的每个问题最终都要回到这些真实实现边界。
  • 它是整个学习系列的收口页，帮助读者把前面理解过的设计边界转化为实际排障路径。
建议把 backend/docs/CONFIGURATION.md、backend/docs/STREAMING.md、backend/docs/AUTH_DESIGN.md、backend/app/gateway/ 和 backend/app/channels/ 作为排障时的第一层索引。
