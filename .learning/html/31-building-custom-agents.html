构建自定义Agent

DeerFlow 学习系列 - 第 31 章

本章讲的是怎样沿着 DeerFlow 现有边界做自定义 agent，而不是重新造一个平行框架

在 DeerFlow 里做自定义 agent，最稳的路径通常不是绕开所有现有能力，而是复用已有的 client、factory、skills、middleware 和配置结构。理解这个边界，比“怎么写一个新 prompt”更重要。

这项能力在 DeerFlow 中解决什么问题

有些场景下，你并不想只是在默认 lead agent 上加一个 skill，而是希望定制 agent 的系统提示、能力组合、运行模式，甚至把 DeerFlow 当成 Python 库嵌入自己的应用。第 31 章回答的就是这个问题：如何在不背离 DeerFlow 当前实现的前提下，构建自定义 agent。

核心源码：自定义 Agent 的真正入口不是“写 prompt”，而是工厂如何接管参数

节选自 backend/packages/harness/deerflow/agents/factory.py。如果你想知道“自定义 agent 到底在哪里生效”，最重要的就是这段工厂函数。

核心源码：外部调用方传进来的参数，最终是怎样在 Client 里变成真正 Agent 实例的

节选自 backend/packages/harness/deerflow/client.py。这段代码把“我给 DeerFlowClient 传了 model、skills、middlewares，最终到底在哪里生效”完整串起来。

核心源码：自定义 Agent 真正跑起来时，Client 的 `stream()` 又如何把它变成可消费事件流

配置面与使用方式

  • backend/packages/harness/deerflow/client.py 当前实际公开的定制面主要是 model_name、agent_name、available_skills 和 middlewares 等参数；这里讲的是现状，不等同于 RFC 里更理想化的未来接口分层。
  • backend/packages/harness/deerflow/agents/factory.py 则负责把这些覆盖项落实成工具集合、middleware 链和 prompt/runtime 组合。
  • backend/docs/rfc-create-deerflow-agent.md 强调的是未来希望达到的参数覆盖优先、配置回退其次路线；阅读时要把它和 backend/packages/harness/deerflow/client.py 当前已经实现的公开参数区分开。
  • 如果你还需要自定义 skill、自定义 subagent 或不同模型策略，这些也应沿 DeerFlow 现有配置和注册机制扩展，而不是硬 fork 一套平行工厂。

关键实现细节与权衡

  • 走 backend/packages/harness/deerflow/client.py 的好处是可以直接复用 DeerFlow 现有能力；代价是你也会继承它当前已经公开、但仍然有限的参数面以及配置与运行时假设。
  • backend/packages/harness/deerflow/agents/factory.py 提供了很强的组合能力，但它本质仍是内部装配层，不适合被当成外部稳定 SDK 契约使用。
  • backend/docs/rfc-create-deerflow-agent.md 之所以强调“唯一公开 API”与“纯参数工厂”，是为了减少隐式 I/O 和全局状态给自定义 agent 带来的困扰；但这应被理解为 RFC 设计目标，而不是已经完整兑现的现状描述。

当前限制、边界与非目标

  • 构建自定义 agent 不等于当前就有一套完全独立、零隐式依赖的 agent SDK。
  • 如果你直接深入内部工厂层，可能会碰到全局配置、extensions 状态和运行时装配细节，而这些并不都是面向外部承诺的稳定接口。
  • 本章不鼓励复制 DeerFlow 默认执行引擎再魔改，那样最容易偏离仓库的真实能力演进路径。

它与 DeerFlow 其他能力的关系

  • 它和第 10 章执行引擎相连，因为自定义 agent 最终还是要回到 DeerFlow 的 agent 构建与执行主线。
  • 它和第 11、30 章 skills 相连，因为自定义 agent 通常也要决定允许哪些 skills、如何使用这些能力包。
  • 它和第 21 章模型集成相连，因为模型覆盖是自定义 agent 最常见的定制面之一。
  • 它和 RFC 文档相连，因为 backend/docs/rfc-create-deerflow-agent.md 实际上就是理解这层边界的最佳设计说明。
