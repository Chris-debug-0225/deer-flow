简介

Agent 执行引擎是 DeerFlow 的核心心脏。它负责接收用户输入，驱动 LLM 进行推理，调用工具，处理结果，直到任务完成。本章深入讲解执行引擎的设计原理、状态管理机制、流程控制逻辑和错误恢复策略。

学习目标：
  • 理解 Agent 执行循环的完整流程
  • 掌握状态管理和转移机制
  • 学会处理错误和异常恢复
  • 理解决策逻辑和流程控制

Agent 执行循环

Agent 执行的核心是一个 循环决策过程：接收输入 → LLM 推理 → 工具调用 → 结果处理 → 判断是否完成 → 重复或返回。这个循环直到 Agent 决定任务完成或达到最大步数为止。

执行循环的代码实现

下面是 Agent 执行循环的核心伪代码，展示了整个流程的逻辑：

class AgentExecutor: def run(self, user_input: str) -> str: # 初始化执行状态 state = ExecutionState( messages=[HumanMessage(content=user_input)], step=0, status="running" ) # 执行循环 while state.status == "running": # 1. 调用 LLM 进行推理 response = self.llm.invoke(state.messages) state.messages.append(response) # 2. 解析 LLM 输出，判断是否需要调用工具 action = self.parse_action(response) # 3. 如果 LLM 决定完成，退出循环 if action.type == "finish": state.status = "completed" break # 4. 调用工具 tool_result = self.execute_tool(action) state.messages.append( ToolMessage(content=tool_result) ) # 5. 检查是否超过最大步数 state.step += 1 if state.step >= self.max_steps: state.status = "max_steps_reached" break return state.get_final_output()
关键点： Agent 执行循环的核心是 消息链。每一步的结果都被追加到消息历史中，供下一步的 LLM 推理使用。这样 LLM 始终能看到完整的上下文。

状态管理机制

Agent 在执行过程中需要维护多种状态：当前消息历史、执行步数、工具调用结果、错误信息等。良好的状态管理是确保执行可靠性的基础。

执行状态的组成

状态字段 说明 用途
messages 消息历史列表 保存用户输入、LLM 回复、工具结果
step 当前执行步数 控制最大迭代次数，防止无限循环
status 执行状态 running / completed / failed / max_steps_reached
context 执行上下文 存储 Agent 配置、工具集、记忆等
error_info 错误信息 记录执行过程中的错误，用于调试

决策逻辑与流程控制

Agent 的决策逻辑由 LLM 驱动。LLM 根据当前消息历史和系统提示，决定下一步应该做什么：继续调用工具、返回最终答案，还是请求人类干预。

LLM 输出的解析

LLM 的输出需要被解析成结构化的 Action 对象，包含动作类型和参数：

class Action: """LLM 决策的结构化表示""" def __init__(self, action_type: str, **kwargs): self.type = action_type # "tool_call" / "finish" / "wait_human" self.params = kwargs # 示例：调用工具 action1 = Action( "tool_call", tool_name="weather", args={"city": "北京"} ) # 示例：返回最终答案 action2 = Action( "finish", result="北京今天天气晴朗，温度 25°C" ) # 示例：等待人类确认 action3 = Action( "wait_human", message="是否确认删除该数据？" )

流程控制的条件判断

执行引擎根据 Action 类型进行不同的处理：

def handle_action(self, action: Action) -> str: if action.type == "tool_call": # 调用工具并获取结果 result = self.execute_tool( action.params["tool_name"], action.params["args"] ) return result elif action.type == "finish": # 任务完成，返回最终答案 return action.params["result"] elif action.type == "wait_human": # 等待人类输入 human_input = input(action.params["message"]) return human_input
提示： 好的流程控制设计应该支持多种决策类型，不仅仅是工具调用和完成，还包括人类干预、错误恢复等。这样可以构建更灵活、更安全的 Agent 系统。

错误处理与恢复

在实际执行中，各种错误都可能发生：工具调用失败、LLM 输出格式错误、网络超时等。良好的错误处理机制能让 Agent 更加健壮。

常见的错误类型

错误类型 原因 处理策略
工具不存在 LLM 调用了未注册的工具 记录错误，告知 LLM，继续执行
工具执行失败 工具运行时异常 捕获异常，返回错误信息给 LLM
LLM 输出格式错误 LLM 输出无法解析 重试或降级处理
超时 工具或 LLM 调用超时 中断并返回超时错误

错误处理的实现

def execute_tool(self, tool_name: str, args: Dict) -> str: try: # 检查工具是否存在 if tool_name not in self.tools: raise ToolNotFoundError( f"Tool '{tool_name}' not found" ) # 执行工具 tool = self.tools[tool_name] result = tool.execute(**args) return result except ToolNotFoundError as e: # 工具不存在，记录并返回错误信息 error_msg = f"Error: {str(e)}" self.log_error(error_msg) return error_msg except Exception as e: # 其他异常，记录并返回错误信息 error_msg = f"Tool execution failed: {str(e)}" self.log_error(error_msg) return error_msg
警告： 不要让错误中断整个执行流程。应该捕获错误，记录日志，然后将错误信息作为消息返回给 LLM，让 LLM 决定如何处理。这样可以让 Agent 更加健壮。

重试机制

对于某些可恢复的错误（如网络超时、临时服务不可用），可以实现自动重试机制。DeerFlow 提供了灵活的重试策略配置。

class RetryPolicy: """重试策略配置""" def __init__(self, max_retries=3, backoff_factor=2): self.max_retries = max_retries self.backoff_factor = backoff_factor self.retry_count = 0 def should_retry(self, error: Exception) -> bool: # 判断是否应该重试 if self.retry_count >= self.max_retries: return False # 只重试特定类型的错误 retriable_errors = ( TimeoutError, ConnectionError, RuntimeError ) return isinstance(error, retriable_errors) def get_wait_time(self) -> float: # 计算指数退避等待时间 return self.backoff_factor ** self.retry_count
示例： 带重试的工具执行
def execute_tool_with_retry(self, tool_name: str, args: Dict) -> str: retry_policy = RetryPolicy(max_retries=3) while True: try: tool = self.tools[tool_name] result = tool.execute(**args) return result except Exception as e: if not retry_policy.should_retry(e): raise # 等待后重试 wait_time = retry_policy.get_wait_time() time.sleep(wait_time) retry_policy.retry_count += 1

性能优化

Agent 执行的性能直接影响用户体验。优化执行引擎的性能需要从多个方面考虑：消息管理、并发执行、缓存等。

消息压缩与总结

长时间运行的 Agent 会积累大量消息历史，导致 LLM 调用成本增加和响应变慢。可以实现消息压缩机制来解决这个问题。

class MessageCompressor: """消息压缩器，用于减少消息历史大小""" def __init__(self, max_messages=50, summary_threshold=100): self.max_messages = max_messages self.summary_threshold = summary_threshold def compress(self, messages: List) -> List: # 如果消息数超过阈值，进行压缩 if len(messages) > self.summary_threshold: # 保留最近的 max_messages 条消息 # 对早期消息进行总结 old_messages = messages[:-self.max_messages] recent_messages = messages[-self.max_messages:] # 生成总结 summary = self.summarize(old_messages) return [summary] + recent_messages return messages def summarize(self, messages: List) -> str: # 使用 LLM 生成总结 pass

并发工具执行

当 Agent 需要调用多个独立的工具时，可以使用并发执行来加速。DeerFlow 支持异步工具执行。

import asyncio from typing import List, Dict async def execute_tools_concurrently( self, tool_calls: List[Dict] ) -> List[str]: """并发执行多个工具调用""" # 创建异步任务 tasks = [ self.execute_tool_async(call['name'], call['args']) for call in tool_calls ] # 并发执行 results = await asyncio.gather(*tasks) return results

详解：为什么列表推导式创建的是"协程列表"？

这是 Python asyncio 中一个常见但容易困惑的点。关键在于理解 调用 async def 函数时发生了什么。

核心概念：调用 async def 函数 ≠ 执行函数
当你调用一个用 async def 定义的函数时，Python 不会立即执行函数体内的代码，而是返回一个 协程对象（coroutine object）。协程对象可以理解为 一个"尚未启动的异步任务的封装"，只有通过 await 或 asyncio.gather() 驱动时才会真正开始执行。

让我们拆解列表推导式的两个部分：

# 第一步：self.execute_tool_async(...) 的返回值是什么？ # execute_tool_async 是 async def 定义的函数 async def execute_tool_async(self, name, args): # ... 函数体 ... return result # ★ 直接调用（不加 await） coro = self.execute_tool_async('search', {'q': 'hello'}) # coro 是一个 coroutine 对象，函数体尚未执行！ # 可以用 type() 验证： print(type(coro)) # # 必须用 await 才能驱动它真正执行： result = await coro # 此时函数体才真正开始运行
# 第二步：列表推导式遍历 tool_calls，每次调用都产生一个协程对象 # 假设 tool_calls = [{'name': 'search', 'args': {...}}, {'name': 'calc', 'args': {...}}] tasks = [ self.execute_tool_async(call['name'], call['args']) for call in tool_calls ] # 展开等价于： tasks = [ self.execute_tool_async('search', {...}), # → coroutine 对象 #1（未执行） self.execute_tool_async('calc', {...}), # → coroutine 对象 #2（未执行） ] # 所以 tasks 的类型是 List[Coroutine]，而非 List[结果] print(type(tasks[0])) #

与普通函数的对比：

普通函数 def 异步函数 async def
调用时 立即执行函数体 调用时 不执行，返回协程对象
返回值就是函数的结果 返回值是 coroutine 对象
列表推导式得到的是结果列表 列表推导式得到的是协程列表
[foo(x) for x in items] [bar(x) for x in items] → [coro, coro, ...]
# 第三步：asyncio.gather(*tasks) 将协程列表"展开"并并发驱动 results = await asyncio.gather(*tasks) # *tasks 解包：gather(coro_1, coro_2, ...) # gather 同时调度所有协程，它们在事件循环中交替执行 # await 等待全部完成，results 按顺序对应每个协程的返回值 # results = [搜索结果, 计算结果] ← 真正的结果列表

总结流程图：

列表推导式 asyncio.gather 最终结果 ───────────── ────────────────── ────────── tool_calls ╔═══╗ gather() 同时启动 results ┌──┐ ┌──┐ ║ ★ ║ ┌──────────┐ ┌──┐ ┌──┐ │A │ │B │ ──→ ║ ║ │ coro_A ▓▓ │ ──→ │A'│ │B'│ └──┘ └──┘ ║ ║ │ coro_B ▓▓ │ └──┘ └──┘ │ │ ╚═══╝ └──────────┘ (真正的结果) 协程对象 并发执行(交替运行) (未启动)

缓存机制

对于相同的工具调用，可以缓存结果以避免重复执行。这对于那些执行时间长或成本高的工具特别有用。

class ToolCache: """工具执行结果缓存""" def __init__(self, ttl=3600): self.cache = {} self.ttl = ttl # 缓存有效期（秒） def get_key(self, tool_name: str, args: Dict) -> str: # 生成缓存键 import json return hash(json.dumps( {"tool": tool_name, "args": args}, sort_keys=True )) def get(self, tool_name: str, args: Dict): key = self.get_key(tool_name, args) if key in self.cache: cached_value, timestamp = self.cache[key] if time.time() - timestamp < self.ttl: return cached_value return None def set(self, tool_name: str, args: Dict, result): key = self.get_key(tool_name, args) self.cache[key] = (result, time.time())

详解：get_key 中的 hash(json.dumps(..., sort_keys=True))

get_key 方法的作用是为每次工具调用生成一个唯一且稳定的缓存键。 核心逻辑是 hash(json.dumps(...))，从内到外分两层理解：

第一层：json.dumps(..., sort_keys=True)
将 {"tool": tool_name, "args": args} 字典序列化为 JSON 字符串， sort_keys=True 确保键按字母顺序排列， 使参数顺序不同但内容相同的调用生成相同的字符串。

# 不加 sort_keys：键的顺序不确定，相同参数可能生成不同字符串 d1 = {"page": 1, "q": "hello"} d2 = {"q": "hello", "page": 1} json.dumps(d1) # '{"page": 1, "q": "hello"}' json.dumps(d2) # '{"q": "hello", "page": 1}' ← 不同！缓存失效 # 加了 sort_keys=True：统一按字母排序，结果完全一致 json.dumps(d1, sort_keys=True) # '{"page": 1, "q": "hello"}' json.dumps(d2, sort_keys=True) # '{"page": 1, "q": "hello"}' ← 一致！缓存命中 ✓

第二层：hash(...)
将 JSON 字符串压缩为一个固定大小的整数， 用整数做字典键比长字符串更省内存、查找更快。

# 完整流程示例 # 输入：tool_name="search", args={"q": "hello", "page": 1} # ↓ 组装字典 {"tool": "search", "args": {"q": "hello", "page": 1}} # ↓ json.dumps(sort_keys=True) — 键按字母排序 '{"args": {"page": 1, "q": "hello"}, "tool": "search"}' # ↓ hash() — 压缩为整数 3847291058374 # ↓ 作为 cache 字典的 key cache = {3847291058374: ("搜索结果", timestamp)}

⚠ 注意：Python 内置 hash() 在不同进程间结果不一致
出于安全考虑（防止哈希碰撞攻击），Python 3 默认为字符串加入随机盐， 同一字符串在不同进程中的 hash() 值不同。 如果缓存需要跨进程持久化（如写入 Redis / 文件）， 应改用 hashlib.md5() 等稳定哈希算法：

import hashlib, json def get_key_stable(self, tool_name: str, args: Dict) -> str: # 使用 md5 代替内置 hash，保证跨进程一致性 raw = json.dumps( {"tool": tool_name, "args": args}, sort_keys=True ) return hashlib.md5(raw.encode()).hexdigest() # → 'a1b2c3d4...'（固定长度十六进制字符串，跨进程稳定）

高级模式

除了基本的执行模式，DeerFlow 还支持多种高级执行模式，适应不同的应用场景。

流式执行（Streaming）

对于需要实时反馈的应用，可以使用流式执行模式，逐步返回结果而不是等待完全完成。

流式执行是一个 async generator（异步生成器）， 用 yield 逐步产出事件，调用方可以实时消费每一步的结果， 而不必等整个 Agent 跑完。

async def run_streaming(self, user_input: str): """流式执行 Agent，逐步 yield 事件给调用方""" state = ExecutionState( messages=[HumanMessage(content=user_input)] ) while state.is_running(): # 1. LLM 流式推理：逐块接收，边收边拼接完整回复 response = "" async for chunk in self.llm.invoke_streaming(state.messages): response += chunk # 拼接完整文本，后面 parse_action 要用 yield {"type": "llm_chunk", "data": chunk} # 实时推送每个片段 # 2. 把 LLM 回复记录到消息历史（下一轮 LLM 需要看到） state.add_message(AIMessage(content=response)) # 3. 解析 LLM 输出，决定下一步动作 action = self.parse_action(response) # 4. 如果 LLM 决定结束，更新状态并终止 if action.type == "finish": state.status = "finished" # 更新状态 → is_running() 返回 False yield {"type": "finished", "data": action.params} break # 5. 执行工具调用，把结果也记录到消息历史 result = await self.execute_tool_async(action) state.add_message(ToolMessage(content=result)) # 没有这行 → 下一轮 LLM 看不到工具结果 → 死循环 yield {"type": "tool_result", "data": result} # 6. 步数保护：防止 Agent 无限循环 state.step += 1 if state.step >= self.max_steps: state.status = "max_steps_reached" break

关键设计解析

① response 的拼接 —— 用具体例子理解
流式模式下，LLM 不是一次性返回完整文本，而是像打字机一样 一个字一个词地吐出 chunk（文本片段）。 每个 chunk 到达时做两件事：response += chunk（拼起来）和 yield（立即推给前端）。

# ━━━━━━━━━━━━ 场景：用户问"北京天气如何？" ━━━━━━━━━━━━ # LLM 决定调用 weather 工具，它的回复是一个 JSON 指令 # 完整文本应该是： # {"action": "tool_call", "tool": "weather", "city": "北京"} # # 但流式模式下不是一次返回，而是分成多个 chunk： # ━━━━ chunk 1 到达 ━━━━ chunk = '{"acti' response += chunk # response = '{"acti' yield {"type": "llm_chunk", "data": '{"acti'} # 前端立刻显示这段文字 # ━━━━ chunk 2 到达 ━━━━ chunk = 'on": "tool' response += chunk # response = '{"action": "tool' yield {"type": "llm_chunk", "data": 'on": "tool'} # ━━━━ chunk 3 到达 ━━━━ chunk = '_call", "too' response += chunk # response = '{"action": "tool_call", "too' yield {"type": "llm_chunk", "data": '_call", "too'} # ━━━━ chunk 4 到达（最后一个）━━━━ chunk = 'l": "weather", "city": "北京"}' response += chunk # response = '{"action": "tool_call", "tool": "weather", "city": "北京"}' yield {"type": "llm_chunk", "data": 'l": "weather", "city": "北京"}'} # ━━━━ async for 循环结束 ━━━━ # 此时 response 已拼接成完整文本： # '{"action": "tool_call", "tool": "weather", "city": "北京"}' # parse_action(response) 就能把它解析成结构化的 Action 对象 action = self.parse_action(response) # action.type = "tool_call" # action.params = {"tool": "weather", "city": "北京"}

为什么要 yield 每个 chunk？
用户体验！前端收到 llm_chunk 后立刻追加显示，就像 ChatGPT 打字效果一样。 如果等全部 chunk 拼完再返回，用户要盯着空白屏幕干等好几秒。 response += chunk 是对内用的（拼完整文本给 parse_action）， yield chunk 是对外用的（实时推给前端展示）。两者不矛盾。

② state.messages 必须更新
Agent 是一个 while 循环，每轮 LLM 都要看到完整对话历史。 如果不把 LLM 回复和工具结果加入 messages， 下一轮 LLM 会看到和上一轮完全一样的输入 → 死循环，反复做同样的事。

一轮循环中消息历史的变化：

# 循环开始时： messages = [HumanMessage("北京天气如何？")] # LLM 回复后，加入 AIMessage： messages = [HumanMessage("北京天气如何？"), AIMessage("我需要查一下天气")] # 工具执行后，加入 ToolMessage： messages = [HumanMessage("北京天气如何？"), AIMessage("..."), ToolMessage("25°C 晴")] # 下一轮 LLM 看到完整历史 → 能基于工具结果给出最终答案

③ state.is_running() 与 state.status
is_running() 返回 self.status == "running"， 循环中需要在关键节点更新 status：

  • "finished" — LLM 决定结束
  • "max_steps_reached" — 超过最大步数保护
  • "waiting" — 等待人类介入

虽然代码里也用了 break，但更新 status 的意义是 让外部调用方能通过 state.status 知道 Agent 是正常结束还是异常退出。

④ yield 的三种事件类型
调用方通过 async for event in run_streaming(...) 消费：

事件类型含义频率
llm_chunkLLM 输出的一个文本片段每轮可能多次
tool_result工具执行完毕的结果每轮最多一次
finishedAgent 决定结束，附带最终答案仅最后一次

如何调用 run_streaming？—— 完整使用示例

run_streaming 是一个 async generator（异步生成器）， 调用方需要用 async for 来逐个消费它 yield 出的事件。

import asyncio # ━━━━━━━━━━━━ 调用方代码 ━━━━━━━━━━━━ async def chat(agent, user_input: str): # run_streaming 是 async generator，必须用 async for 迭代 async for event in agent.run_streaming(user_input): if event["type"] == "llm_chunk": # LLM 吐出一个文字片段 → 实时追加到前端界面 print(event["data"], end="", flush=True) # 输出效果：{"acti ←on": "tool← ←_call"...（打字机效果） elif event["type"] == "tool_result": # 工具执行完毕 → 显示结果 print(f"\n[工具结果] {event['data']}") # 输出：[工具结果] 北京今天 25°C，晴 elif event["type"] == "finished": # Agent 决定结束 → 展示最终答案 print(f"\n[最终回答] {event['data']}") # 输出：[最终回答] 北京今天 25°C，晴，适合出行。 # ━━━━━━━━━━━━ 实际运行 ━━━━━━━━━━━━ agent = Agent(llm=my_llm, tools=[weather_tool]) asyncio.run(chat(agent, "北京天气如何？"))
# ━━━━━━━━━━━━ 终端实际输出效果 ━━━━━━━━━━━━ # 第一轮：LLM 流式输出调用指令（打字机效果逐字显示） # {"action": "tool_call", "tool": "weather", "city": "北京"} # [工具结果] 北京今天 25°C，晴 # # 第二轮：LLLM 看到工具结果后，流式输出最终回答 # {"action": "finish", "result": "北京今天 25°C，晴，适合出行。"} # [最终回答] {"result": "北京今天 25°C，晴，适合出行。"}

关键理解：yield 和 async for 是交替执行的，不是批量的

这是理解流式执行最关键的一点： run_streaming（生产者）和 chat（消费者）是 交替执行的。每 yield 一次，控制权立刻交给消费者， 消费者处理完（比如 print）后，控制权再回到生产者继续往下跑。 绝不是生产者先跑完整个 while 循环，消费者再一次性处理所有事件。

❌ 常见误解：run_streaming 把整个 while 循环跑完， 把所有 yield 的事件攒成一批，然后再交给 chat 统一打印。

✅ 实际行为：yield 像一个"暂停按钮"，每按一次， 生产者暂停，消费者立刻拿到这一条事件并处理， 处理完后生产者才恢复继续执行。

下面用逐行编号的方式展示完整的交错执行顺序：

# ━━━━━━━━━━━━ 生产者 run_streaming 和 消费者 chat 交替执行 ━━━━━━━━━━━━ # # 左侧 ═══ run_streaming（生产者） # 右侧 ─── chat（消费者） # # ┌─────────────────────────────────────────────────────────────┐ # │ 消费者 chat 先启动：async for event in agent.run_streaming() │ # │ → 触发 run_streaming 开始执行，进入 while 循环 │ # └─────────────────────────────────────────────────────────────┘ # # 第一轮 while，LLM 流式推理： # # ① ═══ response = "" # ② ═══ chunk 1 到达: '{"acti' # ═══ response += chunk → response = '{"acti' # ═══ yield {"type":"llm_chunk", "data": '{"acti'} ← 暂停！控制权交给消费者 # ③ ─── event = {"type":"llm_chunk", "data":'{"acti'} # ─── 匹配 llm_chunk → print('{"acti') ← 前端立刻显示 {"acti # ─── async for 循环体跑完，回到 for，请求下一个事件 ← 控制权还给生产者 # ④ ═══ 恢复！继续 async for chunk 循环 # ═══ chunk 2 到达: 'on": "tool' # ═══ response += chunk → response = '{"action": "tool' # ═══ yield {"type":"llm_chunk", "data":'on": "tool'} ← 又暂停！ # ⑤ ─── event = {"type":"llm_chunk", "data":'on": "tool'} # ─── print('on": "tool') ← 前端追加上去 # ─── 回到 for，请求下一个事件 # ⑥ ═══ 恢复！chunk 3 到达... # ═══ ... 同上交替 ... # ⑦ ═══ chunk 4 到达（最后一个），yield 后消费者 print # ⑧ ═══ async for chunk 循环结束 # ═══ response = '{"action": "tool_call", ...}'（完整文本） # ═══ state.add_message(AIMessage(...)) # ═══ action = parse_action(response) → type="tool_call" # ═══ 执行工具 → result = "25°C 晴" # ═══ state.add_message(ToolMessage(...)) # ═══ yield {"type":"tool_result", "data":"25°C 晴"} ← 暂停！ # ⑨ ─── event = {"type":"tool_result", ...} # ─── 匹配 tool_result → print("[工具结果] 25°C 晴") # ─── 回到 for，请求下一个事件 # ⑩ ═══ 恢复！state.step += 1，未超限，while 继续下一轮 # ═══ 第二轮 LLM 推理... yield llm_chunk × N ... # ═══ action.type == "finish" # ═══ state.status = "finished" # ═══ yield {"type":"finished", "data": {...}} ← 暂停！ # ⑪ ─── event = {"type":"finished", ...} # ─── print("[最终回答] 北京今天 25°C，晴") # ─── 回到 for # ⑫ ═══ generator 执行完毕（break + 函数结束），没有更多 yield # ═══ async for 捕获 StopAsyncIteration，循环正常结束

一句话总结执行顺序：

yield 暂停生产者 → 消费者拿到事件并 print → 消费者回到 async for 请求下一个 → 生产者从暂停处恢复继续 → 遇到下一个 yield 再暂停 → ...
交替穿插，直到生产者没有更多 yield 为止。

# 时间线（从左到右 = 时间流逝）： # # 时刻1 时刻2 时刻3 时刻4 时刻5 时刻6 时刻7 时刻8 时刻9 ... # │ │ │ │ │ │ │ │ │ # 生产者 消费者 生产者 消费者 生产者 消费者 生产者 消费者 ... # yield print yield print yield print yield print # chunk1 chunk1 chunk2 chunk2 chunk3 chunk3 chunk4 chunk4 # # 而不是： # 生产者生产 chunk1 → chunk2 → chunk3 → chunk4 → tool_result → finished # 然后消费者一次性打印全部 ← ❌ 不会这样！

为什么这样设计？
如果生产者先跑完再批量交给消费者，那 llm_chunk 的"实时打字机效果"就没了—— 用户还是要等 LLM 全部吐完才看到内容。正是因为 yield / async for 交替执行， 每个文字片段到达后立刻被推送到前端显示，用户才能看到逐字输出的效果。

批量执行（Batch）

对于需要处理大量任务的场景，可以使用批量执行模式，提高吞吐量。

def run_batch(self, inputs: List[str]) -> List[str]: """批量执行 Agent""" results = [] for user_input in inputs: result = self.run(user_input) results.append(result) return results async def run_batch_async(self, inputs: List[str]) -> List[str]: """异步批量执行 Agent""" tasks = [self.run_async(inp) for inp in inputs] results = await asyncio.gather(*tasks) return results

条件执行（Conditional）

根据不同的条件选择不同的执行路径，实现更复杂的工作流。

def run_conditional(self, user_input: str, condition: Callable) -> str: """条件执行""" # 根据条件选择执行策略 if condition(user_input): # 路径 A：使用高级 Agent agent = self.advanced_agent else: # 路径 B：使用简单 Agent agent = self.simple_agent return agent.run(user_input)

详解：condition 是什么？怎么定义？

condition 的类型是 Callable，即可调用对象—— 通俗说就是一个函数。它接收 user_input（用户输入的字符串）， 返回 True 或 False，决定走哪条路径。 run_conditional 本身不关心条件是什么，它只负责调用这个函数看结果。

核心概念：condition 是调用方传入的函数，不是在 run_conditional 内部定义的
这是一种叫高阶函数的设计模式——函数 A 接收函数 B 作为参数， 在合适的时候调用它。Python 中 sorted(key=...)、 filter(函数, 列表) 都是这个模式。

# ━━━━━━━━━━━━ 定义 condition：就是一个普通函数 ━━━━━━━━━━━━ from typing import Callable # 示例 1：根据输入长度判断 # 长问题 → 用高级 Agent（更聪明但更贵更慢） # 短问题 → 用简单 Agent（快但能力有限） def is_complex_question(user_input: str) -> bool: # 超过 50 个字 → 复杂问题 return len(user_input) > 50 # 示例 2：根据关键词判断 # 涉及代码/分析/研究 → 高级 Agent # 简单问答 → 简单 Agent def needs_advanced_agent(user_input: str) -> bool: keywords = ["分析", "研究", "代码", "对比", "报告"] return any(kw in user_input for kw in keywords) # 示例 3：用 lambda 一行搞定 condition = lambda text: len(text) > 50
# ━━━━━━━━━━━━ 调用 run_conditional ━━━━━━━━━━━━ # 把定义好的 condition 函数作为参数传进去 result = router.run_conditional( "请分析 2024 年中国新能源汽车市场的发展趋势", # user_input needs_advanced_agent # condition（传函数名，不加括号！） ) # needs_advanced_agent("...分析...") → True → 走高级 Agent # ━━━━━━━━━━━━ 执行过程展开 ━━━━━━━━━━━━ def run_conditional(self, user_input, condition): # ↓ user_input = "请分析 2024 年..." # ↓ condition = needs_advanced_agent（函数本身） if condition(user_input): # ↑ 等价于 needs_advanced_agent("请分析 2024 年...") # ↑ "分析" in user_input → True agent = self.advanced_agent # ← 走路径 A else: agent = self.simple_agent # （没走到这） return agent.run(user_input)

易错点：传函数名还是传函数调用？

写法含义对错
condition=needs_advanced_agent 把函数本身传进去，run_conditional 内部再调用它 ✅ 正确
condition=needs_advanced_agent("xxx") 在传参时就立刻执行了，传入的是 True/False 值，不是函数 ❌ 错误
# ✅ 正确：传函数本身（不加括号） router.run_conditional(user_input, needs_advanced_agent) # run_conditional 内部：condition(user_input) → 此时才调用 # ❌ 错误：加了括号 = 立刻执行了，传的是结果不是函数 router.run_conditional(user_input, needs_advanced_agent(user_input)) # 这里 needs_advanced_agent(user_input) 立刻返回 True # 实际传入的是 run_conditional(user_input, True) # 内部 True(user_input) → 报错！True 不是可调用的

为什么要把 condition 设计成参数，而不是写死在 run_conditional 里？
因为不同的调用场景需要不同的判断逻辑。 把 condition 抽象成参数，调用方可以灵活决定何时用高级 Agent， 而不需要修改 run_conditional 的代码。这就是"开闭原则"—— 对扩展开放（传入不同 condition），对修改关闭（不改原方法）。

最佳实践

1. 设置合理的最大步数

设置 max_steps 参数可以防止 Agent 陷入无限循环。通常 10-20 步足以完成大多数任务。

2. 完整的日志记录

记录每一步的执行过程：LLM 输入、输出、工具调用、结果等。这对调试和优化至关重要。

3. 超时控制

为 LLM 调用和工具执行设置超时时间，防止长时间等待。

4. 消息历史管理

对于长时间运行的 Agent，消息历史可能变得很长，导致 LLM 调用成本增加。考虑实现消息压缩或总结机制。

示例： 完整的 Agent 执行器实现
class RobustAgentExecutor: def __init__(self, llm, tools, max_steps=10): self.llm = llm self.tools = tools self.max_steps = max_steps def run(self, user_input: str) -> str: state = ExecutionState( messages=[HumanMessage(content=user_input)] ) while state.is_running(): try: # LLM 推理 response = self.llm.invoke(state.messages) state.add_message(response) # 解析动作 action = self.parse_action(response) # 处理动作 if action.type == "finish": state.status = "completed" break result = self.execute_tool(action) state.add_message( ToolMessage(content=result) ) state.step += 1 if state.step >= self.max_steps: state.status = "max_steps_reached" break except Exception as e: state.error_info = {"error": str(e)} state.status = "failed" break return state.get_final_output()

最佳实践详解

1. 系统设计原则

设计 Agent 执行引擎时应遵循以下原则：

可靠性第一： 优先考虑系统的稳定性和可靠性，而不是功能的复杂性。
可观测性： 提供完整的日志、追踪和监控能力，便于调试和优化。
可扩展性： 设计应该支持轻松添加新的工具、技能和执行模式。
性能优先： 考虑缓存、并发和异步执行来提高性能。

2. 常见陷阱与解决方案

陷阱 问题 解决方案
无限循环 Agent 无法决定完成，陷入循环 设置 max_steps，添加循环检测
消息爆炸 消息历史过长，导致成本增加 实现消息压缩和总结机制
工具失败 工具执行失败导致整个流程中断 实现重试和降级策略
超时问题 LLM 或工具调用超时 设置合理的超时时间和重试机制
内存泄漏 长时间运行导致内存占用过高 定期清理过期数据，使用流式处理

3. 测试策略

为了确保 Agent 执行引擎的质量，应该实施全面的测试策略：

单元测试： 测试各个组件的功能，如状态管理、错误处理等。
集成测试： 测试完整的执行流程，包括 LLM 调用和工具执行。
压力测试： 测试系统在高负载下的表现，如并发执行、长时间运行等。
端到端测试： 测试完整的用户场景，确保系统能够正确处理各种情况。

Agent 执行引擎是 DeerFlow 的核心，它通过一个精心设计的循环来驱动 Agent 完成任务。本章介绍了执行引擎的完整设计和实现，包括执行循环、状态管理、决策逻辑、错误处理、性能优化、监控调试和高级模式等多个方面。
