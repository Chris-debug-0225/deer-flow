Agent执行引擎的设计与实现

DeerFlow 学习系列 - 第 10 章

本章要追踪的不是抽象 while-loop

DeerFlow 的执行引擎是一条真实的 lead agent runtime 主线：agent 如何被构建、middlewares 如何接入、ThreadState 如何扩展状态、Gateway 如何拥有 run/stream 生命周期，以及这些能力怎样共同维持一次长任务执行。

这项能力在 DeerFlow 中解决什么问题

执行引擎负责把一个“已经装配好能力面”的 DeerFlow agent，推进成一次真正的运行过程。它要解决的问题包括：如何构建 lead agent、如何把工具和中间件连接起来、如何在一次 run 中维护 thread state、如何把 token 流和状态流回传给调用方，以及如何在长任务中插入 memory、todo、summarization、skill activation、tool error recovery 等机制。

所以 DeerFlow 的执行引擎不是“LLM 调工具直到结束”的最小循环，而是“agent 构建 + middleware orchestration + state ownership + run streaming”的组合系统。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/packages/harness/deerflow/agents/lead_agent/agent.py lead agent 构建、中间件装配、runtime 入口 这是 DeerFlow 执行主线的核心文件
backend/packages/harness/deerflow/agents/factory.py 模型工厂，给执行引擎提供实际可用的 LLM 执行引擎并不自己发明模型对象
backend/packages/harness/deerflow/agents/thread_state.py 扩展 LangGraph AgentState，承载 DeerFlow 运行态字段 解释状态为何不是只有 messages
backend/packages/harness/deerflow/agents/middlewares/ 一系列 runtime 中间件，如 memory、todo、skill activation、loop detection 等 DeerFlow 的“执行逻辑”大量体现于 middleware，而非单函数主循环
backend/packages/harness/deerflow/runtime/runs/worker.py Gateway 路径下驱动 agent.astream() 的 worker run lifecycle 和 streaming 在这里落地
backend/app/gateway/routers/thread_runs.py 对外暴露线程运行、流式事件等 Gateway 接口 解释 DeerFlow 如何把内部 runtime 变成公开 API
backend/docs/STREAMING.md 区分 Gateway 路径与 DeerFlowClient 路径的流式语义 避免把两条流式链路混成一个抽象实现

这些文件在执行引擎里不是并排关系，而是接力关系。backend/app/gateway/routers/thread_runs.py 负责把外部请求转换成 DeerFlow 的 run；backend/packages/harness/deerflow/runtime/runs/worker.py 真正拥有这次 run 的生命周期，安装 runtime context、callbacks、stream bridge 和 run journal；然后它调用 backend/packages/harness/deerflow/agents/lead_agent/agent.py 去构建 agent。agent.py 自己并不“手写主循环”，而是先向 factory.py 要模型，再把 thread_state.py、middlewares 和 tools 全部绑进 LangGraph agent graph 里。

换句话说，执行引擎真正值得读的不是某个 while-loop，而是“谁负责构建 graph、谁负责拥有 run、谁负责把中间件插进去”。这正是上面这些文件组合起来承担的职责。

真实运行链路或数据链路

默认 Gateway 路径下，一次典型运行可以压缩成下面这条链：

execution flow :: run 请求进入后怎样跑到 agent
1"backend/app/gateway/routers/thread_runs.py"
2 -> "runtime/runs/worker.py"
3 -> "backend/packages/harness/deerflow/agents/lead_agent/agent.py"
4 -> "build middlewares + bind tools + ThreadState"
5 -> "agent.astream(stream_mode=['values','messages','custom'])"
6 -> "stream bridge / SSE"
7 -> "caller receives events"

把这条链再拆成“源码里的动作”，会看到很清楚的层次。thread_runs.py 先创建 RunRecord，并通过 service 层触发后台任务；runtime/runs/worker.py 进入 run_agent(...) 后，会先设置 run status、抓取 pre-run checkpoint、向 stream bridge 发送 metadata 事件，再把 thread_id、run_id、app_config 写进 runtime context；只有这些运行态基础设施就位后，才会调用 agent_factory 创建真正的 agent graph。这样后面的 middleware、tool 和 tracing 才能在统一上下文里工作。

而 backend/packages/harness/deerflow/agents/lead_agent/agent.py 的重点，是按顺序把 Clarification、DynamicContext、Sandbox、Todo、Title、Memory、Summarization、TokenUsage、SubagentLimit 等中间件装配进去，再把模型和工具绑定给 create_agent(...)。也就是说，DeerFlow 的“执行语义”很多不是写在 worker 里，而是写在 middleware append 顺序里。

在这条链里，backend/packages/harness/deerflow/agents/factory.py 先提供模型，backend/packages/harness/deerflow/agents/thread_state.py 负责定义扩展状态，而 backend/packages/harness/deerflow/agents/middlewares/ 目录下的中间件则逐步插入动态上下文、skills、memory、todo、summarization、subagent limit、loop detection 等 DeerFlow 运行语义。

  • TitleMiddleware 和 MemoryMiddleware 靠后，说明它们更像“本轮回答之后的整理工作”，而不是一开始就改变工具面。
  • ClarificationMiddleware 被固定放在最后，意味着任何需要“先问清楚再干活”的动作，都会在整条链收尾阶段拦截并中断执行。

核心源码 2：Gateway worker 怎样把 run 变成可执行 graph

节选自 backend/packages/harness/deerflow/runtime/runs/worker.py。这一段说明 run ownership 不在 router，而在 worker。

  • 第一步不是立刻调模型，而是先把 run 状态标记为 running，这样取消、join、状态查询才有一致依据。
  • bridge.publish(..., "metadata", ...) 会先把 run_id 和 thread_id 发给前端流，这就是为什么前端一开始就能知道自己连的是哪次运行。
  • runtime_ctx 是 DeerFlow 的关键胶水层。后面中间件和工具能拿到 thread_id、run_id、app_config，就是因为 worker 先把它们塞进了 runtime context。
  • config["configurable"]["__pregel_runtime"] = runtime 这一句把 DeerFlow 自己准备好的运行时塞回 LangGraph，这样 graph 内部执行时就能读到同一份上下文。

  • merge_artifacts(...) 和 merge_promoted(...) 不是普通 helper，而是 graph state reducer。多个节点同时更新这些字段时，最后怎样合并，完全由这里定义。

  • ThreadState 最后一段就是执行引擎的数据骨架。前面各个 middleware 能互相串起来，靠的就是它们都在读写这同一份扩展状态。

配置面与使用方式

执行引擎并不只受模型配置影响，它还受多个运行时开关控制：

  • summarization 配置决定是否挂入总结中间件。
  • plan mode 会影响 todo middleware 是否介入。
  • skills、memory、sandbox、subagent 等能力是否可用，会反映到 middleware 和 tool 组合上。
  • Gateway run 配置和 worker 生命周期影响 SSE streaming、取消、重连和 run ownership。

这意味着读者不能把 DeerFlow 执行引擎理解成固定流程图。它更像一个“按配置裁剪的执行管线”，而 backend/packages/harness/deerflow/agents/lead_agent/agent.py 就是这条管线的装配厂。

关键实现细节与权衡

DeerFlow 在执行层有几项非常核心的设计取舍：

  • 大量运行语义通过 middleware 实现，而不是挤进一个巨型主循环，这样更容易组合能力，也更便于局部替换。
  • ThreadState 扩展了 DeerFlow 自身需要的运行态信息，说明 DeerFlow 的状态不只是 LangGraph 的原始消息历史。
  • Gateway 通过 runtime/runs/worker.py 拥有 run/stream 生命周期，这让取消、桥接、SSE fan-out 有明确归属，但也带来单进程状态边界。
  • 流式实现明确区分 values、messages、custom 三类事件，不把它们压平成一种“详细程度不同的流”。

源码里还能看到一个很关键的设计选择：tracing callback 不直接挂在每个模型对象上，而是由 graph 根调用统一装配，agent.py 里甚至专门写了 invariant 注释来防止重复挂 tracing。这个细节说明 DeerFlow 的执行引擎不只是关心“能不能跑”，还关心一次 run 的观测边界应该落在哪一层。

要点： DeerFlow 的执行引擎设计重点是“可组合的运行语义”和“可拥有的运行生命周期”，不是最少代码量。

当前限制、边界与非目标

  • 执行引擎不是一个通用分布式调度器。默认实现围绕 Gateway 进程内 run ownership 展开。
  • 多 worker 横向扩展当前不是默认安全路径，因为 run state、stream bridge、取消和部分渠道语义会被破坏。
  • 本章不展开每个 middleware 的所有细节；重点是它们如何进入执行主线，而不是逐一替代源码阅读。
  • DeerFlow 也不是把所有流式调用完全统一成同一个接口层。backend/docs/STREAMING.md 明确说明 Gateway 与 DeerFlowClient 路径不能强行合并。

它与 DeerFlow 其他能力的关系

执行引擎是 DeerFlow 各能力的汇合点：

  • 第 2 章的模型与工具装配，会在这里真正成为可运行 agent。
  • 第 11 章的 skills 不只影响 prompt，还会通过 skill activation 与 tool policy 改变执行行为。
  • memory、sandbox、sub-agents、view-image、todo、summarization 都是通过工具或 middleware 进入这条主线。
  • Gateway API 章节和 streaming 文档共同解释了为什么 DeerFlow 的运行所有权不能只看 agent.py，还要看 backend/app/gateway/routers/thread_runs.py 与 backend/packages/harness/deerflow/runtime/runs/worker.py。
建议把 backend/packages/harness/deerflow/agents/lead_agent/agent.py、backend/packages/harness/deerflow/agents/thread_state.py、backend/packages/harness/deerflow/agents/middlewares/、backend/packages/harness/deerflow/runtime/runs/worker.py 和 backend/docs/STREAMING.md 放在一起对读，这样最容易看到 DeerFlow 执行引擎的真实边界。
