多渠道集成：从IM到API

DeerFlow 学习系列 - 第 22 章

本章讲的是 DeerFlow 怎样把 IM 接入收进同一运行时

DeerFlow 的渠道接入不是“每个平台自己跑一套 agent 逻辑”。当前仓库的核心设计是：IM/channel worker 把外部消息转换成 DeerFlow 线程语义，再复用 Gateway runtime、认证与 runs API。这样浏览器会话和 IM 会话虽然入口不同，但共享的是同一条 DeerFlow 执行主线。

这项能力在 DeerFlow 中解决什么问题

DeerFlow 既要支持浏览器聊天，也要支持飞书等 IM 接入。如果每个渠道都直接内嵌一套 agent 执行逻辑，就会很快出现线程不一致、认证不一致、uploads / artifacts / streaming 行为分叉的问题。channels 集成要解决的就是这个复用问题。

因此 DeerFlow 当前实现选择让 channel worker 做协议适配，而不是做独立 runtime。外部平台消息进入后，会被归一化成 DeerFlow 的 thread、run、message 和 attachment 语义，再通过 Gateway 继续执行。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/app/channels/manager.py 消费 inbound 消息、解析 thread、调用 Gateway run API、处理 streaming 这是 IM 接入复用 DeerFlow runtime 的主控文件
backend/app/channels/base.py 渠道抽象基类、消息收发约定、附件发送钩子 说明 DeerFlow 对“channel”本身的最小接口建模

backend/app/gateway/routers/channel_connections.py 浏览器侧 provider 列表、连接、运行时凭据管理与绑定关系 说明用户连接面也统一经过 Gateway

这里真正承担“把平台消息翻译成 DeerFlow 语义”的核心，是 backend/app/channels/manager.py。它不只负责调 HTTP API，还要把 inbound message、附件、线程映射、session override、slash skill 命令和流式回复能力统一起来。base.py 和平台文件如 feishu.py 主要解决的是“平台协议怎么接”；manager 才解决“接进来以后怎样进入 DeerFlow 主运行时”。

源码里这条链还多了几个特别关键的翻译步骤。backend/app/channels/manager.py 会先把外部消息转成 DeerFlow 可接受的 message payload，再根据渠道能力和当前会话状态决定是否走 runs.wait 或 runs.stream；如果有附件，还会先把平台文件读出来，再转成 DeerFlow uploads/thread 语义，保证后续 agent 在 sandbox 里看到的是统一虚拟路径，而不是某个平台的临时 URL。这样 channel 输入才真正融进同一运行时，而不是挂在外面当旁注。

backend/app/channels/manager.py 里最关键的点，不是“怎么发 HTTP 请求”，而是它主动复用了 DeerFlow 的 thread 语义、run 配置、stream mode、skills slash command 解析和附件处理。默认情况下它会走 runs.wait；只有像 Feishu、WeCom 这样声明支持流式回显的渠道，才会走 runs.stream。也正因为如此，channel 消息不是旁路执行，而是进入同一个 Gateway 运行面。

核心源码 2：ChannelManager 怎样从 runs.wait / runs.stream 的结果里提取可回显文本

节选自 backend/app/channels/manager.py。这一段最适合解释“为什么 IM 渠道不是简单照抄 Gateway 响应”。

app/channels/manager.py :: response extraction / stream text merge
1# result: runs.wait 或 runs.stream 聚合后的结果结构，职责是从中提炼出最适合回给 IM 用户的最终文本
2def _extract_response_text(result: dict | list) -> str:
3 # 先把 wait / stream 两种返回形态统一成 messages 列表，后面的抽取逻辑就不用分叉两遍
4 if isinstance(result, list):
3 messages = result
4 elif isinstance(result, dict):
5 messages = result.get("messages", [])
6 else:
7 return ""
8
9 # 倒序找最后一个真正可展示给 IM 用户看的结果，但遇到本轮最后一条 human 就停
10 for msg in reversed(messages):
11 if not isinstance(msg, dict):
12 continue
13
14 msg_type = msg.get("type")
15 if msg_type == "human":
16 if _is_hidden_human_control_message(msg):
17 continue
18 break
19
20 # clarification interrupt 不是普通 AI 文本，而是 ask_clarification 工具消息
21 if msg_type == "tool" and msg.get("name") == "ask_clarification":
22 content = msg.get("content", "")
23 if isinstance(content, str) and content:
24 return content
25
26 # clarification tool message 和普通 AI 消息分开处理，避免把“等待补充信息”误读成空响应
27 if msg_type == "ai":
27 content = msg.get("content", "")
28 if isinstance(content, str) and content:
29 return content
30 if isinstance(content, list):
31 parts = []
32 for block in content:
33 if isinstance(block, dict) and block.get("type") == "text":
34 parts.append(block.get("text", ""))
35 elif isinstance(block, str):
36 parts.append(block)
37 text = "".join(parts)
38 if text:
39 return text
40 return ""
41
42# existing / chunk: 已经推给渠道的前缀文本和这次新来的流式片段，职责是尽量无重复地拼出当前完整回复
43def _merge_stream_text(existing: str, chunk: str) -> str:
43 if not chunk:
44 return existing
45 if not existing or chunk == existing:
46 return chunk or existing
47 if chunk.startswith(existing):
48 return chunk
49 if existing.endswith(chunk):
50 return existing
51 return existing + chunk
  • 这段代码很好地说明了：channel manager 不是拿到 Gateway 返回值就直接原样转发给 IM，而是要先把 DeerFlow 内部结果重新解释成“适合人看的文本”。
  • _extract_response_text(...) 会倒序扫描本轮消息，但遇到最后一条真实 human 消息就停，避免把上一轮的 AI 回复错拿来当本轮结果。
  • clarification 也是专门处理的，因为 DeerFlow 的“请先澄清”有时不是普通 AI 文本，而是 `ask_clarification` 工具消息。
  • 对流式路径来说，_merge_stream_text(...) 也很关键。它要同时兼容“增量 token”流和“累计全文”流，避免重复拼接、倒退覆盖或丢字。
  • 也就是说，IM 渠道适配不仅要会“发请求”，还要会“理解 DeerFlow 返回的消息结构”，然后再把它压成平台能展示的一条回复。

核心源码 3：ChannelManager 真正调用 Gateway 时，已经在复用 thread/run/auth 语义

节选自 backend/app/channels/manager.py。这段最适合解释“为什么 IM 渠道不是单独跑 agent，而是进同一个 Gateway runtime”。

  • `_thread_channel_metadata(...)` 很关键，因为它把外部平台的 chat、topic、thread 等身份信息翻译成 DeerFlow thread metadata，这样后续线程检索和会话复用才有依据。
  • `_owner_headers(...)` 直接表明 channels 调 Gateway 不是匿名内部调用，而是会带上 internal auth headers，把 owner_user_id 传回 Gateway 的认证体系里。
  • `_safe_user_id_for_run(...)` 则说明渠道层连用户目录名都不敢直接信任原始外部 ID，而是要先走 DeerFlow 自己的路径安全转换。
  • 这三步合在一起，实际上就是把“平台消息”翻译成 DeerFlow 的 user/thread/run 三元语义，再进入统一 runtime。
认证边界要一起看： 根据 backend/docs/AUTH_DESIGN.md，IM channel worker 不持有浏览器 cookie，而是通过内部 token 与匹配的 CSRF 头进入 Gateway。默认情况下，这些请求落在 internal / default 用户桶语义上。

配置面与使用方式

DeerFlow 的 channels 集成至少有两层配置面：

  • backend/app/gateway/routers/channel_connections.py 提供浏览器可见的 provider 列表、connect code、运行时凭据录入和已绑定连接查询。
  • backend/app/gateway/routers/channels.py 提供实例级 channel service 状态与重启能力，更偏向运维控制面。
  • backend/app/channels/base.py 和具体平台实现如 backend/app/channels/feishu.py 负责平台协议适配，但不重复发明 DeerFlow 的线程与运行配置。
  • backend/app/channels/manager.py 还负责把外部附件同步成 DeerFlow 线程可见资产；这些文件随后会进入 uploads / artifact / sandbox 语义，而不是永远停留在平台私有 URL 上。
  • backend/docs/AUTH_DESIGN.md 明确了内部调用必须复用 Gateway 的 auth / CSRF 设计，而不是偷偷绕过 HTTP 入口。

因此，从用户视角看是“连上一个 IM 平台”，从仓库视角看其实是“让一个外部渠道获得进入 Gateway runtime 的合法方式”。

关键实现细节与权衡

  • backend/app/channels/manager.py 直接面向 Gateway API 工作，这让 browser 与 IM 共享同一运行主线；代价是 channels 对 Gateway wire contract 有明确耦合。
  • backend/app/channels/base.py 保持抽象最小化，只定义生命周期与消息收发契约，避免把所有平台差异硬塞进统一大接口。
  • backend/app/channels/feishu.py 自己处理平台级 websocket、卡片、文件资源，但真正的 agent runtime 仍委托给 Gateway，这减少了逻辑分叉。
  • backend/app/gateway/routers/channel_connections.py 把浏览器侧连接与实例级运行时凭据拆开，区分“谁绑定了渠道”和“节点上能否真正启动该渠道”。
  • 附件同步是这层设计的一个关键细节：channel 先下载或读取平台文件，再通过 DeerFlow uploads/thread 语义变成 sandbox 可见路径，这样后续 agent 才能在统一路径契约里处理它们。
  • 并不是所有渠道都走流式路径。backend/app/channels/manager.py 会按渠道能力分流，只有支持 streaming 的 provider 才会进入 runs.stream，其余渠道保持 runs.wait 这条更稳的默认主路径。
  • backend/docs/AUTH_DESIGN.md 记录了 internal token + CSRF 的设计，说明 channels 复用 Gateway runtime 并不是“内部代码所以可以跳过认证”。

从实现方式上说，这也是一种“协议适配优先”的设计。DeerFlow 宁可让 channel manager 多理解一些 Gateway contract、线程语义和附件流程，也不愿意在每个平台 SDK 里再复制一套 agent orchestration。对维护者来说，这样的耦合更集中，也更容易排查入口差异。

核心取舍： DeerFlow 选择把 channels 设计成 Gateway runtime 的外部入口适配层，而不是独立 agent 宿主。这样线程、权限和文件语义更统一，但渠道层必须理解更多 DeerFlow 运行契约。

当前限制、边界与非目标

  • channels 当前默认进入 internal / default 用户桶语义，这适合平台级 bot，但不等于“每个外部 IM 用户都天然映射成独立 DeerFlow 用户”。
  • 不同平台的 streaming 能力不完全一致，backend/app/channels/manager.py 里也有按平台能力降级的逻辑。
  • 如果是 Docker Compose 或分离部署，channels 能否正常上传、读取附件和访问 Gateway，取决于运行时网络、共享存储与 thread 文件目录是否按 DeerFlow 预期打通。
  • 渠道实现不是统一到零差异的“万能 connector”；具体平台文件如 backend/app/channels/feishu.py 仍保留大量平台细节。
  • 本章不覆盖所有 provider 的逐项接入步骤；重点是它们怎样复用 Gateway runtime、auth 与 thread 语义。

它与 DeerFlow 其他能力的关系

channels 集成是 DeerFlow 运行时复用能力的直接体现：

  • 它和第 15 章 Gateway API 相连，因为 channel manager 的目标就是复用 Gateway，而不是绕过它。
  • 它和第 20 章上下文工程相连，因为 IM 消息、附件、slash skills 和 clarification 都会进入同一上下文组装流程。
  • 它和认证设计相连，因为 backend/docs/AUTH_DESIGN.md 里专门描述了 internal channel 调用的身份边界。
  • 它和 streaming 相连，因为平台是否支持逐步回显，会影响 backend/app/channels/manager.py 采用 stream 还是 wait 路径。
建议把 backend/app/channels/manager.py、backend/app/channels/base.py、backend/app/channels/feishu.py、backend/app/gateway/routers/channels.py、backend/app/gateway/routers/channel_connections.py 和 backend/docs/AUTH_DESIGN.md 放在一起读。这样最容易看清 DeerFlow 如何把 IM/channel 接入收编到同一 Gateway runtime 下。
