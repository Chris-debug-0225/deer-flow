真实运行链路或数据链路

从零搭完整 Agent 系统的主线流程图 先把外部接口和运行主线打通，再往里面加能力层
01
先明确系统要接什么输入入口

无论是浏览器、IM 还是其他客户端，第一步都是把入口统一起来。

public client / browser / IM
02
用 Gateway 统一承接外部 API 面

这一层负责把不同来源的请求收进来，并转换成系统内部统一的调用协议。

Gateway API surface
03
把线程、认证和运行生命周期先定清楚

如果 thread、user、auth、run 这些主语不统一，后面 files、memory、channels 都会越做越乱。

authenticated thread + run lifecycle
04
再把 Agent 运行时嵌进这条主线

到了这里，模型、工具和中间件才能在统一上下文里运行，而不是各自为战。

embedded agent runtime
05
围绕主线继续挂沙箱、上传、产物和记忆

这些都应该附着在统一 thread/user 语义上，而不是零散外挂。

sandbox / uploads / artifacts / memory
06
最后再把结果送回 Gateway 或嵌入式客户端

这样系统才形成完整闭环，外部入口和内部运行时之间不会断层。

Gateway / embedded client

如果把 DeerFlow 当作一个“从零构建系统”的样本，最值得学的不是单个算法，而是这条主线：先定义好外部接口与内部运行时的关系，再让文件、记忆、channels、skills、subagents 都围绕同一 thread/user 语义工作。

核心源码：从零做系统时，为什么认证中间件必须是“默认拒绝”

节选自 backend/app/gateway/auth_middleware.py。如果你想抄走 DeerFlow 最有价值的系统设计经验，这段比很多概念图都重要。

  • 第 1 到 5 行先定义 public path 判断，这是“默认拒绝，只放行少数公开入口”的核心思想。
  • 第 11 到 20 行说明 DeerFlow 一开始就考虑了浏览器用户之外的内部调用，例如 channels worker 用 internal token 走系统内调用链。
  • 第 21 到 31 行是最值得借鉴的部分: 不是只要有 cookie 就算登录，而是严格校验 token；没有 token 或 token 无效就立即在中间件层返回 401。
  • 第 33 到 40 行把用户同时写进 request.state 和 runtime contextvar，这一步非常关键，因为后面的 repository、thread path、memory 隔离都要靠它拿到当前用户。
  • 如果你从零做 Agent 系统却忽略了这一层，后面所有 thread、文件、memory、artifact 的“多用户隔离”都会变成补丁式修复。

  • 也就是说，多入口系统的关键不是为每个平台各写一套业务执行器，而是把平台差异消化在入口层，把运行时协议压缩成统一模型。
  • 同步渠道走 `runs.wait`，支持流式的渠道走 `runs.stream`，但两边都共享线程 ID、owner headers、clarification 语义和 artifact 回传逻辑，这就是 DeerFlow 当前系统架构的骨架。
  • 如果你要从零做类似系统，这一页里最值得抄的不是具体平台适配代码，而是这种“入口归一化 -> 统一执行协议 -> 出口再适配渠道”的结构。

配置面与使用方式

  • backend/docs/ARCHITECTURE.md 帮你建立默认运行时主路径心智模型：Gateway 是承载体，不是无状态代理。
  • backend/docs/API.md 展示如何同时保留兼容路径与本地原生 API，而不把内部结构暴露给外部调用方。
  • backend/docs/AUTH_DESIGN.md 提醒你多用户、internal calls、default user bucket 和 CSRF 不能事后再补。

关键实现细节与权衡

  • DeerFlow 选择 Gateway embedded runtime，牺牲了一些“服务拆得更漂亮”的表象，换来更统一的 thread/run/file/auth 语义。
  • 同时保留 Gateway 流式路径和 DeerFlowClient 路径看似重复，但它忠实反映了 HTTP 消费者和嵌入式 Python 消费者的根本差异。
  • 把认证、路径契约和用户隔离前置，会让前期设计更重，但能避免后期为多用户、文件越界和 channel 内部调用付出更高改造成本。

当前限制、边界与非目标

  • 本章不是让你复制 DeerFlow 的全部代码，而是理解它当前架构里最有复用价值的边界设计。
  • DeerFlow 的实现仍带有当前仓库的历史与部署权衡，所以“从零构建”不等于逐字照搬每个模块。
  • 本章不深入某个具体子系统的实现细节；对应细节请回到前面相关章节。
