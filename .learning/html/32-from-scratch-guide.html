这项能力在 DeerFlow 中解决什么问题

很多人会把“从零构建 Agent 系统”理解成列一份通用组件清单，但 DeerFlow 给出的更有价值答案是：哪些边界必须一开始就想清楚，否则后面会不断返工。比如 API 是否只是代理、认证和用户隔离是否内建、文件路径是否有统一虚拟契约、运行时是否支持浏览器和嵌入式双消费面。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/docs/ARCHITECTURE.md 整体系统图、Gateway embedded runtime、部署主路径 这是 DeerFlow 级别的系统设计总览

真实运行链路或数据链路

public client / browser / IM -> Gateway API surface -> authenticated thread + run lifecycle -> embedded agent runtime -> sandbox / uploads / artifacts / memory -> results back through Gateway or embedded client

如果把 DeerFlow 当作一个“从零构建系统”的样本，最值得学的不是单个算法，而是这条主线：先定义好外部接口与内部运行时的关系，再让文件、记忆、channels、skills、subagents 都围绕同一 thread/user 语义工作。

配置面与使用方式

  • backend/docs/ARCHITECTURE.md 帮你建立默认运行时主路径心智模型：Gateway 是承载体，不是无状态代理。
  • backend/docs/API.md 展示如何同时保留兼容路径与本地原生 API，而不把内部结构暴露给外部调用方。
  • backend/docs/AUTH_DESIGN.md 提醒你多用户、internal calls、default user bucket 和 CSRF 不能事后再补。

关键实现细节与权衡

  • DeerFlow 选择 Gateway embedded runtime，牺牲了一些“服务拆得更漂亮”的表象，换来更统一的 thread/run/file/auth 语义。
  • 同时保留 Gateway 流式路径和 DeerFlowClient 路径看似重复，但它忠实反映了 HTTP 消费者和嵌入式 Python 消费者的根本差异。
  • 把认证、路径契约和用户隔离前置，会让前期设计更重，但能避免后期为多用户、文件越界和 channel 内部调用付出更高改造成本。

当前限制、边界与非目标

  • 本章不是让你复制 DeerFlow 的全部代码，而是理解它当前架构里最有复用价值的边界设计。
  • DeerFlow 的实现仍带有当前仓库的历史与部署权衡，所以“从零构建”不等于逐字照搬每个模块。
  • 本章不深入某个具体子系统的实现细节；对应细节请回到前面相关章节。
