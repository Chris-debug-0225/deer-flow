自定义Skills完全指南

DeerFlow 学习系列 - 第 30 章

本章讲的是用户如何在 DeerFlow 里定义、安装和维护自己的 skill

这一章不是解释整个 skills 系统如何工作，而是贴着 DeerFlow 当前仓库，讲“你要自己做一个 skill 时，目录长什么样、归档怎么安装、Gateway 怎么管理、为什么它会影响 agent 行为，以及哪些资源文件会被保留给 skill 使用”。

这项能力在 DeerFlow 中解决什么问题

当你希望 DeerFlow 的 agent 形成某种稳定工作法，例如固定的排障流程、特定格式的文档生成方式、某类外部系统的使用约定，最合适的扩展点通常不是再加一个 tool，而是做一个自定义 skill。skill 更适合承载工作说明、模板、参考资料和配套脚本资源。

分工边界： 本章讲用户如何定义和构建自定义 skills；11-skills-system.html 讲整个 skills 系统如何运作。

这项能力在代码库中的位置

路径 职责 为什么要看
backend/packages/harness/deerflow/skills/ skills 类型、校验、存储、权限与 slash 解析 这是自定义 skill 真正落地的主目录
backend/packages/harness/deerflow/skills/installer.py 解包 .skill 归档、安全扫描、路径检查与安装 说明 DeerFlow 接受的不是任意 ZIP，而是受控 skill 包
backend/app/gateway/routers/skills.py 列出、安装、编辑、删除、回滚自定义 skills 这是你通过 Gateway 管理 skill 的入口
backend/packages/harness/deerflow/tools/builtins/tool_search.py 延迟工具发现 很多 skill 会教 agent 什么时候去搜索工具，而不是自己替代工具

真实运行链路或数据链路

author creates skill folder -> SKILL.md + optional references/templates/scripts -> package as .skill archive -> backend/app/gateway/routers/skills.py -> backend/packages/harness/deerflow/skills/installer.py -> safe_extract + security scan + storage install -> skill enabled in storage / config -> lead agent prompt sees enabled skill in later runs

如果是浏览器或脚本触发安装，自定义 skill 往往先出现在某个 thread 的 artifact 或输出目录里，再由 backend/app/gateway/routers/skills.py 通过 thread 虚拟路径找到归档并安装。安装后，skills storage 会把它归入 DeerFlow 的可管理 skills 集合，后续 run 才能看到它的 skills 元数据并在需要时再去读取具体内容。

这条链非常重要，因为它说明“写一个 skill 文件”并不等于“agent 立刻可用”。真正生效之前，还要经过归档、扫描、安装、启用和 prompt 刷新这几步。

配置面与使用方式

  • 自定义 skill 至少要包含 SKILL.md，其余支持文件通常落在 references/、templates/ 或 scripts/ 下。
  • backend/packages/harness/deerflow/skills/installer.py 会拒绝路径穿越、符号链接和不安全归档成员，所以 skill 包结构必须干净且可扫描。
  • backend/app/gateway/routers/skills.py 提供安装、启停、编辑、查看历史和回滚能力，这意味着自定义 skill 不是“一次拷贝完就再也不管”的静态资源。
  • backend/packages/harness/deerflow/tools/builtins/tool_search.py 让 skill 可以把“先搜索某类 MCP 工具再执行”的工作法教给 agent，而不是把所有工具 schema 永远写死进 skill 文本。

关键实现细节与权衡

  • DeerFlow 选择 skill 目录 + 归档安装模型，而不是让用户直接往数据库里写一段提示词，这样能携带模板、参考资料和脚本。
  • backend/packages/harness/deerflow/skills/installer.py 的安全扫描会增加安装成本，但它是把自定义 skill 当成系统边界处理的必要代价。
  • Gateway 支持在线编辑和回滚自定义 skills，这对迭代很友好，但也意味着 skill 不是“写完就固定”的一次性产物。
  • skill 更适合提供工作法和资源，而不是直接代替 runtime orchestration；这能保持边界清晰，但也要求用户分清该用 skill、tool 还是 subagent。

当前限制、边界与非目标

  • 自定义 skill 不是任意插件执行环境。是否能携带脚本、脚本是否会在后续场景中被使用，都要服从 DeerFlow 当前的安全与工具边界。
  • 写好 SKILL.md 并不会自动给 agent 新增 API 权限；如果需要新工具，还要回到工具与配置层处理。
  • 本章不教你修改 skills 系统底层实现，那是第 11 章和源码层面的主题。
  • 自定义 skill 的价值主要体现在后续运行时“变得可见且可按需激活”，而不是安装动作本身。默认情况下进入提示面的是 skills 元数据与使用规则；完整 SKILL.md 通常要靠按需 read_file，或通过显式 slash 激活进入更直接的注入路径。

它与 DeerFlow 其他能力的关系

  • 它和第 11 章 skills 系统相连，因为第 11 章解释系统怎么运作，本章解释你如何往系统里放入自己的能力包。
  • 它和第 20 章上下文工程相连，因为启用中的自定义 skill 会进入后续 run 的可见 skills 提示面，而完整 skill 内容则通常按需读取或在显式激活时进入更直接的运行上下文。
  • 它和第 23 章高级模式相连，因为很多实用 skill 会指导 agent 何时使用 tool_search、何时委派任务、如何分阶段执行。
  • 它和 Gateway API 相连，因为安装、编辑、回滚和启停这些动作都通过 backend/app/gateway/routers/skills.py 完成。
建议把 backend/packages/harness/deerflow/skills/、backend/packages/harness/deerflow/skills/installer.py、backend/app/gateway/routers/skills.py 和 backend/packages/harness/deerflow/tools/builtins/tool_search.py 放在一起读。
