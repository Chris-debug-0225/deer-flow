默认主路径先记住

DeerFlow 的默认服务主路径不是“单独起 LangGraph Server，再额外挂一层 API”，而是 Gateway-embedded runtime。也就是说，请求通常先进入 Gateway，再在其中直接触发 agent runtime。

这项能力在 DeerFlow 中解决什么问题

这一章负责把 DeerFlow 作为一个“完整系统”讲清楚。读者需要知道 DeerFlow 不只是几个 agent 模块，而是一个以 Gateway-embedded runtime 为默认主路径的整合型 harness：它同时提供前端、HTTP Gateway、LangGraph-compatible routes、skills、uploads、artifacts、channels、sandbox、memory 和 embedded client。

如果没有这一层总图，后面各章会变成局部知识堆叠。读者可能知道某个工具、某个 middleware、某个 router 在做什么，却不知道它在 DeerFlow 全局中处于哪一层，最终还是无法建立真实心智模型。

路径 架构含义 阅读价值

如果你真的顺着源码读，会发现这五个锚点承担的是五种不同层级的职责。README.md 和 backend/docs/ARCHITECTURE.md 先定义“系统应该长什么样”；backend/app/gateway/app.py 负责把这个系统启动起来，挂上 routers、auth、CORS、CSRF 和 runtime 生命周期；backend/packages/harness/deerflow/client.py 则提供不经过 HTTP 的另一条接入方式；最后 backend/packages/harness/deerflow/agents/lead_agent/agent.py 才真正开始组装模型、工具和中间件，把一条请求变成可执行 agent。

所以这里不是“哪几个文件重要”的清单，而是一条分层阅读路线：先看谁接请求，再看谁建运行时，再看谁拼出 agent，最后才去追 memory、sandbox、skills 这些横切能力是怎样挂上去的。

默认 Web 路径下，DeerFlow 的流向大致是：

front proxy such as nginx
03
Gateway 应用接住这次请求

真正属于 DeerFlow 的默认 HTTP 入口，是 Gateway 这层 FastAPI 应用。

backend/app/gateway/app.py
04

runs、threads、skills、uploads、models 等能力，都会在这里被分发到各自入口。

Gateway routers
05
Lead Agent 运行时开始真正组装和执行

到了这里，模型、工具和上下文能力才会进入实际运行。

lead agent runtime
06
横切能力在运行中持续介入

sandbox、memory、skills、tools 和 subagents 都会在这一层参与同一轮执行。

sandbox / memory / skills / tools / subagents
07
结果再通过 Gateway 回到外部调用方

最终流式输出或普通响应，都会沿着 Gateway 这条主线返回用户。

streaming response back through Gateway

这里把 nginx 放进链路，是为了描述统一入口部署中的常见前置代理，不是说 DeerFlow 架构本体必须依赖它。真正属于 DeerFlow 本体的默认 HTTP 入口仍然是 backend/app/gateway/app.py。

把这条链再翻译成“代码里谁接谁”的版本，会更好懂：浏览器请求先被 backend/app/gateway/app.py 创建出来的 FastAPI 应用接住；具体进入哪个 router，由 app.include_router(...) 的挂载关系决定；run 类请求进一步落到 backend/app/gateway/routers/thread_runs.py 和 Gateway service 层，再交给 runtime worker；worker 安装好 run_id、thread_id、callbacks、stream bridge 等运行上下文后，才调用 backend/packages/harness/deerflow/agents/lead_agent/agent.py 产出的 agent graph。也就是说，Gateway 不只是“外壳”，它实际负责把请求改造成 DeerFlow 内部能运行的那种形态。

核心源码：为什么说 Gateway 真的是整个系统的统一入口

节选自 backend/app/gateway/app.py。总览页只要看懂这段，就能先建立 DeerFlow 的“入口层”心智模型。

gateway/app.py :: create_app()
1# Auth: reject unauthenticated requests to non-public paths
2app.add_middleware(AuthMiddleware)
3
4# CSRF: browser state-changing requests must pass the same gateway guard
5app.add_middleware(CSRFMiddleware)

8if cors_origins:

10
11# 下面开始把 DeerFlow 的所有外部能力面统一挂到同一个 app 上
12app.include_router(models.router)
13app.include_router(mcp.router)
14app.include_router(memory.router)
15app.include_router(skills.router)
16app.include_router(artifacts.router)
17app.include_router(uploads.router)
18app.include_router(threads.router)
19app.include_router(thread_runs.router)

  • 总览页最重要的不是背目录，而是看懂 Gateway 入口在源码里怎么成立。这段代码就是答案。
  • 前几行先挂鉴权、CSRF、CORS，说明 DeerFlow 对外入口不是裸的 agent 服务，而是完整 Web runtime。
  • 真正体现“统一入口”的是后面连续的 include_router(...)。模型、记忆、技能、工件、上传、线程、运行，全都挂在同一个 app 上。
  • 所以后面每一章细节虽然不同，但从系统视角看，几乎都能回到这一个 Gateway app 上找到进入点。

核心源码 2：嵌入式 DeerFlowClient 为什么不是“调用 Gateway 的壳”

节选自 backend/packages/harness/deerflow/client.py。这一段用来对照 HTTP 路径，帮助读者建立第二条入口链。

deerflow/client.py :: DeerFlowClient.__init__() / _get_runnable_config()
1# DeerFlowClient: 嵌入式 Python 入口，把本地函数调用装配成 DeerFlow 标准运行时
1class DeerFlowClient:
2 """Embedded Python client for DeerFlow agent system."""
3
4 # config_path / checkpointer: 配置来源和状态持久化入口；后面的开关则决定默认模型、技能、模式和附加中间件
5 def __init__(self, config_path: str | None = None, checkpointer=None, *, model_name: str | None = None, thinking_enabled: bool = True, subagent_enabled: bool = False, plan_mode: bool = False, agent_name: str | None = None, available_skills: set[str] | None = None, middlewares: Sequence[AgentMiddleware] | None = None, environment: str | None = None):
6 # 初始化阶段先读配置，再把这些默认开关缓存起来，真正造 agent 会延后到首个实际调用时
5 if config_path is not None:
6 reload_app_config(config_path)
7 self._app_config = get_app_config()
8
9 self._checkpointer = checkpointer
10 self._model_name = model_name
11 self._thinking_enabled = thinking_enabled
12 self._subagent_enabled = subagent_enabled
13 self._plan_mode = plan_mode
14 self._agent_name = agent_name
15 self._available_skills = set(available_skills) if available_skills is not None else None
16 self._middlewares = list(middlewares) if middlewares else []
17
18 # 这里故意做 lazy agent：第一次真正调用 chat/stream 时才组装 runtime
19 self._agent = None
20 self._agent_config_key: tuple | None = None
21
22 # self / thread_id / overrides: 当前 client 实例、这次运行归属线程，以及本轮临时覆盖值
23 def _get_runnable_config(self, thread_id: str, **overrides) -> RunnableConfig:
24 # 这里把实例默认值和本轮覆盖值折叠成统一 configurable，保证嵌入式路径和 Gateway 路径语义一致
25 configurable = {
24 "thread_id": thread_id,
25 "model_name": overrides.get("model_name", self._model_name),
26 "thinking_enabled": overrides.get("thinking_enabled", self._thinking_enabled),
27 "is_plan_mode": overrides.get("plan_mode", self._plan_mode),
28 "subagent_enabled": overrides.get("subagent_enabled", self._subagent_enabled),
29 }
30 return RunnableConfig(configurable=configurable, recursion_limit=overrides.get("recursion_limit", 100))
  • 这段代码最直接地证明了 `DeerFlowClient` 不是去请求 Gateway API。它在本进程里直接加载配置、持有 checkpointer、保存模型和运行开关。
  • 初始化阶段只存参数，不立刻创建 agent，这和 Gateway 某些懒加载策略是同一路数，说明两条入口共享的是运行时思想，不只是共享某几个类名。
  • _get_runnable_config(...) 组出来的也是同一种 `RunnableConfig` 语义：`thread_id`、`model_name`、`thinking_enabled`、`is_plan_mode`、`subagent_enabled` 都在这里落地。
  • 所以嵌入式路径和 HTTP 路径的差异，核心不在“有没有 agent runtime”，而在“谁来提供请求协议、谁来持有流式输出、谁来触发运行”。

核心源码 3：HTTP 请求是怎样从 thread run 路由真正发车的

节选自 backend/app/gateway/routers/thread_runs.py。这一段把“Gateway 是入口”继续往后推进了一层，能直接看到 run 是怎样创建、等待和流式返回的。

gateway/routers/thread_runs.py :: create_run() / stream_run() / wait_run()
1# thread_id / body / request: 线程标识、本次 run 输入和 HTTP 请求对象，路由层先把它们编排成统一启动入口
1@router.post("/{thread_id}/runs", response_model=RunResponse)
2@require_permission("runs", "create", owner_check=True, require_existing=True)
3async def create_run(thread_id: str, body: RunCreateRequest, request: Request) -> RunResponse:
4 # 这里只返回 run 记录，真正执行在后台 task 里继续跑
5 record = await start_run(body, thread_id, request)
6 return _record_to_response(record)
7
8# stream_run 参数和 create_run 一样，但这里的目标是把后台 run 挂到一条 SSE 连接上
8@router.post("/{thread_id}/runs/stream")
9@require_permission("runs", "create", owner_check=True, require_existing=True)
10async def stream_run(thread_id: str, body: RunCreateRequest, request: Request) -> StreamingResponse:
11 # 先拿 bridge 和 run manager，再启动 run，确保第一批事件从一开始就有消费者接住
11 bridge = get_stream_bridge(request)
12 run_mgr = get_run_manager(request)
13 record = await start_run(body, thread_id, request)
14 # StreamingResponse 只是出口包装，真正的执行生命周期仍然在后台 worker + sse_consumer 那一侧
14 return StreamingResponse(
15 sse_consumer(bridge, record, request, run_mgr),
16 media_type="text/event-stream",
17 headers={
18 "Cache-Control": "no-cache",
19 "Connection": "keep-alive",
20 "X-Accel-Buffering": "no",
21 "Content-Location": f"/api/threads/{thread_id}/runs/{record.run_id}",
22 },
23 )
24
25# wait_run 仍走同一套后台 run，只是这里会主动等它结束，并把最终 checkpoint 摘要同步返回
25@router.post("/{thread_id}/runs/wait", response_model=dict)
26async def wait_run(thread_id: str, body: RunCreateRequest, request: Request) -> dict:
27 bridge = get_stream_bridge(request)
28 run_mgr = get_run_manager(request)
29 record = await start_run(body, thread_id, request)
30 # 如果后台 task 存在就阻塞等待，等完再从 checkpointer 拿最终 channel_values 给同步调用方
30 completed = True
31 if record.task is not None:
32 completed = await wait_for_run_completion(bridge, record, request, run_mgr)
33 if completed:
34 checkpointer = get_checkpointer(request)
35 checkpoint_tuple = await checkpointer.aget_tuple({"configurable": {"thread_id": thread_id}})
36 if checkpoint_tuple is not None:
37 checkpoint = getattr(checkpoint_tuple, "checkpoint", {}) or {}
38 return serialize_channel_values(checkpoint.get("channel_values", {}))
  • 这段代码把 Gateway 的三种典型 run 入口一次讲透了: `runs` 创建后台任务、`runs/stream` 走 SSE、`runs/wait` 等待结束后返回最终 state。
  • 你会发现三条路的共同点都是先调用 `start_run(...)`，这说明不管前端想流式还是同步结果，真正的 DeerFlow run 生命周期是在同一个 service 层里发起的。
  • `stream_run(...)` 最关键的不是 `StreamingResponse` 本身，而是它把 `bridge`、`run_mgr` 和 `record` 交给 `sse_consumer(...)`，这就把“后台 agent 执行”和“前台 HTTP 连接”桥接起来了。
  • `wait_run(...)` 则说明同步等待路径并没有绕开运行时，而是仍然等待同一条后台 task，然后从 checkpoint 拿回最终 `channel_values`。
  • 所以从总览角度看，Gateway 不是“收到请求直接调 agent”，而是先把请求纳入统一的 run 语义，再决定怎么把结果返回给调用方。
关键认识： Gateway 路径和 DeerFlowClient 路径不是两套无关系统。它们共享核心 runtime 能力，但消费方式不同：一个面向 HTTP/SSE 和多外部入口，一个面向本地 in-process 调用。

配置面与使用方式

从部署和使用角度看，DeerFlow 的架构图会被几个配置面共同塑形：

  • config.yaml：决定模型、工具、sandbox、memory、summarization 等运行参数。
  • extensions_config.json：决定 MCP servers 与 skills 状态。
  • Gateway 相关环境变量：决定 worker、线程目录、运行时主目录等。
  • skills / deer-flow home 相关路径：决定本地状态、技能和运行时产物的持久化位置。

这也是为什么 DeerFlow 总体架构不能只用“几个 Python 包”来理解。很多能力边界并不写死在 agent 代码里，而是由 Gateway 路由、共享配置和本地持久化路径一起决定。

关键实现细节与权衡

DeerFlow 选择 Gateway-embedded runtime，有几个直接结果：

  • 公共 API 可以对外保持 LangGraph-compatible 形态，但默认服务入口仍由 Gateway 统一承载。
  • skills、uploads、artifacts、threads 清理、模型列表等非纯 agent 能力，都可以在同一进程服务面里管理。
  • 运行态所有权更集中，但也意味着 run state 与 stream bridge 默认持有在 Gateway 进程内。
设计取向： DeerFlow 优先选择“统一系统入口和一致体验”，而不是把所有能力拆成一组松散微服务。

当前限制、边界与非目标

这张总图必须明确几个边界，否则很容易产生错误理解。

  • Gateway-embedded runtime 是默认主路径，但 langgraph.json 仍保留兼容用途，不代表默认运行方式被反转。
  • Gateway 默认持有 run state，因此生产部署不能简单通过增加 worker 数来横向扩展，否则会破坏取消、重连和部分渠道语义。
  • DeerFlowClient 不是“调用 Gateway 的轻封装”，它是直接复用核心 runtime 的 in-process 路径。
  • 这一章不细讲单个 subsystem 的内部机制，例如 memory 更新策略、sandbox provider 差异、skills 安装安全扫描；那些属于后续分章。
