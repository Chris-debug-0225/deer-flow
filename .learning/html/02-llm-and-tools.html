这项能力在 DeerFlow 中解决什么问题

DeerFlow 不是“先选一个模型，再顺手挂几个工具”的简单封装。它要解决的是：同一个 super agent harness 如何在不同模型提供商、不同工具来源、不同 skill 约束、不同 runtime 入口之间，稳定地构建出一套可执行能力面。

因此本章的核心不是“LLM 会不会调工具”，而是 DeerFlow 如何把模型选择、工具集合、prompt 注入、tool policy 和运行时路径装配成一个整体。读者如果只记住通用的 tool-calling loop，却不知道 DeerFlow 的装配点，就无法读懂后面的执行引擎、skills、sandbox 和 context engineering。

一句话概括： DeerFlow 的 LLM 调用与工具系统，负责把“模型能思考什么”和“Agent 实际能做什么”绑定成同一条运行主线。

这项能力在代码库中的位置

最重要的两个锚点是 backend/packages/harness/deerflow/tools/tools.py 和 backend/packages/harness/deerflow/agents/factory.py。前者负责把 DeerFlow 可见的工具汇总起来，后者负责把配置中的模型实例化，并把它交给后续 agent 构建路径使用。

路径 职责 为什么关键
backend/packages/harness/deerflow/tools/tools.py 聚合 configured tools、built-in tools、MCP tools、ACP tools，并做去重 这里决定一个 runtime 最终能拿到哪些工具
backend/packages/harness/deerflow/agents/factory.py 根据配置创建模型、规范化 provider 差异、输出 agent 可用的 LLM 对象 这里决定 DeerFlow 不同模型接入的统一入口
backend/packages/harness/deerflow/agents/lead_agent/prompt.py 组装系统提示，注入 skills 和 deferred tools 信息 模型看到的能力面不只来自工具列表，也来自 prompt 结构
backend/packages/harness/deerflow/tools/builtins/tool_search.py 为 deferred MCP tools 提供按需发现与晋升机制 DeerFlow 没有把所有工具 schema 一次性塞给模型
读代码时的顺序建议： 先看 backend/packages/harness/deerflow/agents/factory.py 怎么建模型，再看 backend/packages/harness/deerflow/tools/tools.py 怎么组装工具，最后回到 lead agent 的 prompt 和 middleware，才会看到完整的装配闭环。

真实运行链路或数据链路

从运行时视角看，DeerFlow 的链路不是“用户输入 → 模型 → 工具 → 完成”这么简单，而是“配置解析 → 模型构建 → 工具汇总 → tool policy 过滤 → deferred tool 装配 → prompt 注入 → runtime 执行”。

  • 工具列表不会原封不动地暴露给模型，skills 允许通过 tool policy 收窄能力面。这里的具体落点不是抽象规则名，而是 deerflow.skills.tool_policy.filter_tools_by_skill_allowed_tools，它会在 backend/packages/harness/deerflow/agents/lead_agent/agent.py 和 backend/packages/harness/deerflow/subagents/executor.py 的装配路径里对工具集做运行前过滤。
  • MCP 工具在 DeerFlow 中可以走 deferred 模式，模型先看到名字，需要时再通过 tool_search 取回完整 schema。

为什么说它是联动系统

同一份工具目录，在不同的 skills whitelist、不同的 prompt section、不同的 runtime 路径下，对模型来说是不同的“能力世界”。这就是 DeerFlow 和通用“绑几个 tools 给 LLM”教程的根本差异。

配置面与使用方式

模型和工具的入口都受配置控制。模型通常来自 config.yaml 的 models 配置，工具则来自 built-in 默认能力、显式配置工具、MCP server 和 ACP agent 等多个来源。

配置面 控制内容 会影响什么
models provider、model 名、API key、base_url、responses API、thinking 能力等 factory 产出的模型对象与能力标签
tools 启用哪些配置工具 最终可聚合到 agent 的工具集合
extensions_config.json MCP server 和 skills 启用状态 影响 MCP 工具缓存和技能注入
skills frontmatter allowed-tools 等约束 同一个 agent 在某些工作流下的可用工具边界

对使用者来说，最重要的不是背住所有配置项，而是知道：配置并不会直接“让模型会用工具”，它只是决定 DeerFlow 在运行时能装配出什么能力组合。

关键实现细节与权衡

DeerFlow 在这里做了几项很有代表性的取舍。

  • 工具去重按名称进行，避免重复 schema 让模型看到歧义函数定义。
  • MCP 工具可以延迟暴露，而不是把完整 schema 全量推给模型，减少 prompt 噪声和上下文压力。
  • 模型构建与工具汇总分离，保证 DeerFlow 可以同时支持多 provider 和多工具来源，而不让 factory 变成巨型杂糅入口。
  • prompt 会反向塑造模型如何理解工具，这意味着“系统提示”在 DeerFlow 里是装配的一部分，不是文案附属物。
对读者最重要的权衡： DeerFlow 优先追求“能力面可控”和“运行时一致性”，而不是最短路径把所有工具都直接绑给模型。

当前限制、边界与非目标

这一层不是 DeerFlow 的模型原理教程，也不是统一抽象所有 provider 差异的终极层。它的目标是把配置、模型对象和工具集合稳定送入 DeerFlow runtime。

  • 不同 provider 仍可能存在推理、工具调用、streaming 语义差异，factory 只能做统一入口，不能消除全部行为差别。
  • tool_search 只解决 deferred tool discovery，不等于 DeerFlow 对所有外部能力都做了惰性按需装载。
  • skills 可以影响工具边界，但 skill 本身不是 tool，也不是任意代码插件。
  • 本章不展开具体 run lifecycle、SSE、thread state，这些属于第 10 章执行引擎主线。

它与 DeerFlow 其他能力的关系

这一章是后续多个章节的底座：

  • 和第 10 章的关系：第 10 章关注“这些模型和工具如何真正跑起来”，本章关注“它们如何被装配出来”。
  • 和第 11 章的关系：skills 不直接替代工具，而是通过 prompt 与 tool policy 改变模型对工具的使用方式。
  • 和第 20 章的关系：上下文工程并不是独立于工具系统存在的，skills section、deferred tools section 都属于上下文工程的一部分。
  • 和 sandbox、memory、sub-agent 的关系：这些能力最后都要以工具、中间件或 prompt 能力的形式进入同一条 agent 执行主线。
