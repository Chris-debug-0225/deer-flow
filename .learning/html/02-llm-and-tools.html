1. LLM API的基本调用流程

理解LLM API的调用方式是构建Agent的基础。 虽然不同的LLM提供商有差异，但基本流程是相同的。

最简单的调用

from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4", api_key="your-api-key") # 调用LLM response = llm.invoke("你好，请介绍一下自己") print(response.content) # 输出LLM的响应

带有消息历史的调用

在Agent中，我们通常需要维护对话历史。

from langchain_core.messages import HumanMessage, AIMessage messages = [ HumanMessage(content="你好"), AIMessage(content="你好！很高兴认识你。"), HumanMessage(content="你叫什么名字？") ] response = llm.invoke(messages) print(response.content)

调用流程图

准备消息
→
调用API
→
接收响应
→
处理结果

重要的参数

参数 说明 影响
temperature 控制输出的随机性（0-2） 0=确定性，1=平衡，2=最随机
max_tokens 限制输出的最大长度 影响成本和响应时间
top_p 核采样参数 控制词汇多样性
frequency_penalty 降低重复词汇的概率 0=无惩罚，2=最大惩罚
最佳实践：
对于Agent，通常使用较低的temperature（0.3-0.7）， 以获得更可靠的决策。

2. Token、Context Window、Cost的概念

这三个概念直接影响Agent的性能和成本。

Token是什么？

Token是LLM处理文本的基本单位。 它不是字符，而是更大的单位。

Token的大小：
- 英文：1 Token ≈ 4个字符
- 中文：1 Token ≈ 1.3个字符
- 代码：1 Token ≈ 2-3个字符
Token计算例子：
"Hello, how are you?" ≈ 5 tokens
"你好，你最近怎么样？" ≈ 8 tokens

Context Window（上下文窗口）

Context Window是LLM一次能处理的最大Token数。

模型 Context Window 说明
GPT-3.5 Turbo 16K tokens 较小，适合简单任务
GPT-4 8K / 32K / 128K tokens 多个版本可选
Claude 3.5 Sonnet 200K tokens 非常大，适合长文档
Gemini 2.0 1M tokens 超大，可处理整个项目
重要： Context Window包括输入和输出。 如果你有10K tokens的输入，最多只能输出6K tokens（对于16K模型）。

成本计算

LLM的成本通常按Token计费，输入和输出的价格不同。

成本计算例子：
假设GPT-4的价格是：
- 输入：$0.03 / 1K tokens
- 输出：$0.06 / 1K tokens

一次调用：
- 输入：1000 tokens → $0.03
- 输出：500 tokens → $0.03
- 总成本：$0.06

Context Window的限制

Lost in the Middle问题：
当Context Window很大时，LLM有时会忽略中间的信息， 只关注开头和结尾。这是一个已知的问题。
优化建议：
1. 定期清理不必要的历史记录
2. 使用摘要来压缩长对话
3. 将重要信息放在开头和结尾
4. 选择合适大小的Context Window

3. Function Calling / Tool Use 的原理

Function Calling（或Tool Use）是Agent能够调用工具的关键机制。

什么是Function Calling？

Function Calling是一种让LLM能够请求调用特定函数的机制。 LLM不直接执行函数，而是告诉系统"我想调用这个函数，参数是这些"。

工作原理：
1. 你告诉LLM有哪些函数可用
2. LLM分析用户的请求
3. 如果需要，LLM生成一个函数调用请求
4. 系统执行该函数
5. 将结果返回给LLM
6. LLM继续处理或生成最终答案

定义工具

首先，你需要定义LLM可以使用的工具。

from langchain_core.tools import tool @tool def get_weather(location: str) -> str: """获取指定位置的天气信息 Args: location: 位置名称，例如"北京" Returns: 天气信息字符串 """ # 这里应该调用真实的天气API return f"{location}的天气：晴天，温度25°C" @tool def calculate(expression: str) -> str: """计算数学表达式 Args: expression: 数学表达式，例如"2+2" Returns: 计算结果 """ try: result = eval(expression) return str(result) except: return "计算错误" # 将工具绑定到LLM tools = [get_weather, calculate] llm_with_tools = llm.bind_tools(tools)

LLM生成工具调用

当LLM需要使用工具时，它会生成一个工具调用请求。

from langchain_core.messages import HumanMessage messages = [ HumanMessage(content="北京现在的天气怎么样？") ] response = llm_with_tools.invoke(messages) # 检查是否有工具调用 if response.tool_calls: for tool_call in response.tool_calls: print(f"工具名称: {tool_call['name']}") print(f"参数: {tool_call['args']}") # 输出: # 工具名称: get_weather # 参数: {'location': '北京'}

执行工具调用

系统需要执行LLM请求的工具调用。

from langchain_core.messages import ToolMessage # 执行工具 tool_results = [] for tool_call in response.tool_calls: tool_name = tool_call['name'] tool_args = tool_call['args'] # 根据工具名称调用相应的工具 if tool_name == "get_weather": result = get_weather.invoke(tool_args) elif tool_name == "calculate": result = calculate.invoke(tool_args) # 创建工具消息 tool_message = ToolMessage( content=result, tool_call_id=tool_call['id'] ) tool_results.append(tool_message) # 将工具结果返回给LLM messages.append(response) messages.extend(tool_results) # LLM继续处理 final_response = llm_with_tools.invoke(messages) print(final_response.content)

完整的工具调用循环

用户输入
→
LLM分析
→
生成工具调用
→
执行工具
→
返回结果
→
LLM生成答案

4. 不同LLM提供商的差异

不同的LLM提供商有不同的API、定价和能力。

提供商 主要模型 特点 成本
OpenAI GPT-4, GPT-3.5 功能完整，生态成熟 中等
Anthropic Claude 3.5 长Context，安全性好 中等
Google Gemini 2.0 超大Context，多模态 便宜
DeepSeek DeepSeek v3 推理能力强，成本低 很便宜
字节跳动 Doubao 中文优化，成本低 便宜

API差异

虽然大多数提供商都支持OpenAI兼容的API，但仍有一些差异。

OpenAI API：
from langchain_openai import ChatOpenAI llm = ChatOpenAI( model="gpt-4", api_key="sk-..." )
Anthropic API：
from langchain_anthropic import ChatAnthropic llm = ChatAnthropic( model="claude-3-5-sonnet-20241022", api_key="sk-ant-..." )
OpenAI兼容的API（如DeepSeek）：
from langchain_openai import ChatOpenAI llm = ChatOpenAI( model="deepseek-chat", api_key="sk-...", base_url="https://api.deepseek.com/v1" )

选择模型的建议

对于Agent开发：
- 开发阶段：使用便宜的模型（如GPT-3.5、DeepSeek）
- 生产环境：使用可靠的模型（如GPT-4、Claude 3.5）
- 需要推理：使用推理模型（如o1、DeepSeek R1）
- 需要长Context：使用Claude或Gemini

5. 工具定义、调用、结果处理

工具是Agent的"手"，让Agent能够与外部世界交互。

工具的类型

类型 说明 例子
Python函数 最简单的工具类型 @tool装饰器定义的函数
Runnable LangChain的Runnable接口 任何实现Runnable的对象
API调用 调用外部API HTTP请求、数据库查询
自定义类 实现特定接口的类 Tool基类的子类

定义工具的最佳实践

from langchain_core.tools import tool from typing import Optional @tool def search_web(query: str, max_results: Optional[int] = 5) -> str: """在网络上搜索信息 这是一个详细的文档字符串，LLM会使用它来理解工具的功能。 Args: query: 搜索查询，例如"Python教程" max_results: 返回的最大结果数，默认为5 Returns: 搜索结果的字符串，包含标题和摘要 Raises: ValueError: 如果查询为空 """ if not query: raise ValueError("查询不能为空") # 这里应该调用真实的搜索API # 例如Google Search API、Bing Search API等 results = [ f"结果{i+1}: 关于{query}的信息" for i in range(max_results) ] return "\n".join(results) # 工具的关键要素： # 1. 清晰的函数名称 # 2. 详细的文档字符串 # 3. 类型注解 # 4. 合理的参数 # 5. 有意义的返回值

处理工具结果

工具的结果需要被正确处理和返回给LLM。

from langchain_core.messages import ToolMessage def process_tool_result(tool_call, tool_result): """处理工具结果""" # 创建工具消息 tool_message = ToolMessage( content=tool_result, tool_call_id=tool_call['id'], name=tool_call['name'] ) return tool_message # 错误处理 def execute_tool_safely(tool_call, tools_dict): """安全地执行工具""" try: tool_name = tool_call['name'] tool_args = tool_call['args'] if tool_name not in tools_dict: return f"错误：工具'{tool_name}'不存在" tool = tools_dict[tool_name] result = tool.invoke(tool_args) return str(result) except Exception as e: return f"执行工具时出错：{str(e)}"

工具的错误处理

常见错误：
1. 工具不存在
2. 参数类型错误
3. 工具执行超时
4. 外部API不可用
5. 权限不足
最佳实践：
1. 总是使用try-except捕获异常
2. 返回有意义的错误消息
3. 实现重试机制
4. 记录工具调用日志
5. 设置超时限制

6. 工具调用循环的完整流程

现在让我们看一个完整的工具调用循环。

from langchain_core.messages import HumanMessage, AIMessage, ToolMessage def run_agent_loop(user_input: str, llm_with_tools, tools_dict, max_iterations=10): """运行Agent的工具调用循环""" messages = [HumanMessage(content=user_input)] iteration = 0 while iteration < max_iterations: iteration += 1 print(f"\n--- 迭代 {iteration} ---") # 1. 调用LLM response = llm_with_tools.invoke(messages) print(f"LLM响应: {response.content}") # 2. 检查是否有工具调用 if not response.tool_calls: # 没有工具调用，返回最终答案 print(f"\n最终答案: {response.content}") return response.content # 3. 添加LLM的响应到消息历史 messages.append(response) # 4. 执行工具调用 for tool_call in response.tool_calls: tool_name = tool_call['name'] tool_args = tool_call['args'] print(f"执行工具: {tool_name}({tool_args})") # 执行工具 if tool_name in tools_dict: tool_result = tools_dict[tool_name].invoke(tool_args) else: tool_result = f"错误：工具'{tool_name}'不存在" print(f"工具结果: {tool_result}") # 5. 创建工具消息 tool_message = ToolMessage( content=str(tool_result), tool_call_id=tool_call['id'], name=tool_name ) # 6. 添加工具消息到历史 messages.append(tool_message) # 超过最大迭代次数 return "错误：超过最大迭代次数" # 使用示例 tools_dict = { "get_weather": get_weather, "calculate": calculate, "search_web": search_web } result = run_agent_loop( "北京现在的天气怎么样？明天会下雨吗？", llm_with_tools, tools_dict )

循环的关键点

  • 消息历史： 保持完整的对话历史，包括LLM的响应和工具的结果
  • 迭代限制： 防止无限循环，设置最大迭代次数
  • 错误处理： 优雅地处理工具执行失败
  • 终止条件： 当LLM不再调用工具时，循环结束

7. 最佳实践与常见问题

最佳实践

1. 工具设计：
- 保持工具简单和专注
- 提供清晰的文档
- 使用类型注解
- 返回结构化数据
2. 错误处理：
- 总是捕获异常
- 返回有意义的错误消息
- 实现重试机制
- 记录详细的日志
3. 性能优化：
- 缓存工具结果
- 使用异步调用
- 设置超时限制
- 批量处理请求
4. 成本控制：
- 使用较小的模型进行简单任务
- 压缩消息历史
- 限制工具调用次数
- 监控Token使用

常见问题

Q: LLM经常调用不存在的工具怎么办？
A: 在提示词中明确列出可用的工具，并在工具定义中提供清晰的文档。
Q: 工具调用陷入无限循环怎么办？
A: 设置最大迭代次数，并在LLM的提示词中明确说明何时停止调用工具。
Q: 如何处理工具执行超时？
A: 使用threading或asyncio实现超时机制，返回超时错误消息。
Q: 如何优化成本？
A: 使用更便宜的模型，压缩消息历史，缓存常见查询的结果。

📝 关键要点总结

1. LLM API调用： 准备消息 → 调用API → 接收响应 → 处理结果
2. Token和成本： 理解Token大小和Context Window，优化成本
3. Function Calling： LLM生成工具调用请求，系统执行工具
4. 工具设计： 简单、专注、文档清晰、类型注解
5. 工具调用循环： 迭代调用LLM和工具，直到获得最终答案
6. 错误处理： 总是捕获异常，返回有意义的错误消息
