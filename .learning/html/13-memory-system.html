长期记忆系统

DeerFlow 学习系列 - 第 13 章

本章讲的是 DeerFlow 如何把“跨会话可延续信息”单独建模

DeerFlow 的长期记忆不是 thread checkpoint history，不是 summarization，也不是 event store。它是一套独立的 memory schema、存储层、LLM 更新流程与 prompt 注入机制，用来沉淀跨运行周期仍然有价值的用户上下文与事实。

这项能力在 DeerFlow 中解决什么问题

如果 DeerFlow 每次运行都只依赖当前 thread 的即时消息，那么 agent 无法稳定记住用户长期偏好、持续工作背景或跨任务积累的事实。长期记忆系统就是为此存在的：它把“值得跨会话保留的信息”从一次 run 的瞬时上下文中分离出来，形成可注入、可更新、可审视的结构化记忆。

这里必须先划清三个边界。第一，长期记忆不是 thread checkpoint history；后者是运行态恢复与状态延续。第二，长期记忆不是 summarization；summary 主要为压缩当前对话窗口服务。第三，长期记忆也不是 event store；它不追求保存全部事件，而是有选择地沉淀高价值信息。

一句话理解： DeerFlow 的 memory 关注“未来还值得再次使用的用户与任务上下文”，而不是“把过去发生的一切都原样保存”。

这些文件恰好把 memory 拆成四层。storage.py 负责“记忆存在哪里、怎么按 user/agent 隔离”；updater.py 负责“对话怎样变成结构化 update”；prompt.py 负责“更新 memory 时要给模型看什么格式的上下文”；lead_agent/prompt.py 和运行时中间件负责“哪些记忆在主对话时重新被注入回来”。所以记忆系统不是一个 JSON 文件，而是一条完整的数据管线。

如果你只看 storage.py，会以为 memory 是普通持久化；如果只看 updater.py，又会以为它只是一次模型抽取。只有把这几层拼起来，才看得见 DeerFlow 为什么能做成“跨会话可延续的上下文”。

真实运行链路或数据链路

长期记忆在 DeerFlow 里有两条主线：一条是“运行前注入”，一条是“运行后更新”。两条线都围绕 backend/packages/harness/deerflow/agents/memory/storage.py 和 backend/packages/harness/deerflow/agents/memory/updater.py 展开。

memory flow :: 注入链与更新链
1# 注入链
2"backend/packages/harness/deerflow/agents/memory/storage.py"
3 -> "load(agent_name, user_id)"
4 -> "backend/packages/harness/deerflow/agents/lead_agent/prompt.py"
5 -> "render memory section"
6 -> "lead agent system prompt gets durable context"
7
8# 更新链
9"conversation / run output"
10 -> "backend/packages/harness/deerflow/agents/memory/updater.py"
11 -> "backend/packages/harness/deerflow/agents/memory/prompt.py"
12 -> "model produces structured update"
13 -> "normalize / validate / merge facts"
14 -> "backend/packages/harness/deerflow/agents/memory/storage.py.save(...)"

再往真实源码里走一步，会看到更新链前面其实还有一层“排队”。backend/packages/harness/deerflow/agents/middlewares/memory_middleware.py 并不直接在用户请求线程里更新记忆，而是把本轮对话放进 backend/packages/harness/deerflow/agents/memory/queue.py 的 debounce 队列。队列稍后再调用 updater.py 完成真正的 LLM 抽取和存储。这解释了为什么 memory 更新更像后台整理，而不是同步阻塞在主回答路径上。

在存储层，memory 会根据是否传入 user_id、agent_name 选择不同文件路径。也就是说，DeerFlow 支持全局 memory、per-user memory，以及 per-user + per-agent memory，而不是把所有记忆都塞进一个统一文件。这个路径分层正是长期记忆与 thread history 的重要差异。

核心源码 2：memory queue 怎样做 debounce

节选自 backend/packages/harness/deerflow/agents/memory/queue.py。这段代码解释了为什么 memory 更新更像异步整理工人。

agents/memory/queue.py :: MemoryUpdateQueue
1class MemoryUpdateQueue:
2 """带 debounce 的记忆更新队列。"""
3
4 def __init__(self):
5 self._queue: list[ConversationContext] = []
6 self._lock = threading.Lock()
7 self._timer: threading.Timer | None = None
8 self._processing = False
9
10 @staticmethod
11 def _queue_key(thread_id: str, user_id: str | None, agent_name: str | None) -> tuple[str, str | None, str | None]:
12 # 同一个 thread 里，不同 user/agent 的长期记忆不能混成一份
13 return (thread_id, user_id, agent_name)
14
15 def add(self, thread_id: str, messages: list[Any], agent_name: str | None = None, user_id: str | None = None, correction_detected: bool = False, reinforcement_detected: bool = False) -> None:
16 config = get_memory_config()
17 if not config.enabled:
18 return
19
20 with self._lock:
21 self._enqueue_locked(thread_id=thread_id, messages=messages, agent_name=agent_name, user_id=user_id, correction_detected=correction_detected, reinforcement_detected=reinforcement_detected)
22 self._reset_timer()
23
24 def add_nowait(self, thread_id: str, messages: list[Any], agent_name: str | None = None, user_id: str | None = None, correction_detected: bool = False, reinforcement_detected: bool = False) -> None:
25 config = get_memory_config()
26 if not config.enabled:
27 return
28
29 with self._lock:
30 self._enqueue_locked(thread_id=thread_id, messages=messages, agent_name=agent_name, user_id=user_id, correction_detected=correction_detected, reinforcement_detected=reinforcement_detected)
31 self._schedule_timer(0)
  • 队列初始化时就同时维护了 _queue、_lock、_timer 和 _processing，说明它不是“简单 append 一个列表”，而是一个受线程同步保护的小型后台调度器。
  • _queue_key(...) 把去抖身份定义成 (thread_id, user_id, agent_name)，意味着同一线程里不同用户、不同 agent 的记忆更新会被拆开处理。
  • add(...) 第一件事不是排队，而是先看 memory_config.enabled。记忆系统关掉时，这条旁路会在入口直接短路退出。
  • 真正入队动作包在锁里，再调用 _reset_timer()，这说明 DeerFlow 希望把短时间内连续发生的对话合并成一批处理，而不是每轮都立刻调一次 memory LLM。
  • add_nowait(...) 则是另一条更激进的路径，它不是等待 debounce 窗口，而是直接 _schedule_timer(0)，马上把后台处理线程唤起来。

核心源码 3：memory updater 真正做的是“结构化抽取 + 校验 + 持久化”

节选自 backend/packages/harness/deerflow/agents/memory/updater.py。这一段能把“为什么 memory 不是直接把对话存文件”讲透。

  • `get_memory_data(...)` 看起来简单，但它点明了一个架构边界: updater 并不自己知道具体落盘路径，而是委托给 storage provider。
  • `create_memory_fact(...)` 也不是直接往列表里塞内容就结束，它会先 trim、校验 confidence、生成稳定 ID、补时间戳，再统一走保存路径。
  • 最关键的是 `_normalize_memory_update_fact(...)`。这一步说明来自模型的结构化更新结果并不能直接信任，必须再做类型、空字符串、数值可解析性和有限数校验。
  • 所以 DeerFlow 的 memory update 链不是“模型说了算”，而是“模型提建议，updater 做协议清洗，storage 再持久化”。
更新不是“原样转存对话”： backend/packages/harness/deerflow/agents/memory/updater.py 会通过 LLM 产出结构化 update，然后做归一化、置信度校验、事实去留处理与持久化。这是一条“抽取和整理”链，而不是日志归档链。

配置面与使用方式

从使用面看，memory 至少有四个配置或操作点：

  • backend/packages/harness/deerflow/agents/memory/storage.py 根据 user_id 与 agent_name 决定 memory 文件位置，形成 per-user / per-agent 路径分层。
  • backend/packages/harness/deerflow/config/memory_config.py 控制存储类与默认存储路径，允许 DeerFlow 切换 memory provider，而不是写死单一实现。
  • backend/packages/harness/deerflow/agents/memory/updater.py 提供导入、清空、create/update/delete fact 等接口，说明长期记忆不仅能被模型更新，也能被系统或用户控制面管理。
  • backend/packages/harness/deerflow/agents/lead_agent/prompt.py 决定哪些 memory 片段被注入给 lead agent，这意味着“存了什么”与“运行时看见什么”并不完全是同一层概念。

这也解释了一个常见现象：memory 文件已经更新了，但你还要继续看 prompt 注入层和动态上下文层，才能判断它是否已经在下一轮真正影响模型。DeerFlow 在这里明确把“存储成功”和“运行时可见”拆成了两个阶段。

如果你在排查某条记忆为什么影响了回答，通常要同时看存储文件、updater 更新逻辑和 prompt 注入逻辑，而不是只盯着单个 JSON 文件。

关键实现细节与权衡

  • backend/packages/harness/deerflow/agents/memory/storage.py 使用结构化 schema 和文件缓存，而不是把 memory 当作任意文本 blob，这让 DeerFlow 可以区分 user context、history 和 facts。
  • per-user / per-agent 路径设计让记忆隔离更清楚，但也意味着理解某条记忆来源时必须知道当前 run 的 user_id 与 agent_name。
  • backend/packages/harness/deerflow/agents/memory/updater.py 采用 LLM 生成结构化更新，再做 normalize 与 validation；这样可获得较强抽取能力，但也引入模型判断质量的波动。
  • 事实层支持置信度和显式删除，不把 memory 视作只增不减的档案，这更贴近“工作记忆维护”而不是“永不清理的事件仓库”。
  • backend/docs/MEMORY_IMPROVEMENTS_SUMMARY.md 所描述的改进方向，与 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 的注入方式结合起来，说明 DeerFlow 更看重“让记忆可被 agent 正确使用”，而不是单纯积累更多字段。
核心取舍： DeerFlow 选择让长期记忆成为一个独立、可更新、可注入的数据层，而不是把它混入 checkpoint、summary 或 event log 中，这样边界最清晰，但也要求读者分开理解多条上下文链路。

当前限制、边界与非目标

  • 长期记忆不负责恢复完整运行状态；那是 thread state / checkpoint 体系的职责。
  • 长期记忆不追求保存每条事件和每个 token 输出；它不是 event store，也不保证历史全量可追溯。
  • summary 与 memory 可能都出现在同一次运行中，但二者目的不同：summary 主要为上下文压缩，memory 主要为跨会话延续。
  • 当前默认实现以文件存储为主，适合 DeerFlow 现有部署形态；如果需要更复杂的共享记忆后端，需要通过存储抽象扩展，而不是直接修改 prompt 层。

它与 DeerFlow 其他能力的关系

长期记忆和 DeerFlow 其他章节存在明确分工：

  • 它和第 10 章执行引擎相连，因为 memory 最终是通过 backend/packages/harness/deerflow/agents/lead_agent/prompt.py 进入运行主线，而更新通常在运行结束后发生。
  • 它和第 20 章上下文工程相连，因为长期记忆本质上是一种稳定、可复用的上下文注入来源，只是比一次性 prompt 拼接更持久。
  • 它和 backend/docs/AUTH_DESIGN.md 描述的用户隔离设计相连，因为 per-user memory 路径依赖 DeerFlow 的用户/线程目录模型。
  • 它和 sandbox、artifact、上传系统并不等价。即使 memory 里会引用文件相关事实，真正的文件生命周期仍然由 sandbox 和 user-data 路径体系负责。
